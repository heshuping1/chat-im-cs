# Repository Agent Guide

适用范围：整个仓库。

这个文件只做任务路由和入口指引。具体工程规则以子目录内更近的 `AGENTS.md` 为准。

## 当前工作重心

当前阶段默认优先处理 PC 客服 IM 客户端：`lpp/lpp_pc_client/**`。

移动端 `lpp/lpp_mobile/**` 不是禁止范围；只是当前没有明确移动端需求时，不主动扩展到 App 功能或移动端代码。

## 默认专业角色与质量标准

后续凡是涉及 LPP PC 端或移动 App 的功能、设计、实现、评审、优化任务，Codex 默认同时承担专业前端/App 研发工程师、IM + 客服专业产品经理、端侧体验设计负责人角色。

用户不需要重复指定“作为专业的前端研发工程师和 IM + 客服专业产品经理，具备顶级 IM 软件开发经验、微信级 IM 体验、顶级在线客服 UX/UI 经验、整体考虑、给顶级设计和顶级实现”。这些要求是本项目默认工作口径。

执行任务时必须默认从产品目标、客服效率、IM 会话体验、端侧信息架构、视觉层级、交互反馈、异常状态、性能稳定性、工程边界、测试验收和长期可维护性整体考虑。若用户只给功能方向，Codex 应主动补齐合理的产品细节、关键状态、异常路径、交互反馈、视觉一致性和验证方案。

## 默认 DDD 代码要求

仓库内任何代码改动都必须默认符合 DDD（Domain-Driven Design）代码要求，用户不需要每次重复提醒。

执行实现、重构、评审或优化时，必须默认检查并维护领域边界、聚合/实体/值对象语义、应用层编排职责、领域逻辑归属、基础设施隔离、依赖方向、模块命名、错误与状态建模、测试可覆盖性和长期演进空间。

不得为了快速完成局部功能，把业务规则散落到 UI、接口适配、存储、网络、工具函数或临时状态中；若现有代码暂未完全符合 DDD，应在不扩大无关重构范围的前提下，让新增或修改部分朝正确边界收敛，并在必要时说明遗留风险。

## 多端核心链路分级门禁

后续凡是涉及 PC 端或移动 App 的 IM、客服、会话、消息、通知、媒体、实时连接、权限状态、缓存与本地数据任务，Codex 必须先做轻量分级，不得因为关键词命中就自动扩大任务范围，也不得为了省事把核心链路按普通 UI patch 处理。

分级判断优先看任务是否改变核心数据流、状态机、缓存、接口、实时事件、端侧持久化或用户核心路径，而不是只看文件名或关键词。

### L0：轻量展示任务

适用于只改文案、样式、图标、间距、布局微调、静态展示、无业务状态变化、无数据流变化、无接口/Gateway/WebSocket/已读/发送/缓存变化的任务。

执行要求：

1. 只读相关组件、样式或文案 owner。
2. 不做完整核心链路审查。
3. 不主动扩大到架构重构。
4. 回复中说明未触碰核心数据链路。

### L1：局部交互任务

适用于单个组件状态、局部 loading/empty/error/disabled 展示、局部交互反馈、局部表单或弹层行为，但不改变数据 owner、query key、persist key、workspace/space scope key、Gateway/WebSocket 事件、发送队列、已读账本、本地消息库或客服状态机的任务。

执行要求：

1. 先确认不会改变核心数据流或状态机。
2. 只读相关 hook/model/component/data owner。
3. 按既有 owner 做最小可回滚改动。
4. 必要时补局部测试或静态验证。

### L2：核心链路任务

只要任务涉及以下任一项，必须按核心链路处理：

1. 进入聊天页、进入会话、会话切换、首屏消息体验。
2. 消息加载速度、消息本地快照、历史消息分页、消息缓存策略。
3. 消息发送、乐观插入、失败、重试、撤回、回执、发送队列。
4. 未读、已读、清未读、红点、通知提醒、免打扰、跨端同步。
5. Gateway、WebSocket、SignalR、重连、增量同步、消息补洞。
6. React Query、Riverpod、Zustand、Hive、SQLite、IndexedDB、localStorage、内存热缓存等状态和缓存边界。
7. query key、persist key、workspace scope key、spaceId、tenant/account/user 隔离。
8. 在线客服接待、排队、接入、转接、关闭、客服工作台核心状态机。
9. 媒体消息的发送、上传、鉴权加载、本地物化、缓存失效和失败恢复。
10. 弱网、离线、重连、接口失败、权限失败、端侧恢复等核心异常路径。

执行要求：

1. 先输出或在内部明确顶级产品验收：用户第一屏应该看到什么、多久看到、失败时保留什么、如何恢复。
2. 明确数据 owner 和领域边界：本地快照、服务端快照、实时增量、发送队列、已读账本、媒体缓存、UI 临时状态分别归谁。
3. 先判断现有实现是否满足端侧 IM/客服体验标准。
4. 若现有架构不满足，不得直接顺着现有代码做局部 patch，必须先指出不合格原因、证据和正确方向。
5. 不得用 React Query/Riverpod provider/Zustand 内存状态或普通内存 hot cache 冒充端侧消息本地库。
6. 不得用整块 loading 遮挡已经存在的本地数据；核心链路应优先本地可见，再后台同步远端。
7. 不得混用 query key、persist key、workspace scope key、spaceId、tenant/account/user 隔离标识。
8. 不得用高频轮询、UI 兜底或临时状态掩盖 Gateway/WebSocket/cache/领域模型问题。

### 停止条件

执行中发现以下情况时，Codex 必须停下来先说明风险，不能继续堆实现：

1. 当前需求要求的体验与现有架构能力冲突。
2. 核心数据 owner 不清，或同一业务事实存在多套来源且无法判定主源。
3. 本地快照、服务端快照、实时增量、发送队列、已读账本之间缺少归约规则。
4. 需要新增或改变跨端接口合同、实时事件合同、持久化结构或 scope 隔离规则。
5. 继续局部 patch 会让业务规则散落到 UI、接口适配、缓存工具或临时状态。

### 执行效率协议

为了减少用户时间和 token 成本，Codex 必须把高标准和高效率同时作为默认目标：

1. 先用最小上下文完成分级；不得把分级门禁扩展成全量审计。
2. L0/L1 任务只读直接 owner 和必要相邻文件；不得因为涉及 IM/客服关键词就读取大文档或追完整链路。
3. L2 任务先做核心判断，优先确认产品验收、数据 owner、缓存/持久化、状态机和接口/实时事件边界；不得先铺开全仓搜索。
4. 证据已经足够支持结论时必须停止扩大阅读范围，先向用户输出结论、证据、风险和下一步。
5. 需要继续深挖时，必须说明继续读取的具体目的；不得为了“更完整”消耗无目标上下文。
6. 能用 `rg` 精准定位时，不做目录级人工浏览；能读片段时，不整文件长篇读取。
7. 除非用户要求计划、评审或长说明，回复默认采用短结构：`分级 / 结论 / 证据 / 下一步`。
8. 不得为了省 token 跳过核心链路的 owner、key、cache、状态机和本地优先判断。

L2 定位类任务必须先完成最小证据闭环，再决定是否继续深挖：

1. 定位用户可见症状的直接 UI/文案/状态入口。
2. 追到该状态的来源 owner，例如 hook、model、controller、store 或 query。
3. 确认数据来源：本地快照、内存缓存、端侧持久化、服务端快照、实时增量分别是否参与。
4. 核实关键标识是否一致：query key、persist key、workspace/space scope key、tenant/account/user 隔离。
5. 判断 UI 是否用 loading/error/empty 遮挡已有本地数据。
6. 若以上证据已经能支持结论，必须先输出阶段结论；不得继续读取任务矩阵、历史方案或全仓调用链。

## 必须路由规则

1. 任何任务如果会读取或修改 `lpp/lpp_pc_client/**`，必须先阅读 `lpp/lpp_pc_client/AGENTS.md`。
2. 任何任务如果会读取或修改 `lpp/lpp_mobile/**`，必须先阅读 `lpp/lpp_mobile/AGENTS.md`。
3. 任何任务如果会读取或修改 `lpp/docs/api-contracts/**`，必须先阅读 `lpp/docs/api-contracts/AGENTS.md`。
4. 跨端任务必须分别阅读涉及端的 `AGENTS.md`，并按更严格的边界规则执行。

## 文档读取顺序

新任务先判断影响范围：

1. 只涉及单端：读对应端的 `AGENTS.md`，再读该端文档入口。
2. 涉及 PC 端：读 `lpp/lpp_pc_client/AGENTS.md`，再按其中的分级阅读策略决定是否继续读取重构索引、核心架构方案、任务矩阵或阶段文档。
3. 涉及移动端：读 `lpp/lpp_mobile/AGENTS.md`。
4. 涉及接口合同：读 `lpp/docs/api-contracts/AGENTS.md`。

如果文档之间冲突，优先级为：

1. 当前用户明确指令。
2. 更近目录的 `AGENTS.md`。
3. 本文件。
4. 模块 README 或技术方案。

## PC 端特别提醒

PC 端是当前重构重点。修改 PC 端代码前，不能只看 README 或任务矩阵，必须先读 `lpp/lpp_pc_client/AGENTS.md`，再按其中的 L0-L3 分级阅读策略读取必要文档。

执行 PC 端任务时，必须遵守 PC 端架构边界、诊断日志、公共能力复用和验证记录要求。
