# APP AI 协作规则

状态：当前有效

日期：2026-06-06

适用范围：`lpp_mobile`

## 1. 读取顺序

所有 APP 任务先读：

1. `AGENTS.md`
2. `docs/README.md`
3. `docs/constraints/01-通用开发约束.md`
4. 按任务进入 `requirements/`、`technical/`、`release/` 或 `constraints/`

涉及接口、权限、客服工作台、通话、发布或测试验收时，还必须读：

1. `docs/05-服务端支持.md`
2. `docs/constraints/05-测试与验收规则.md`
3. `docs/release/01-发布检查清单.md`

涉及 IM、Gateway、未读、已读、发送、在线客服线程、客服接入或消息底座时，还必须读：

1. `docs/technical/02-消息与会话详细方案.md`
2. `docs/03-APP端功能矩阵.md`

## 2. 协作硬规则

1. 不把聊天结论只留在对话里；影响需求、接口、设计、验证或发布判断的内容必须落文档。
2. 不机械复制 PC 文档；APP 只吸收路由、状态和约束方法。
3. 不扩大到移动端未要求的复杂管理能力。
4. 不把未联调、未真机、未多账号验证的能力写成已完成。
5. 修改代码前先确认需求状态和设计状态；不满足准入时先补文档或说明阻塞。
6. 发现接口异常时提供 endpoint、method、status code、server code、message、requestId、脱敏请求和完整响应。

## 3. 输出要求

- 方案讨论要说明产品目标、角色边界、状态闭环、服务端依赖、技术落点和验证方式。
- 实现完成要说明改了什么、验证了什么、未验证什么、剩余风险是什么。
- 发布判断必须引用 `release/` 和未实现清单，不凭主观感觉给结论。
