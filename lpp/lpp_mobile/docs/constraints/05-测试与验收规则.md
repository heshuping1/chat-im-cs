# APP 测试与验收规则

状态：当前有效

日期：2026-07-01

适用范围：`lpp_mobile`

## 1. 验收原则

APP 验收不只看能运行，还要证明真实账号、真实空间、真实服务端、Android 真机、弱网、多角色和多端关键路径可用。

正式版验收必须同时覆盖 Android 和 iOS。每次正式版、候选正式版或灰度正式版发布前，都必须按 `../release/04-正式版测试用例.md` 执行，并按 `../release/05-正式版测试报告模板.md` 产出测试结果报告。

## 2. 默认验证

| 改动类型 | 默认验证 |
| --- | --- |
| 纯文档或方案 | 文档链接、状态口径和工作区状态检查。 |
| 单组件 UI 或低风险逻辑 | `flutter analyze --no-fatal-infos` 或定向 `flutter test`。 |
| Provider / UseCase / Repository | 定向测试对应文件或目录。 |
| 入口、权限、token、消息、客服、缓存 | 升级到相关自动化和接口联调。 |
| 推送、通话、媒体、权限、后台 | Android 真机专项，必要时双真机。 |
| 正式版、候选正式版、灰度正式版 | Android+iOS 双端正式版测试用例、测试报告、问题台账和发布判断。 |

## 4. 正式版测试报告要求

每次正式版测试报告必须记录：

- Android 和 iOS 设备型号、系统版本、App 版本、构建类型。
- 测试账号、角色、空间、网络环境和服务端环境。
- 自动化命令、构建命令、真机操作结果。
- `通过`、`失败`、`阻塞`、`不适用` 的用例结果。
- 失败和阻塞项的问题编号、复现步骤、实际结果、预期结果、证据和负责人。
- requestId、traceId、截图、录屏、日志或命令输出路径。
- 发布结论：`可发布` / `不可发布` / `带风险发布` / `仅限灰度`。

失败、阻塞、重复出现或延期接受的问题必须同步登记到 `../release/06-发布问题跟踪台账.md`。同根因重复问题必须沿用原问题编号。

## 5. 真机记录要求

每次真机验收必须记录：

- 设备型号和 Android/iOS 系统版本。
- App 构建类型和版本。
- 测试账号、角色、空间。
- 网络环境。
- 执行时间。
- 通过项、失败项、日志、requestId。
- 截图或录屏路径。
