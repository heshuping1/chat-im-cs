# APP 通用开发约束

状态：当前有效

日期：2026-06-06

适用范围：`lpp_mobile`

## 1. 硬规则

1. 所有新功能必须先有需求记录。
2. 所有新功能必须先有技术设计。
3. `需求状态 = 完成` 且 `设计状态 = 完成`，才允许开发。
4. 不允许绕过既有分层直接写页面逻辑。
5. 不允许为了赶进度跳过服务端 API 支持确认。
6. 不允许新增需求只写代码不更新功能矩阵。
7. 不允许把临时方案写成正式实现；临时方案必须标记风险、owner 和后续任务。
8. 不允许用 mock、sample、demo、fake 或硬编码业务数据冒充真实会话、客户、权限、队列或未读。
9. 不允许在 `build()` 中触发 API 请求或状态变更。
10. 不允许把 PC 专属能力写成 APP 已完成能力。

## 2. 默认工程口径

- Android 真机是当前优先运行和验收目标。
- 新增状态必须确认 owner、生命周期、持久化 key、`spaceId` 隔离和回滚策略。
- 新增依赖必须确认现有 owner、包体积、许可证、真机影响和替换隔离层。
- 日志、报告、诊断包和 API 交接默认脱敏 Authorization、token、cookie、password、verification code。
- 服务端权限校验是最终边界；APP 只做入口展示、体验控制和前置提示。

## 3. 文档同步

功能变化必须同步：

1. `requirements/01-APP端需求池.md`。
2. `03-APP端功能矩阵.md`。
3. `technical/` 对应方案。
4. `05-服务端支持.md`。
5. `release/` 对应发布和回归记录。
