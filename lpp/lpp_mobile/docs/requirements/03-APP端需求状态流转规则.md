# APP 端需求状态流转规则

状态：当前有效

日期：2026-06-06

适用范围：`lpp_mobile`

## 1. 目标

本文件让需求从想法到开发、验收、发布形成闭环，避免只写代码、不更新矩阵、不确认服务端、不留验证证据。

## 2. 状态流转

```text
草稿 -> 评审中 -> 完成 -> 开发中 -> 待验收 -> 已完成
             \-> 暂缓
完成/开发中/待验收 -> 变更中 -> 评审中
```

## 3. 状态定义

| 状态 | 进入条件 | 退出条件 |
| --- | --- | --- |
| 草稿 | 只有需求方向、用户反馈或口头想法。 | 补齐目标角色、场景、范围和不做范围。 |
| 评审中 | 产品、设计、接口、权限或验收仍需确认。 | 需求和技术设计均完成，服务端依赖明确。 |
| 完成 | 需求、状态、异常、接口、验收标准清楚。 | 进入开发，或被标记为变更中/暂缓。 |
| 开发中 | 满足开发准入并开始实现。 | 完成实现和最小验证。 |
| 待验收 | 代码完成但真机、接口、多账号或发布回归未关闭。 | 验收通过或发现问题回到开发中/变更中。 |
| 已完成 | 验收标准满足，证据位置明确。 | 后续变更时进入变更中。 |
| 变更中 | 已完成或开发中需求发生范围变化。 | 重新评审并回到评审中或暂缓。 |
| 暂缓 | 本版本不做或等待长期依赖。 | 产品重新启动并回到评审中。 |

## 4. 准入检查

开发前逐项确认：

1. 需求池有记录。
2. 需求状态是 `完成`。
3. 设计状态是 `完成`。
4. 功能矩阵已有对应行。
5. 服务端 API 支持已确认或阻塞已登记。
6. 验收范围已明确。
