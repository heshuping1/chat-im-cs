## 1. 目的

本目录用于支撑 APP 端继续研发、需求讨论、技术方案讨论、交付验收和历史追溯。它不是 PC 端文档镜像，只吸收 PC 文档里适合 AI 协作的组织方式：入口清楚、状态闭环、分层约束、验证可追踪。

后续不要把影响需求、实现、接口、验证或发布判断的结论只留在聊天记录里；必须落到对应文档。

## 5. 新需求闭环

新增需求按这个顺序处理：

1. 在 `requirements/01-APP端需求池.md` 登记需求。
2. 补齐 `需求状态` 和 `设计状态`。
3. 只有 `需求状态：完成` 且 `设计状态：完成`，才允许进入开发。
4. 在 `03-APP端功能矩阵.md` 新增或更新功能行。
5. 如涉及接口、字段、事件、权限或错误码，更新 `05-服务端支持.md`。
6. 如涉及模块边界、数据流或技术策略，更新 `04-APP端技术方案总览.md` 或 `technical/` 对应详细方案。
7. 实现后补最小测试和验证命令。
8. 发布前更新 `06-APP端发布文档.md` 和 `release/03-版本变更记录.md`。

开发准入硬规则：

1. `需求状态` 可选值：`草稿` / `评审中` / `完成` / `变更中` / `暂缓`。
2. `设计状态` 可选值：`草稿` / `评审中` / `完成` / `变更中` / `暂缓`。
3. 只有 `需求状态：完成` 且 `设计状态：完成` 可以进入开发。
4. 需求或设计进入 `变更中` 后，相关功能必须暂停新增实现，只允许明确的 P0/P1 修复或回滚。
5. 如果确认不需要新增技术方案，也必须把 `设计状态` 标为 `完成`，并说明沿用的方案文档。

## 6. 当前 APP 结论

- 当前优先运行和验收端是 Android 真机。
- APP 是移动端 IM + 在线客服轻量工作台，不承载 PC 端三栏客服指挥台、完整交易表、质检审计大屏、批量运营报表和桌面能力。
- APP 当前具备登录、空间、会话、聊天、好友、通讯录、群、客服工作台、客户管理摘要、通话、设置、企业和公告等代码入口。
- 自动化与单机真机已有一定证据，但完整多角色、多账号、多端、推送后台、在线客服端到端、admin token 真接口专项和大数据性能仍未关闭。
- 发布判断仍是“不建议直接交付”，除非 S0/S1 风险完成处理或被明确延期确认。
