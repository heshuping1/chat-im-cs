# APP 正式版测试用例

状态：当前有效

日期：2026-07-01

适用范围：`lpp_mobile` Android / iOS 正式版、候选正式版、灰度正式版

## 1. 目的

本文件定义每次正式发版必须执行的 Android 和 iOS 回归测试用例。执行结果必须记录到 `05-正式版测试报告模板.md` 生成的当次测试报告；失败、阻塞、反复出现或延期接受的问题必须登记到报告的问题清单，并同步到 `06-发布问题跟踪台账.md`。

## 2. 执行规则

1. 正式版不得只测 Android 或只测 iOS；两端都必须有测试结果。
2. 每条 P0 用例必须记录结果：`通过` / `失败` / `阻塞` / `不适用`。
3. `不适用` 必须写明原因，例如本版本未启用通话或推送。
4. 任一 S0 或 S1 问题未关闭时，默认不能发布。
5. 若业务决定带问题发布，必须在测试报告写明风险接受人、影响范围、回滚方案和下一次回归版本。
6. 所有失败或阻塞项必须有问题编号，格式为 `APP-REL-YYYYMMDD-NNN`。
7. 自动化测试结果、真机截图、录屏、日志、requestId、traceId 必须写入测试报告证据区。

## 13. 结果记录

每次正式版测试报告必须至少包含：

| 字段 | 要求 |
| --- | --- |
| 用例编号 | 使用本文档编号，例如 `AUTH-005` |
| 平台 | Android / iOS |
| 结果 | 通过 / 失败 / 阻塞 / 不适用 |
| 问题编号 | 失败或阻塞必填 |
| 证据 | 截图、录屏、日志、requestId、traceId、命令输出 |
| 备注 | 风险、延期原因、复测说明 |

失败或阻塞的问题必须同步进入 `06-发布问题跟踪台账.md`。重复出现的问题沿用原问题编号，不新建编号；若根因不同，必须在原问题中说明拆分原因并登记新编号。
