# APP 发布问题跟踪台账

状态：当前有效

日期：2026-07-01

适用范围：`lpp_mobile` Android / iOS 正式版发布问题跟踪

## 1. 目的

本台账用于长期跟踪正式版测试中发现的问题，避免同类问题反复出现却只停留在聊天记录或单次报告里。每次正式版测试报告中的失败、阻塞、重复出现或延期接受问题，都必须登记到本台账。

## 2. 编号规则

| 字段 | 规则 |
| --- | --- |
| 问题编号 | `APP-REL-YYYYMMDD-NNN`，例如 `APP-REL-20260701-001` |
| 重复问题 | 沿用首次登记的问题编号，不创建新编号 |
| 根因不同 | 可以新建编号，但必须在原问题备注中说明拆分原因 |
| 关联用例 | 使用 `04-正式版测试用例.md` 中的用例编号 |
| 关联报告 | 使用当次测试报告路径，例如 `reports/mobile/release/20260701_1.0.0/summary.md` |

## 3. 严重级别

| 级别 | 定义 | 发布影响 |
| --- | --- | --- |
| S0 | 崩溃、白屏、无法启动、无法登录、主要角色不可用、普通消息不可发送、严重数据串空间或权限越权 | 必须阻断发布 |
| S1 | P0 主链路失败、注册/客服/消息/空间核心异常、正式品牌错误、重要平台专项失败 | 默认阻断发布；带风险发布必须明确风险接受人 |
| S2 | P1 场景失败、局部体验明显异常、弱网/媒体/设置等非阻断问题 | 可灰度或延期，但必须登记复测计划 |
| S3 | 文案、样式、低风险兼容问题 | 可延期，需记录处理版本 |

## 4. 状态流转

```text
新增 -> 已确认 -> 修复中 -> 待回归 -> 已验证 -> 关闭
新增 -> 已确认 -> 延期接受 -> 待回归
新增 -> 无法复现 -> 关闭
```

状态说明：

| 状态 | 含义 |
| --- | --- |
| 新增 | 测试报告中首次登记，尚未完成确认 |
| 已确认 | 已确认可复现、影响范围和责任边界 |
| 修复中 | 已分配负责人，正在修复 |
| 待回归 | 修复已进入候选包，等待 Android/iOS 复测 |
| 已验证 | 复测通过，等待关闭 |
| 延期接受 | 明确带风险发布或后续版本处理 |
| 无法复现 | 当前版本无法复现，保留证据说明 |
| 关闭 | 问题已验证关闭 |

## 7. 发布报告同步要求

每次正式版测试报告必须同步填写：

1. 新发现问题：在报告“问题清单”登记，并追加到本台账。
2. 重复问题：报告中引用原问题编号，并在本台账“复测历史”追加记录。
3. 已修复问题：本台账状态改为 `待回归`，复测通过后改为 `已验证`，最终关闭。
4. 延期接受问题：报告和台账都必须写明风险接受人、影响范围、补救方案和目标修复版本。
5. 发布结论：若存在未关闭 S0/S1，报告必须给出 `不可发布` 或 `带风险发布`，不得留空。

## 8. 台账维护检查

| 检查项 | 要求 |
| --- | --- |
| 问题编号唯一 | 不允许两个不同问题共用编号 |
| 重复问题复用编号 | 同根因重复问题必须沿用原编号 |
| 证据完整 | 至少包含截图、录屏、日志、requestId 或命令输出中的一种 |
| 状态可追踪 | 每次状态变化都要有报告或复测历史支撑 |
| 发布可判断 | 当前未关闭 S0/S1 必须能一眼看出是否阻断发布 |
