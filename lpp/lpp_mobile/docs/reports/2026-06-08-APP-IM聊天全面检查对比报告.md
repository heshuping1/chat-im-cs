# APP IM 聊天全面检查对比报告

日期：2026-06-08  
范围：`lpp/lpp_mobile/**` 普通 IM 聊天、群聊、媒体消息、本地数据、Gateway 同步、在线客服聊天  
分级：L2 核心链路任务  
结论等级：当前具备较完整的 IM 底座，但尚未达到“顶级 IM 聊天工具 / 微信级闭环”标准；不建议直接进入大规模补功能，必须先修复 P0 数据隔离、待发队列、DDD 边界和核心测试缺口。

> 说明：本报告基于当前工作区代码与移动端文档检查生成。`lpp/docs/api-contracts/AGENTS.md` 在当前仓库中不存在，因此接口合同部分按 `client-api-reference.md` 与 `field-enum-reference.md` 直接核对。

## 1. 顶级 IM 产品验收口径

进入聊天页与会话页时，端上必须优先显示本地已有会话和消息；远端失败、弱网、重连中不能用整块 loading 覆盖已有内容。

发送消息时，文本、图片、视频、语音、文件都必须先本地乐观插入，拥有稳定 `clientMsgId`，失败后本地可见、可重试、跨重启不丢，重试不能产生重复消息。

多媒体发送不能被单条上传全局阻塞。用户一次选择多张图片/视频后，每条都应该立即上屏、独立上传、独立失败、独立重试。

离线、弱网、重连后，服务端快照、Gateway 增量、本地 SQLite、待发队列、已读账本必须有明确归约规则，且按 `spaceId + account/user` 隔离。

在线客服聊天必须和普通 IM 域隔离：`temp_session` 不进入普通会话列表；排队、接入、AI、人工、关闭、历史只读、转接和未读状态必须形成闭环。

## 2. 总体结论

当前 APP IM 不是从零开始，已经具备以下强底座：

- 本地 SQLite 按 `spaceId` 建库，存在会话、消息、媒体本地账本。
- 聊天页消息和会话列表已有 local-first 加载路径。
- `SendMessageUseCase` 已实现 `clientMsgId`、乐观插入、上传进度、失败状态、重试队列入口。
- Gateway + `/sync` 已接入，能够处理 `msg.new`、已读、撤回、客服 temp session 分流等关键事件。
- 消息实体已经覆盖 text、markdown、image、video、voice、file、contact card、call log、location、event。
- API 合同已经支持多数后续要补齐的微信级能力，包括视频、文件、草稿、搜索、转发、撤回、已读、语音转文字、翻译、在线状态、定时消息和在线客服工作台。

但当前还不能称为闭环，核心原因有四类：

- P0 数据隔离风险：待发队列是全局 Hive box，没有 `spaceId/userId` 维度；同步游标只按 `spaceId` 存，缺少 account/user 隔离。
- P0 DDD 边界不合格：`ChatPage` 直接发 Dio 请求，并承载普通 IM、客服状态机、翻译、媒体加载、建单聊等应用逻辑。
- P0/P1 功能体验缺口：相册不能选视频，多媒体发送串行阻塞，本地草稿离线闭环不足，客服链路仍有 Page 层状态机和轮询兜底。
- P0 测试门禁不足：核心链路没有看到足够覆盖弱网、重启、跨账号、Gateway 重连、发送队列、媒体失败恢复、客服会话状态机的自动化验收。

## 3. 代码与 DDD 检查

| 编号 | 严重级别 | 结论 | 证据 | 风险 |
| --- | --- | --- | --- | --- |
| DDD-01 | P0 | 待发队列不符合领域隔离 | `PendingMessage` 只有 `clientMsgId/conversationId/isGroup/type/body/retryCount/createdAt`，`PendingMessageQueue` 使用全局 Hive `pending_messages`，`flushAll()` 读取全部消息。见 `lpp/lpp_mobile/lib/features/chat/data/datasources/pending_message_queue.dart:20`、`:107`、`:129`、`:155` | 多租户、多账号或切换空间后，待发消息可能被当前空间的 token/repository 重发，造成串号、数据泄漏或错误投递 |
| DDD-02 | P0 | 同步游标缺少用户维度 | Gateway 同步游标使用 `sync_cursor_$spaceId`。见 `lpp/lpp_mobile/lib/features/chat/presentation/providers/gateway_provider.dart:539`、`:652` | 同一设备同一租户下切换账号时，后登录用户可能跳过增量消息，或产生不可解释的消息缺口 |
| DDD-03 | P0 | `ChatPage` 是核心业务 God Page | 页面内直接调用 Dio 建单聊、翻译、加载媒体、同步客服线程、构造客服发送 use case。见 `chat_page.dart:386`、`:927`、`:1207`、`:1508`、`:1578` | UI 层承载接口适配、应用编排和领域规则，后续补功能会持续扩大不可测复杂度 |
| DDD-04 | P1 | 客服状态机没有沉到应用/领域层 | 客服线程同步、是否需要人工接入、关闭态发送限制、客服发送组装散落在 `ChatPage` | 队列、AI、人工、关闭、历史只读、转接等状态难以统一验证，容易和普通 IM 逻辑互相污染 |
| DDD-05 | P1 | 草稿不是本地优先 owner | `ChatInputToolbar._saveDraft()` 走 `ConversationActionsController.saveDraft()`，主要是远端动作。见 `chat_input_toolbar.dart:177` | 离线输入或远端失败时，草稿可能无法稳定本地恢复；新会话无本地 conversation row 时风险更高 |
| DDD-06 | P1 | 媒体运行时能力有沉淀，但 Page 仍绕过 owner | `MediaOpenController/MediaDownloadService/MediaLocalStore` 已存在，但 `ChatPage._loadMediaBytes()` 仍直接用 Dio 拉取媒体。见 `chat_page.dart:927`、`:934` | 鉴权、缓存、失败、下载进度和路径安全策略可能出现两套逻辑 |
| DDD-07 | P1 | 客户端策略不能替代服务端安全 | `MessageSendPolicy` 已在端上拦截 `.exe` 和企业客服链接。见 `message_send_policy.dart:12` | 端上策略可绕过，真正安全限制仍需服务端合同保证；端上只应做体验与前置校验 |

## 4. 核心链路闭环检查

| 链路 | 当前状态 | 是否闭环 | 主要问题 |
| --- | --- | --- | --- |
| 会话列表首屏 | 先本地、后远端 | 基本闭环 | `hidden_conversations_$spaceId` 有空间隔离，但整体仍需补账号维度验收 |
| 聊天页首屏消息 | 先本地、后远端 | 基本闭环 | 远端补洞依赖 `/sync` 和最近页同步，长时间离线需要专项压测 |
| 文本发送 | 乐观插入、失败、重试基础存在 | 部分闭环 | 待发队列隔离 P0 未闭环 |
| 图片发送 | 本地预览、上传、发送基础存在 | 部分闭环 | 多图串行发送，后续图片不能立即独立上屏 |
| 视频发送 | 实体/API 支持 | 未闭环 | 相册入口当前使用 `pickMultiImage()`，不能选视频 |
| 文件发送 | 文件选择、上传、失败状态基础存在 | 部分闭环 | 待发队列和本地失败恢复仍受 P0 影响 |
| 语音消息 | 发送与语音转文字 API 有基础 | 部分闭环 | 需补真实设备录音权限、弱网、失败恢复和转文字验收 |
| 已读/未读 | 本地 markRead + Gateway/read API 基础存在 | 部分闭环 | 需要跨端、重连、离线后读账本专项验收 |
| 撤回/删除/转发/收藏 | API 支持，端上已有部分入口 | 部分闭环 | 需要按消息类型和失败态补完整验收 |
| 搜索 | 本地索引与搜索 API 均存在 | 部分闭环 | 需要明确“本地优先 + 远端补充”的产品策略和空态/失败态 |
| 媒体缓存/打开 | 媒体账本、下载、启动恢复基础存在 | 部分闭环 | 存储治理入口和 Page 绕过媒体 owner 需收敛 |
| Gateway 重连增量 | SignalR + `/sync` 基础存在 | 部分闭环 | 同步游标缺 user/account 隔离；离线大增量需专项验收 |
| 在线客服会话 | temp session、claim/takeover/close API 与部分 UI 存在 | 未完全闭环 | 状态机在 Page 层，端到端接待/转接/关闭/历史只读仍需验收 |
| Push 点击回会话 | 文档列为重点待支持 | 未闭环 | 需要服务端 Push payload、端侧路由和本地补洞联调 |

## 5. 对标微信能力与 API 支持矩阵

### 5.1 API 已支持，端上应补齐或重构的能力

| 能力 | 当前端上状态 | API 是否支持 | 优先级 | 建议 |
| --- | --- | --- | --- | --- |
| 相册选择视频并发送 | 相册入口只 `pickMultiImage()` | 是，`messageType=video` + `/media/upload` | P0 | 改为统一媒体选择模型，图片/视频同批选择后都生成本地资源 |
| 多图/多视频独立发送 | `_handleSendMedia()` 顺序 `await` 每个 item | 是，客户端编排即可 | P0 | 一次选择后每条立即乐观插入，上传/发送独立并发或受控并发 |
| 待发队列跨重启可靠重试 | 有 Hive 队列，但无 space/user 隔离 | 是，依赖 `clientMsgId` 幂等 | P0 | 重建 `PendingMessageQueue` key：`spaceId + userId + clientMsgId`，补迁移与过滤 |
| 同步游标账号隔离 | 当前 `sync_cursor_$spaceId` | 是，客户端本地 key 调整即可 | P0 | 改为 `sync_cursor_${spaceId}_${userId}`，旧 key 迁移需保守处理 |
| 本地草稿离线恢复 | 主要远端 saveDraft | 是，`/drafts`、会话 draft API | P1 | 本地 draft ledger 先写，后台同步远端；新会话也应可恢复 |
| 会话媒体/文件面板 | 文件 API 与本地媒体表存在 | 是，`/direct-chats/{chatId}/files`、`/groups/{groupId}/files` | P1 | 做“图片/视频/文件”聚合页，本地优先，远端补齐 |
| 存储空间管理 | `ChatStorageManager` 存在，入口未闭环 | 客户端能力即可 | P1 | 设置页增加聊天文件占用、按会话清理、保留本地消息索引 |
| 已读回执详情 | direct/group API 支持 | 是，`read-status`、`read-receipts` | P1 | 单聊显示对端已读位置，群聊显示已读/未读成员详情 |
| 在线状态 | Presence API 和 Gateway 事件支持 | 是，`presence/*`、`presence.changed` | P1 | 会话头、成员列表、客服工作台统一展示策略 |
| 客服 AI 建议/快捷回复 | API 支持，页面内有本地启发式建议痕迹 | 是，客服 workbench AI/quick reply APIs | P1 | 替换 Page 内启发式建议为服务端建议 + 人工确认发送 |
| 定时消息 | API 支持，端上需完整核验 | 是，`/scheduled-messages` | P2 | 补待发列表、取消、失败提醒、发送后本地归约 |
| 位置消息 | 实体/API 支持，但端上发送关闭 | 是，`messageType=location` | P2 | 需先确定地图/隐私/权限/审核方案，再启用生产入口 |
| 翻译与语音转文字 | API 支持，端上有入口但部分逻辑在 Page | 是 | P2 | 下沉到 application controller/use case，统一错误与缓存 |

### 5.2 当前 API 不足，不能直接承诺端上补齐的微信能力

| 能力 | API 状态 | 结论 |
| --- | --- | --- |
| 实时 typing indicator | `typing` 是占位接口，不做实时转发 | 不能作为微信级实时输入中能力交付，需服务端事件支持 |
| 表情回应/reaction | 未在当前合同看到标准 reaction API | 暂不列为端上可直接补齐项 |
| 红包/转账 | 未在当前合同看到支付/红包 API | 不属于当前 IM 可补齐范围 |
| 贴纸商店/自定义表情云同步 | 未看到贴纸资源与同步合同 | 需产品和 API 另立需求 |
| 原图/压缩图双资源策略 | 当前媒体合同未明确 original/compressed 双资源 | 可先做端侧体验，但不能声称完整微信级原图能力 |

## 6. 安全与漏洞风险

| 风险 | 严重级别 | 说明 | 修复方向 |
| --- | --- | --- | --- |
| 跨空间/跨账号待发消息串发 | P0 | 全局 `pending_messages` 会在当前空间重连时 flush 全部 pending | 队列实体和 key 增加 `spaceId/userId`，flush 只处理当前身份 |
| 同步游标跨账号污染 | P0 | `sync_cursor_$spaceId` 缺少 user/account | 游标、Gateway 状态、本地恢复统一加入 account/user 维度 |
| UI 层直接接口调用扩大攻击面 | P1 | `ChatPage` 多处直接使用 Dio，绕过统一 repository/service 错误处理 | 所有聊天动作进入 application controller/use case |
| 客户端安全策略可绕过 | P1 | `.exe`/链接限制只在端上无法构成安全边界 | 服务端也必须校验，端上保留即时反馈 |
| 本地敏感数据保护未闭环 | P1 | 待发队列、草稿、媒体本地路径、消息数据库可能包含敏感内容 | 明确加密、备份排除、登出清理和企业空间数据保留策略 |
| 第三方地理编码路径不适合生产 | P2 | 位置发送入口虽关闭，但 Page 中仍有外部 geocoding 痕迹 | 生产启用前迁移到合规地图服务/后端代理 |

## 7. 测试与验收缺口

当前不能只用 `flutter analyze` 或少量 widget 测试作为 IM 核心链路验收。后续至少需要以下测试：

| 测试项 | 优先级 | 必须覆盖 |
| --- | --- | --- |
| 待发队列隔离测试 | P0 | A 空间失败消息不会在 B 空间重连时发送；同 space 不同 user 不串发 |
| 同步游标隔离测试 | P0 | 同一设备切换账号后不会跳过新账号消息 |
| 文本/图片/视频/文件发送生命周期 | P0 | optimistic、uploading、sent、failed、retry、app restart 恢复 |
| 多媒体同批发送 | P0 | 多条立即上屏，单条失败不影响其他条 |
| 本地首屏与远端失败 | P0 | 有本地消息时远端失败不遮挡、不清空 |
| Gateway 重连补洞 | P0 | 离线期间多条消息、撤回、已读、客服事件重连后归约正确 |
| 草稿本地优先 | P1 | 离线输入、退出、重启、远端同步失败后仍恢复 |
| 客服状态机 | P1 | 排队、接入、AI 接管、人工接管、关闭、历史只读、转接、未读 |
| 媒体缓存与清理 | P1 | 下载中断、重启恢复、鉴权失败、清理后可重新下载 |
| Push 点击回会话 | P1 | 冷启动/热启动进入会话，本地先显示，后台补洞 |

## 8. 后续重构与补功能路线

### 阶段 0：先修 P0 数据正确性

1. 重构 `PendingMessageQueue`：实体补 `spaceId/userId`，Hive key 改为 `spaceId:userId:clientMsgId`，flush 按当前身份过滤。
2. 重构 Gateway sync cursor：从 `sync_cursor_$spaceId` 改为 `sync_cursor_${spaceId}_${userId}`，补迁移和回退策略。
3. 给 pending queue、sync cursor、local-first、重启恢复补自动化测试。

### 阶段 1：DDD 边界收敛

1. 拆 `ChatPage`：页面只负责渲染和转发事件。
2. 新增/收敛 application controller：
   - `DirectChatEntryController`：建/复用单聊。
   - `MessageTranslationController`：翻译与缓存。
   - `ChatMediaController`：媒体打开、下载、鉴权、本地账本。
   - `CustomerServiceChatController`：客服线程状态机、发送、关闭、历史只读。
   - `DraftController`：本地草稿和远端草稿同步。
3. Page 不再直接 import Dio，不直接组装 datasource/usecase。

### 阶段 2：补齐 API 已支持的微信级体验

1. 相册支持视频，图片/视频统一 `ChatPickedMedia`。
2. 多媒体同批独立乐观插入，受控并发上传，失败独立重试。
3. 本地草稿离线恢复，远端草稿后台同步。
4. 会话媒体/文件面板，按图片、视频、文件聚合。
5. 已读回执详情、在线状态、定时消息管理补齐 UI 与验收。

### 阶段 3：在线客服闭环

1. 客服状态机从 Page 下沉到专用 controller/use case。
2. 接入、排队、AI、人工、转接、关闭、历史只读、评价、未读统一建模。
3. 替换本地启发式 AI 建议为服务端 workbench AI/quick reply API。
4. Gateway temp session 事件、CS 列表、聊天页和通知统一归约。

### 阶段 4：交付门禁

1. Android 真机弱网、断网、重连、杀进程恢复测试。
2. 跨账号、跨 `spaceId`、跨客服线程隔离测试。
3. `flutter analyze --no-fatal-infos`、核心自动化脚本、debug APK 构建。
4. 发布检查清单更新为“可交付”前，不得标记微信级闭环完成。

## 10. 阶段结论

当前 APP IM 的方向是正确的，底座也已经有一部分微信级能力的雏形；但真正阻塞“顶级 IM”的不是单个按钮或样式，而是核心数据 owner、身份隔离、发送队列、Gateway 游标和 Page 层业务堆积。

下一步应先按 P0 修复数据正确性，再按 DDD 收敛 Page 层职责。完成这两个基础后，再补相册视频、多媒体并发、草稿、本地媒体面板、客服状态机等能力，才能避免越补越复杂、越补越不可靠。
