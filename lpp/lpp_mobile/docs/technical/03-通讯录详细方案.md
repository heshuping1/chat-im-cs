# 通讯录详细方案

状态：当前有效

日期：2026-06-06

适用范围：`lpp_mobile`

## 1. 目标

通讯录为移动端提供好友、组织、群、客户和角色入口，强调快速查找、权限清晰和移动端可读性。不把客户管理、审计和完整 CRM 大表放进普通通讯录。

## 2. 模块边界

| 能力 | APP 落点 | 约束 |
| --- | --- | --- |
| 好友 | `features/contacts`、聊天入口 | 申请、同意、拒绝、重复申请和状态刷新必须闭环。 |
| 组织成员 | `features/contacts`、`features/space` | 按角色和企业空间展示，不泄露无权限成员信息。 |
| 客户入口 | `features/contacts`、`features/customer_service` | 客户只展示移动端必要摘要，不承载完整交易和审计。 |
| 群入口 | `features/chat`、`features/contacts` | 群资料和治理进入群功能方案。 |
| 搜索扫码 | contacts/profile/chat 相关入口 | 二维码有效期、重复申请、无权限和失效态必须明确。 |

## 3. 分层要求

- Page 只负责展示好友、组织、客户和群入口。
- 搜索、申请、审批、刷新流程放到 Controller/UseCase。
- 好友关系状态、是否可申请、是否可聊天等规则放 Domain。
- DTO、分页、缓存和接口兼容放 Data/Repository。

## 4. 验收重点

| 场景 | 验收 |
| --- | --- |
| 搜索用户 | 正常结果、空结果、无权限、接口失败。 |
| 好友申请 | 发送、重复申请、同意、拒绝、双方刷新。 |
| 组织通讯录 | 多角色可见范围、部门大数据、成员详情字段。 |
| 客户入口 | 客户/员工/客服/管理员角色差异和脱敏。 |
| 扫码 | 有效、过期、无权限、重复添加。 |
