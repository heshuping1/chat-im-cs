# 消息与会话详细方案

状态：当前有效

日期：2026-06-07

适用范围：`lpp_mobile`

## 1. 目标

APP 消息系统同时支持普通 IM 和在线客服，但两者入口、权限和状态必须清晰分离。

## 2. 会话类型

| 类型 | APP 处理 |
| --- | --- |
| `direct` | 普通单聊，进入消息首页。 |
| `group` | 普通群聊，进入消息首页。 |
| `temp_session` | 在线客服访客临时会话，只进入客服工作台。 |
| `direct_customer` / `im_direct` | 注册客户与客服直连 IM，按普通消息或客户会话入口处理。 |

客服会话和普通 IM 在业务域上保持隔离；发送生命周期、媒体上传、视频封面、待发恢复、本地消息持久化等端侧消息能力必须复用同一套实现，不能在客服页面单独复制媒体上传逻辑。

## 3. 核心不变量

- `temp_session` 不进入普通消息首页。
- 客服未接入、AI 未接管、历史终态、管理只读时不能发送。
- 自己发送的消息不产生未读。
- Gateway 重连后做增量同步，不全量重拉历史。
- 发送必须使用 `clientMsgId` 乐观插入。
- 服务端拒绝不能本地显示成功。
- 端侧发送策略拒绝不能进入上传、发送接口或待发队列；必须保留本地 failed 消息，但用户侧只展示通用发送失败，不提示具体策略原因。
- 客户画像缺字段不得展示假数据。

## 4. 发送闭环

```text
输入 -> 生成 clientMsgId -> 本地乐观插入 sending -> 调发送接口 -> 成功替换服务端字段 -> 失败保留 failed/可重试
```

媒体发送入口：

- 相册入口必须支持图片和视频选择，选择结果先进入输入栏 presentation model，再由页面分派为 `MessageType.image` 或 `MessageType.video`。
- 照片入口必须使用图片专用选择器，仅展示和发送图片；若系统仍误返回 `xml/json/md/log/rtf/txt/pdf/zip` 等非图片文件，端侧按不支持类型提示，不按照片入口发送文件。
- 文件入口必须使用非媒体附件扩展名过滤，允许文档、压缩包、音频、配置和日志类文本文件，不展示 `jpg/png/mp4/mov` 等图片视频媒体。
- 拍摄入口必须直接进入单一拍摄页，遵循微信式交互：轻点快门拍照，长按快门录视频，录屏不在聊天拍摄入口范围内。
- 发送侧本地图片发送后必须第一时间插入消息流，先展示可识别的本地预览，并在图片上以轻量覆盖层体现上传/发送进度；上传成功替换远端资源时，端侧必须保留 `localPreviewUrl`，避免远端图加载期间回退灰色占位。
- 发送侧本地视频发送后必须第一时间插入消息流；发送前应生成端侧封面图，展示层优先使用本地封面，再回退服务端 `thumbnailUrl`，最后才显示视频占位，并在视频气泡上以覆盖层体现上传/发送进度。
- 图片和视频气泡参考微信式尺寸收敛：图片按资源真实宽高比展示，在移动端最大宽高内约束，不固定裁成正方形；视频默认按 16:9 紧凑比例展示。视频缺少 `localPosterUrl` / `thumbnailUrl` 时，端侧优先基于本地或已缓存视频生成首帧封面，失败后只显示浅色视频占位，不能出现黑屏式大色块。
- 文件上传/发送进度参考 PC 端内嵌方案：文件在文件卡右侧图标内展示圆弧进度；不使用消息外侧发送圆圈、不整体置灰、不恢复底部进度条。本轮只做进度展示，不提供暂停、继续、取消控制。
- 文本和 Markdown 消息发送后立即乐观上屏，`sending` 期间不展示外侧圆形进度；发送失败或被拒绝时再展示失败入口，允许用户重试。
- 群聊中仅自己发送且服务端确认、未撤回的消息展示已读回执入口；消息流内联数字只使用服务端 `readCount`，详情页实时拉服务端已读/未读名单，不通过 `msg.read` 或本地事件累加群已读人数。
- 文件消息禁止发送 `.exe` / Windows 可执行文件；即使选择器入口未暴露该类型，发送用例仍必须按文件名、本地路径和 MIME 做兜底拦截。
- 企业客户/外部用户禁止发送链接；个人空间、企业员工、客服、管理员不受该端侧链接策略限制。链接检测覆盖 `http://`、`https://`、`www.` 和常见裸域名，邮箱地址不作为链接拦截。
- 发送交互参考微信：图片、视频、文件、语音和文本不使用聊天页全局发送锁；一条媒体正在上传时，用户仍可继续发送下一条媒体或文本。每条消息独立维护上传/发送状态；位置选择、失败重发等状态敏感单动作可以保留重复点击保护。
- 发送侧上传中的图片、视频和文件不允许等传输完成后才显示；确认成功后替换服务端字段，视频继续保留 `localPosterUrl` 作为封面兜底。
- 本地消息库必须使用端侧持久化序列化保存 `localPreviewUrl`、`localPosterUrl` 等本地展示字段；接口请求仍使用 API 序列化，不能把本机路径传给服务端。
- 发送失败且属于可恢复网络/上传失败时，待发队列必须保存原始本地媒体 body；网络恢复重发前先上传本地媒体，视频还要上传本地封面并写入远端 `thumbnailUrl`。
- 待发队列恢复成功后必须回写本地消息库并按 `clientMsgId` 替换旧乐观/失败消息，避免重进聊天出现失败消息和成功消息重复。
- 手动重发失败消息必须沿用原 `clientMsgId`，不能重新生成发送 ID。
- 私聊自己发送的消息已读回执以服务端 `msg.read` 和 `GET /api/client/v1/direct-chats/{chatId}/read-status` 的对端已读游标为准；端侧只覆盖 `conversationSeq <= peerLastReadSeq` 的自发服务端消息，并写回本地消息库；当前会话存在待对端已读的自发消息时，短窗口内轻量刷新 read-status，不用 UI 临时状态冒充已读事实。
- 服务端相对路径如 `/media`、`/api`、`/uploads`、`/files` 必须按远端资源处理，不得误判为本地文件路径导致预览卡住或重复上传。
- 接收侧不展示发送方本地上传半成品和上传进度；只有服务端/Gateway 已下发完整消息后才进入消息流，端侧再按已收到的远端资源展示缩略、占位、下载或打开状态。
- 聊天本地消息库、会话快照、发送队列、媒体本体、本地搜索索引、存储治理和启动恢复按 `technical/06-APP微信级本地数据方案.md` 执行；其中接收侧图片、视频、文件、语音列表只展示预览，查看/播放/打开时通过 SQLite 媒体账本、持久文件目录和 Dio 下载器按需下载，已下载媒体 App 重启后必须可复用。
- 上传、文件读取、权限和平台能力归 runtime/service 层或既有平台能力封装处理，UI 不直接解释服务端 DTO。

## 5. Gateway 闭环

```text
connected -> disconnected/reconnecting -> 网络恢复 -> reconnect -> 增量 sync -> 合并会话/消息/已读
```

## 6. 聊天资料页检索

- 单聊和群聊资料页的“查找聊天内容”必须进入会话内搜索，不得退回全局搜索。
- 会话内搜索顶部参考微信：左侧不显示返回箭头，搜索输入框右侧显示“取消”，点击取消返回资料页。
- 会话内搜索空态展示快捷查找：日期、图片及视频、文件、链接；这些快捷入口只在当前 `conversationId` 范围内生效。
- 消息搜索接口使用 `/api/client/v1/search/messages`；会话内搜索必须传 `conversationId`，关键字为空时允许按快捷类型进入结果页，端侧再按消息类型或日期做轻量过滤。
- 会话内搜索结果必须展示真实内容：文本/链接展示正文摘要，文件展示真实文件名和大小，图片展示缩略图，视频展示封面、播放图标和时长；不得用 `[消息]`、`[文件]`、`[视频]`、`[图片]` 作为结果内容。
- 图片及视频快捷入口使用媒体网格展示，图片优先 `thumbnailUrl` 再 `url`，视频优先 `thumbnailUrl`；服务端未返回可展示资源时只显示轻量占位，不伪造历史封面。
- 搜索结果按用户时区做日期分组，标题显示 `今天`、`昨天`、`本周` 或 `yyyy年M月d日`，单条结果右侧保留具体时间。
- 全局搜索继续保留用户/消息 Tab，不展示会话内快捷查找。

## 7. 在线客服状态

```text
排队 -> 接入 -> 人工接待中 -> 关闭 -> 历史只读
AI 接待 -> 人工接管 -> 人工接待中
终态 -> 只读，禁止发送
```

客服 P0 会话增强：

- 客服角色边界对齐 PC 端：`membershipRole=2` 是坐席，可进入坐席工作台、接入/接管、回复、转接、关闭和客服静默撤回；`membershipRole=3/4` 是管理员/所有者，只能走 admin token 查看客服历史、绩效和实时监控，不能进入坐席发送、撤回、转接状态机；普通员工、客户和未知高角色不允许调用坐席或 admin 客服接口。APP 入口展示和禁用态统一消费 `customer_service` domain action permission，服务端权限仍是最终边界。
- 客服会话已读回执沿用普通 IM 的 `conversationSeq` 比较原则，但 owner 独立在 customer-service：`temp_session` 读取 `readStatus.members` 中客户/访客 `lastReadSeq/lastReadAt`，`im_direct/direct_customer` 读取 `peerLastReadSeq/peerLastReadAt` 或监控详情里的 `customerLastReadSeq/customerLastReadAt`；`lastReadAt=null` 展示为未知，不得当作 1970。
- 客服静默撤回只允许在线客服线程使用 `POST /api/client/v1/messages/{messageId}/recall-silent`。收到 `msg.recalled.silent=true` 时从本地消息流移除气泡并删除本地库记录；`silent=false` 继续走普通撤回占位，普通 IM 不得使用静默撤回。
- 客服转接按渠道区分端点：`temp_session` 使用 `/customer-service/temp-sessions/{sessionId}/transfer`，`im_direct/direct_customer` 使用 `/customer-service/im-direct/{threadId}/transfer`。转接备注只进入服务端时间线；客户礼貌提示由服务端系统消息下发，APP 不拼假系统消息。被转接方收到 `temp_session.transferred`/`customer_service.thread.transferred` 后进入工作台通知卡，可打开原线程历史预览并继续接待。
- 客户输入预览是瞬时 Gateway 信号，不落本地库、不进入消息历史，TTL 建议 5 秒；客服端可展示客户未发送文字，客户侧只允许展示“正在输入”指示，不展示客服草稿正文。
- 超时关闭续聊涉及 Widget `visitorToken` 和 `resumeRecentSession/reopen` owner；APP 若承载访客端入口，必须先确认 token 存储、会话重开和 Gateway 重连边界，不在坐席工作台内用临时状态伪造可发送。APP 坐席工作台仅承接服务端重开后的状态：终态展示历史保留与刷新状态入口，刷新到非终态详情后解除路由历史只读锁并恢复接待。

## 8. 管理端客服统计、监控与获客来源

- 管理员/所有者工作台使用 admin token 访问 `/api/admin/v1/customer-service/temp-sessions/stats`，只读展示客服绩效，不改变客服接待状态机、消息发送、已读、Gateway 或本地消息库。
- `staffPerformance` 顶层按客服本人合并 `widget` 与 `im_direct` 两类会话 KPI；`byChannel[]` 仅作为渠道下钻展示。APP 不把该接口当作实时排队/接待状态来源，实时概览仍使用既有客服中心 dashboard。
- `acquisition` 是访客/临时会话的结构化获客来源，属于客户画像/会话上下文展示字段；字段全可空，缺失时隐藏对应行，不用 mock 文案补齐。
- 访客来源展示只读，页面不得直接解释 raw DTO；Data/Model 层负责字段归一，Page 只消费领域模型中的展示项。
- 管理员/所有者实时监控使用 admin token 和 center threads 列表/详情，只读同步查看消息、已读位置和客服状态；列表优先请求 `/api/admin/v1/customer-service/center/threads` 并携带客服、状态、渠道和关键词筛选，详情读取 `/center/threads/{threadType}/{threadId}`。多窗口并排属于管理工作台展示状态，不得写入坐席发送队列或客服接待状态机。APP 监控墙的筛选/窗口容量/替换规则归 `customer_service` domain view-model，页面只消费模型结果并拉取 admin 详情。

## 9. 验收重点

| 场景 | 期望 |
| --- | --- |
| 普通会话列表 | 只展示 `direct/group`，不展示 `temp_session`。 |
| 文本发送 | 乐观插入，发送中不展示即时进度，成功替换，失败可见可重试。 |
| 媒体发送 | 立即插入本地可见气泡，本地预览和待发 body 入库，重进会话不出现空加载。 |
| 媒体接收 | Gateway/接口完整消息落库后再显示，端侧按远端缩略图/文件信息渲染。 |
| 已读未读 | 自发不提醒，进入会话清未读。 |
| 群已读回执 | 自己发送的群消息按服务端 `readCount` 展示“已读 N 人/未读”，详情页按服务端成员读游标分出已读和未读；弱网失败可重试，不出现本地猜测人数。 |
| Gateway 重连 | 恢复后增量同步，不重复不丢关键消息。 |
| 客服排队 | 未接入前只能接入，不能发消息。 |
| 角色边界 | 客服坐席可接待和操作会话；管理员/所有者只能管理只读监控，不显示发送、撤回、转接入口；客户和普通员工不能进入客服工作台接口。 |
| 客服已读 | 客服自己发送的消息按客户/访客 `lastReadSeq` 显示已读/未读和已读时间；`lastReadAt=null` 显示未知。 |
| 客服转接 | 当前接待客服可选择目标客服并填写备注；被转接方收到工作台通知并能预览历史，客户侧看到服务端系统提示。 |
| 客户输入预览 | 客服端显示客户正在输入的临时预览，5 秒无新事件自动清除，不写入历史。 |
| 客服静默撤回 | 客服会话内已发送消息撤回后客户端完全消失，不显示“已撤回”；普通 IM 仍显示撤回占位。 |
| 管理只读 | 管理员/所有者不能第三人发言。 |
| 超时续聊 | 访客端续聊入口重开原会话并保留历史；APP 坐席端在结束态展示历史保留与刷新状态，刷新到服务端非终态详情后恢复接待，不调用 Widget `reopen`。 |
| 实时监控 | 管理员/所有者按客服和状态筛选进行中会话，多窗口只读查看消息和客户已读状态。 |
| 管理客服绩效 | admin token 可读取客服绩效；顶层显示跨渠道合并 KPI，渠道下钻区分 `widget` 与 `im_direct`，统计延迟不影响实时接待状态。 |
| 访客获客来源 | 访客/临时会话存在 `acquisition` 时展示来源平台、设备、UTM 等结构化字段；字段缺失时隐藏，不展示假数据。 |
| 推送点击 | 恢复正确空间并跳到目标会话。 |
| 聊天资料页查找 | 输入框右侧为取消，快捷查找可见，请求携带当前 `conversationId`，结果显示真实内容和日期分组。 |
