# 群功能详细方案

状态：当前有效

日期：2026-06-06

适用范围：`lpp_mobile`

## 1. 目标

APP 群功能服务移动 IM 主链路：建群、聊天、群资料、成员管理、公告、二维码、申请和基础治理。复杂审计、批量治理和运营分析默认由 PC 或后续管理扩展承接。

## 2. 核心能力

| 能力 | 状态要求 |
| --- | --- |
| 创建群 | 支持选择成员、创建失败反馈、成功进入会话。 |
| 群资料 | 群名、头像、公告、二维码、备注、成员列表。 |
| 成员管理 | 添加、移除、群主、管理员、普通成员权限清晰。 |
| 禁言治理 | 全员禁言、单人禁言、权限拒绝码和 UI 禁用态一致。 |
| 群申请 | 申请、审批、拒绝、过期、重复申请和多端同步。 |
| 退出解散 | 普通成员退出，群主转让或解散，退出后同步会话状态。 |
| 已读回执 | 有服务端字段支持时展示；缺字段时不得伪造。 |
| @ 提醒 | 群聊文本输入 `@` 或工具面板 `@成员` 打开成员选择；支持 @成员 与 @所有人；发送、乐观消息、离线 pending、历史/Gateway 回包均保留 `mentions[]`。 |

## 3. 不变量

- 群权限规则不写在 Page。
- 禁言状态必须同时影响输入区、发送动作和服务端写接口结果。
- @所有人必须同时受群角色和 `settings.allowMemberAtAll` 控制；服务端仍是最终权限边界。
- @成员/@所有人的 offset/length 由领域纯函数计算，Page 不直接拼接口 DTO。
- 服务端拒绝不能本地显示成功。
- 群成员、群名、公告、禁言和申请变化需要通过 Gateway 或增量同步保持一致。
- 客户成员、企业成员和外部联系人的可见信息必须按权限脱敏。

## 4. 验收重点

| 场景 | 验收 |
| --- | --- |
| 多角色群治理 | 群主、管理员、普通成员、客户成员权限正确。 |
| 禁言 | 全员禁言、单人禁言、解除禁言和发送拒绝一致。 |
| 群申请 | 申请、审批、拒绝、重复申请、多端刷新。 |
| @ 提醒 | 普通成员 @成员可发送；无权限普通成员不可选择 @所有人；群主/管理员或开启 `allowMemberAtAll` 后可 @所有人；历史/Gateway 消息高亮 mentions。 |
| 退出/解散 | 会话列表、消息页、群详情和 Gateway 状态同步。 |
| 大群性能 | 成员分页、搜索、滚动和头像加载稳定。 |
