## 1. App 定位

- 交付物是 Flutter 移动端 App，当前直接运行和验收优先 Android 真机。
- App 不是 PC 客服三栏工作台，也不是线上 Web 站点。
- App 直接请求现有服务端 API，不能用本地 mock 或示例数据冒充真实业务状态。
- App 首要承载移动 IM、在线客服轻量接待、扫码、通话、通知、设置和轻量管理。
- 完整 CRM、交易/资金大表、质检审计、批量运营分析和桌面能力属于 PC/后续管理扩展。

## 2. 技术栈

- Flutter / Dart
- Riverpod 2
- GoRouter
- Dio + 项目已有 `HttpClient`
- `AdminHttpClient`
- SignalR / WebSocket
- Hive / SQLite / Flutter Secure Storage
- `flutter_webrtc`

## 3. 分层约定

| 目录 | 职责 |
| --- | --- |
| `lib/app` | 启动、路由、主题。 |
| `lib/core` | 网络、存储、数据库、空间、认证、权限、诊断、通知等基础设施。 |
| `lib/features` | 认证、聊天、通话、通讯录、客服、工作台、设置等业务模块。 |
| `lib/shared` | 共享组件、工具和扩展。 |
| `lib/l10n` | 多语言文案。 |
| `assets` | 静态资源。 |
| `android` / `ios` | 平台壳工程和权限配置。 |

## 4. 安全约束

- 服务端权限校验是最终边界，前端只做入口展示和体验控制。
- `platformToken` 只用于 `/api/platform/v1/*`。
- tenant `accessToken` / `refreshToken` 只用于 `/api/client/v1/*`、`/ws/client`、`/hubs/voicecall`。
- admin API 必须使用 admin token，并调用 `/api/admin/v1/*`。
- 不得用普通租户 token 调用 admin API。
- admin 401 时清理 admin token，并用 platform token 重新签发。
- 日志、报告、诊断包和 API 交接默认脱敏 Authorization、token、cookie、password、verification code。

## 5. 真实数据约束

- 运行时 UI 禁止使用 mock、sample、demo、fake、硬编码业务数据冒充真实会话、客户、企业、权限、队列、未读或客服接待信息。
- API、Gateway 或本地缓存未同步、缺字段或失败时，展示同步中、空态、错误态或 `--`。
- 测试 fixture、单测 mock、头像首字母、骨架屏、空态文案可以存在，但必须与真实业务链路隔离。

## 6. 移动体验约束

- App 任务必须按微信级移动 IM 体验处理。
- 重视会话流畅度、输入体验、触控反馈、弱网恢复、通知触达、媒体加载、权限引导、列表性能和移动端视觉克制。
- 不得只按功能可用标准完成。
- 文本必须在移动端完整显示，不允许遮挡。
- 远程图片优先使用项目已有鉴权图片组件或缓存图片方案。

## 7. 开发边界

- 不在 `build()` 中触发 API 请求或状态变更。
- 页面组件不得直接解释 Gateway raw payload。
- UI 层不得直接拼接复杂权限、发送、已读、客服状态机规则。
- 新增状态前确认 owner、持久化 key、空间隔离和回滚策略。
- 新增依赖前确认现有 owner、包体积、许可证、真机影响和替换隔离层。
