功能概述

用户可以让系统给自己生成一个「加我为好友」的二维码，分享出去后别人扫码即可向自己发起好友申请。

  • 服务端只发放一个 token 字符串，并不负责生成二维码图片。客户端拿到返回里的 qrPayload，用本地任意二维码库（qrcode、jsQR 等）渲染成图片即可。
  • 扫码人扫到码之后，是以自己的身份向发码人发起一条好友申请，发码人仍需要在好友申请列表里点「同意」。也就是说，二维码不会绕过任何隐私设置或黑名单——它只相当于一张「让对方知道我是谁」的电子名片。
  • 一张二维码可以设置有效期和使用次数。可以做成长期有效的个人名片码，也可以做成一次性的当面加码。

使用流程

  1. 用户 A 调用 POST /friends/invite-qr 生码，拿到 qrPayload。
  2. 客户端把 qrPayload 渲染成二维码图片展示出来。
  3. 用户 B 扫码后，客户端从二维码里提取 token（即 ?token= 后面那一串）。
  4. 客户端调用 GET /friends/invite-qr/{token}/preview 展示一下「这是 Alice，你确定要加吗」的确认页。
  5. 用户 B 点确认 → 客户端调 POST /friends/invite-qr/{token}/accept → 向 A 发起好友申请。
  6. 用户 A 在好友申请列表里同意 → 双方成为好友。

限流

生码和扫码加好友这两个写操作做了节流，按已登录用户算：5 分钟内最多 20 次。超出会返回 429。

① 生成二维码

POST /friends/invite-qr 限流

给当前用户发放一张「加我为好友」的二维码。可重复调用，生成多张码。

请求体（字段全部可选，整个 body 也可以不传）

字段类型默认说明
ttlHours整数168（7 天）有效期，单位小时。取值范围 1 ~ 720（最长 30 天）。超出会被自动截到边界。
maxUses整数0（不限）最多可被多少人扫码加成功。0 表示不限次数；取值范围 0 ~ 1000。
message字符串null对方扫码后默认带上的附言，最长 500 字符。扫码人可以覆盖。

响应 data（FriendInviteQrDto）

{
  "tokenId":   "019e1fff-764f-7a21-a0a8-50fa837eb26e",
  "token":     "A1B2C3D4E5F6A7B8",
  "qrPayload": "ztchat://friend-invite?token=A1B2C3D4E5F6A7B8",
  "maxUses":   0,
  "usedCount": 0,
  "message":   "扫码加我",
  "status":    "active",
  "expiresAt": "2026-05-20T06:00:00+00:00",
  "createdAt": "2026-05-13T06:00:00+00:00"
}
字段类型说明
tokenIdGUID这张码的内部 ID，撤销时使用。
token字符串真正的二维码内容（16 位十六进制）。一般客户端不直接展示，而是用下面的 qrPayload。
qrPayload字符串直接喂给二维码库渲染图片。格式形如 ztchat://friend-invite?token=XXXX。
maxUses整数最大可被接受的扫码次数。0 表示不限。
usedCount整数已经被成功扫码加好友的次数。
message字符串/null默认附言。
status字符串active=可用 / revoked=已撤销 / exhausted=用尽。
expiresAt时间戳过期时间。
createdAt时间戳创建时间。
提示 qrPayload 是一个自定义 scheme 的链接。客户端只需要把这个字符串完整编码进二维码即可。手机端扫码 App 不会直接识别 ztchat://，但 ZTChat 自己的客户端会拦截并跳转加好友页。

② 列出我发的二维码

GET /friends/invite-qr

列出当前用户名下仍然有效的二维码——已撤销 / 已用尽 / 已过期的不会出现在列表里，避免界面里堆死码。按创建时间倒序。

请求参数

无。

响应 data

数组，每个元素就是 ① 里的 FriendInviteQrDto。

[
  {
    "tokenId":   "019e1fff-...",
    "token":     "A1B2C3D4E5F6A7B8",
    "qrPayload": "ztchat://friend-invite?token=A1B2C3D4E5F6A7B8",
    "maxUses":   0,
    "usedCount": 3,
    "message":   "扫码加我",
    "status":    "active",
    "expiresAt": "...",
    "createdAt": "..."
  }
]

③ 撤销二维码

DELETE /friends/invite-qr/{tokenId}

把自己发出去的某一张码作废。撤销后再有人扫这张码，会被告知「二维码已失效」。

路径参数

字段类型说明
tokenIdGUID① 里返回的 tokenId。不是 token——故意不在 URL 上暴露真正的码值。

响应 data

{ "tokenId": "019e1fff-..." }

行为说明

  • 幂等：重复撤销同一张码，第二次仍然返回 200。
  • 撤销别人的码会返回 403 FRIEND_QR_FORBIDDEN。
  • 找不到这张码返回 404 FRIEND_QR_NOT_FOUND。

④ 扫码后预览

GET /friends/invite-qr/{token}/preview

扫到码之后，先调一次预览拿到发码人的公开资料，让用户在客户端上确认「我要加的是这个人吗」。这一步不会真的发起好友申请。

路径参数

字段类型说明
token字符串从扫到的 qrPayload 里提取的 token 值。

响应 data（FriendInviteQrPreviewDto）

{
  "inviterUserId":      "019d7c8b-...",
  "inviterDisplayName": "Alice",
  "inviterAvatarUrl":   "https://.../avatar.png",
  "message":            "扫码加我",
  "expiresAt":          "2026-05-20T06:00:00+00:00",
  "expired":            false,
  "alreadyFriends":     false
}
字段类型说明
inviterUserIdGUID发码人的用户 ID。
inviterDisplayName字符串发码人昵称。
inviterAvatarUrl字符串/null发码人头像。
message字符串/null发码人设置的默认附言。
expiresAt时间戳过期时间。
expired布尔关键字段。true 表示二维码已不可用（已过期 / 已撤销 / 已用尽）。注意这种情况不会返回错误码，只是 expired=true——客户端据此提示「二维码已失效，请向对方索取新码」。
alreadyFriends布尔true 表示扫码人和发码人已经是好友。客户端可以直接提示「你们已经是好友啦」并跳过申请步骤。
注意 二维码失效不会以 HTTP 错误形式返回——状态码仍然是 200，但 expired 为 true。请前端务必判断这个布尔位，否则会把失效码当作正常码进入下一步。
真正的错误（404 token 不存在 / 发码人账号已注销）才会以错误码形式返回。

⑤ 扫码加好友

POST /friends/invite-qr/{token}/accept 限流

正式向发码人发起好友申请。发码人仍然需要在自己的好友申请列表里点同意，才会真正成为好友。

路径参数

字段类型说明
token字符串从扫到的 qrPayload 里提取的 token。

请求体（可空，整个 body 也可以不传）

{ "message": "你好，我是从你的二维码扫过来的" }
字段类型说明
message字符串/null本次申请的附言。不传或为 null 时，自动使用发码人在生码时设置的默认附言。最长 500 字符。

响应 data

{ "requestId": "019e2000-..." }

新建（或反向命中自动接受）的好友申请 ID。后续在好友申请相关接口里可以用到。

行为说明

  • 本接口走的是完整的「发好友申请」流程，因此会受所有标准规则约束：
    • 对方设置了「不接受任何好友申请」→ 返回 FRIEND_PRIVACY_DENIED
    • 对方设置了「只接受共同好友的申请」且没有共同好友 → 返回 FRIEND_PRIVACY_NO_MUTUAL
    • 对方拉黑了你 → 返回 FRIEND_BLOCKED
    • 你们已经是好友 → 返回 FRIEND_ALREADY_EXISTS
    • 你已经发过一条待处理的申请 → 返回 FRIEND_REQUEST_PENDING
  • 双向扫码自动互加：如果对方此前也扫过你的码、或主动向你发过申请，本次扫码会被识别为「互相同意」，跳过申请直接成为好友。
  • 扫自己的码会被拒绝（FRIEND_QR_SELF）。

常见使用场景

场景 1 · 永久个人名片码

放在个人资料页常驻展示，谁都可以扫，长期有效。

POST /friends/invite-qr
{
  "ttlHours": 720,
  "maxUses":  0,
  "message":  "你好，我是 Alice"
}

有效期 30 天（系统最长）。前端需要做一件事：到期前自动调一次本接口生成新码并替换，否则旧码失效后用户无法被加。

场景 2 · 一次性面对面加

线下见面时临时生成、对方扫一次后自动作废。

POST /friends/invite-qr
{
  "ttlHours": 1,
  "maxUses":  1
}

场景 3 · 小群拉新（限量）

同事/活动里有限名额的加好友码。

POST /friends/invite-qr
{
  "ttlHours": 24,
  "maxUses":  20,
  "message":  "ZTChat 用户群欢迎你"
}

用满 20 次后状态自动变为 exhausted。

常见问题

Q：二维码图片是后端给吗？

不是。后端只返回 qrPayload 字符串，前端自己用任意二维码库（Web 端 qrcode / 小程序原生组件 / 移动端原生）渲染成图片即可。这样可以自由控制尺寸、颜色、Logo、纠错等级。

Q：扫码就成好友吗？

不是。扫码 = 发起一条普通的好友申请，发码人需要主动同意。例外：如果对方此前也扫过你的码（或向你发过申请），系统会判定为「互相同意」，跳过申请直接成为好友。

Q：能扫别的租户/企业的码吗？

可以。如果发码人和扫码人在同一个 ZTChat 平台账号体系下，跨租户/跨组织也能加为好友（前提是发码人的隐私设置允许）。

Q：二维码会泄露什么信息？

只有一个 16 位随机字符串。在被扫码人调用 preview 之前，外人拿到这个串什么也做不了；调用 preview 也只能看到发码人的公开资料（昵称、头像、默认附言）。

Q：可以同时存在多少张码？

没有硬性数量上限。GET /friends/invite-qr 返回的是所有「未失效」的码，建议前端在用户管理界面里展示这个列表。

Q：限流被触发了怎么办？

会收到 HTTP 429。每用户 5 分钟内最多 20 次生码/扫码。正常使用绝不会触发，触发了一般说明被脚本反复调用。

Q：客户端怎么把二维码里的 token 提取出来？

扫码后得到字符串形如 ztchat://friend-invite?token=A1B2C3D4E5F6A7B8。用任意 URL 解析方式取 token query 参数即可。若不是 ztchat://friend-invite 开头，按「不是 ZTChat 加好友码」处理。
