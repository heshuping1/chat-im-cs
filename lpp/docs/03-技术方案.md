# LPP 技术方案

更新时间：2026-05-23

最后核对方式：静态盘点 `lpp/lpp_mobile/lib` 目录、GoRouter 路由、Dio endpoint、WebSocket/SignalR 入口和测试文件。本文档描述当前 Flutter 移动端技术现状和目标方案，不等同于真机验收结论。

## 1. 技术目标

LPP 移动端采用 Flutter 实现，目标是在 Android 真机上交付企业 IM、在线客服、客户管理、群管理、音视频和管理工作台主流程。

技术原则：

- 优先沿用现有 Flutter / Riverpod / Dio / GoRouter / SQLite / Hive / WebRTC 架构。
- token 域必须隔离。
- 本地缓存优先展示，远端同步修正。
- 业务逻辑放在 datasource、repository、provider、service 层，不在 UI 中临时拼接。
- 对涉及权限、客户隐私、admin API、客服会话、通话和后台行为的改动，必须先走 ECC 方案确认。

### 在线客服会话状态口径

- 在线客服默认采用手动接入模式，`reception/status=online` 只表示客服可接待，不触发自动认领会话。
- `queued / waiting / pending` 状态的临时会话只能展示 `接入`，客户端必须禁用人工发送；接入接口成功后再开放输入区。
- AI 接待状态只能展示 `人工接管`，接管接口成功后再开放人工发送。
- 终态会话继续按只读处理，任何写操作失败后刷新线程状态并停止重试。
| `lib/features/startup` | 启动页和启动态分流。 | 负责进入登录、空间选择或首页。 |

## 5. 认证与 token 方案

### 5.1 token 类型

| Token | 用途 | 主要代码 |
| --- | --- | --- |
| platformToken | 登录、租户列表、选择空间、签发 admin token。 | `auth_remote_datasource.dart`、`http_client.dart` |
| tenant accessToken | `/api/client/v1/*`、`/ws/client`、`/hubs/voicecall`。 | `http_client.dart`、`space_repository_impl.dart` |
| refreshToken | 刷新当前空间 accessToken。 | `token_refresh_service.dart` |
| admin accessToken | `/api/admin/v1/*`。 | `admin_http_client.dart` |

### 5.2 admin token

流程：

1. 当前空间必须是企业空间。
2. 使用 platform token 调用 `/api/platform/v1/auth/admin-token`。
3. 获取 admin accessToken。
4. 使用 admin token 调用 `/api/admin/v1/*`。
5. admin 401 后清理缓存并重新签发。

风险：

- 需要逐 admin datasource 检查是否全部走 AdminHttpClient。
- 需要确认 admin token refresh 使用的接口和服务端策略是否一致。

## 6. 数据与缓存方案

### 6.1 空间隔离

`SpaceContext` 是接口授权和本地数据隔离的核心上下文，至少包含：

- spaceId
- accessToken
- refreshToken
- userId
- space type
- membershipRole

要求：

- 会话、消息、草稿、未读、token 必须按空间隔离。
- 切换空间后不能混用上一个空间缓存。
- 恢复登录和临时设置空间不能误写 active space。

### 6.2 空间未读和提醒口径

空间切换页的“未读会话数/未读消息数”只表示普通 IM 消息，和消息首页红点保持同源：

- 当前空间：优先使用 `conversationsProvider(currentSpaceId)` 的可见会话计算，只统计 `direct/group`，不统计 `temp_session`、隐藏会话、事件类假未读和自己发送导致的未读。
- 非当前空间：继续使用平台接口 `GET /api/platform/v1/my/spaces/unread-summary`，但客户端按普通 IM 跨空间红点理解。
- 在线客服排队、进行中、SLA 风险、超时、工作台待办、公告和系统事件走独立提醒体系，不写入消息未读。
- 收到 Gateway `space.notice` 后刷新空间列表；如果 notice 指向当前空间，同时刷新当前消息会话列表，让空间切换页与消息页快速校正。
- 当前空间客户端计算值与服务端 `unread-summary` 不一致时记录 `space.unread` 诊断日志，便于确认服务端是否混入了非 IM 业务提醒。

### 6.3 消息存储

当前存在 SQLite 会话表和 Hive 到 SQLite 迁移逻辑。

要求：

- 会话和消息按 spaceId 存储。
- 支持增量同步。
- 支持撤回、删除、已读、失败重试重复处理。
- 媒体消息需要缓存和鉴权加载。

待补：

- 表结构完整说明。
- 消息同步冲突策略。
- 大数据量性能指标。

## 7. IM 会话与消息方案

### 7.1 会话类型

| 类型 | 说明 |
| --- | --- |
| direct | 单聊 |
| group | 群聊 |
| temp_session | 在线客服临时会话 |
| direct_customer / im_direct | 注册客户与客服的直连 IM，会带客服业务属性，但不属于在线客服访客队列 |

普通消息首页展示 `direct` 和 `group`；注册客户与客服的直连 IM 仍按普通消息/客户会话入口处理。服务端或 Gateway 返回的 `temp_session` 仍保持模型解析能力，但客户端会在会话仓储、provider、分页和 Gateway 摘要合并处过滤，避免访客临时会话进入消息列表；临时会话的列表入口统一由在线客服承载。

### 7.2 消息发送

当前方案：

1. 本地生成 clientMsgId。
2. 插入发送中消息。
3. 调用发送接口。
4. 服务端返回后替换 messageId、seq 和 sentAt。
5. 失败后保留失败状态，允许重试或删除。

定时文本消息当前方案：

1. `ChatInputToolbar` 的 `+` 工具面板增加“定时消息”入口。
2. 用户选择未来时间后，输入区展示 `发送时间` 提示；普通输入区规则保持不变：无文字显示 `+`，有文字显示发送按钮。
3. 设置定时时，发送按钮文案切换为 `定时发送`，点击后把文本和 `scheduledAt` 交给 `ChatPage`。
4. 当前客户端用前台 `Timer` 在当前聊天页存活期间触发普通文本发送链路。
5. 聊天页销毁时取消本地定时器，避免页面已释放后误发。

关键接口：

- `POST /api/client/v1/direct-chats/{chatId}/messages`
- `POST /api/client/v1/groups/{groupId}/messages`
- `POST /api/client/v1/customer-service/workbench/threads/{routeType}/{threadId}/messages`
- `POST /api/client/v1/media/upload`

待服务端支持接口：

- `POST /api/client/v1/scheduled-messages`
- `GET /api/client/v1/conversations/{conversationId}/scheduled-messages`
- `DELETE /api/client/v1/scheduled-messages/{scheduledMessageId}`
- `PUT /api/client/v1/scheduled-messages/{scheduledMessageId}`

待验证：

- 弱网重复发送。
- 敏感词拒绝。
- 服务端发送成功但客户端超时。
- 定时消息的离线可靠投递、取消、编辑、多端同步和失败回执。

### 7.4 弱网和断线恢复方案

客户端按“先本地可用，再后台恢复”的方式处理弱网：

1. 全局网络状态：`connectivity_plus` 监听系统网络，断网后由 `NetworkStatusBanner` 顶部提示“网络不可用，请检查网络设置”。
2. Gateway 状态：`GatewayService` 暴露 `connecting/connected/reconnecting/disconnected` 状态；非 connected 时显示“连接中...”，避免用户误以为实时消息正常。
3. 自动重连：SignalR 内置自动重连 + 客户端指数退避兜底；系统网络恢复时，`gatewayProvider` 主动调用 `connect`，不等待下一轮退避。
4. 离线补偿：Gateway 连接成功后执行 `GET /api/client/v1/sync?sinceSeq=`，用本地游标补齐离线期间的会话消息。
5. 待发消息：文本消息发送遇到网络错误后写入 `PendingMessageQueue`，重连成功后按创建时间顺序 flush；服务端拒绝类错误不重试。
6. 本地优先：聊天页和会话列表优先读本地缓存，后台增量同步失败不阻塞已有内容展示。
7. 错误诊断：`main.dart` 捕获 Flutter、Platform 和 Zone 未处理异常，统一写入 `AppDiagnostics`；`AppErrorReporter` 只监听 error 级事件，异步限流上报 `POST /api/client/v1/client-errors`。上报字段按服务端合同使用 `platform/errorLevel/errorType/message/stackTrace/context/clientTimestamp`，context 内保留 sessionId、traceId、breadcrumbs、spaceId、userId；请求设置 3 秒超时、失败静默、跳过自身诊断日志，接口缺失时本次会话自动停用远端上报，避免影响客户正常使用。

### 7.5 线路切换与站点配置

客户端通过 `SiteLineManager` 统一管理业务 API、Admin API 和 Gateway 使用的当前站点：

1. 默认主站：内置 `主站1`，业务域名为 `https://chat.hearteasechat.com`，主站不提供 `configline/configfile`。
2. 启动刷新顺序：先使用当前站点的 `configfile`，再按本地缓存的其他可切换站点顺序逐个尝试，最后尝试 S3 兜底 configfile；没有 configfile 的站点自动跳过。
3. 无缓存场景：本地没有其他可切换站点时，主站被跳过，只尝试 S3 兜底 configfile。
4. 有缓存场景：当前站点是主站时，顺序为缓存站点 1、缓存站点 2、缓存站点 3、S3；当前站点是缓存站点 2 时，顺序为缓存站点 2、缓存站点 1、缓存站点 3、S3。
5. 缓存更新：任一 configfile 成功返回新可切换站点列表后，替换本地缓存；总体站点列表变为 `主站1 + 新可切换站点列表`。
6. 自动选线：按总体站点列表顺序检测可用性，选择第一个可用站点并写入“当前站点”；下次启动时以这个实际当前站点作为 configfile 尝试顺序的第一优先级。
7. 用户展示：线路切换页只展示 `主站1 + 本地缓存可切换站点`，S3 作为兜底 configfile 不展示。
8. 运行时生效：HTTP Dio、Admin Dio、头像/媒体相对路径解析和 Gateway 连接均读取当前站点；用户手动切换站点后 Gateway 会重连。

当前限制：

- 媒体消息上传失败目前直接失败，暂未支持完整离线排队和断点续传。
- 首次无本地缓存且网络不可用时，部分页面仍是空态或局部错误态，后续需要统一离线空态。
- `/sync` 的可靠性依赖服务端 `sinceSeq` 幂等和消息保留策略。
- 音视频通话弱网质量、后台保活、推送到达仍必须真机专项验证。

状态：

- PC 状态模型拆分为三类，不得混用：`imPresenceStatus` 用于普通 IM 在线状态和托盘；`customerServiceStatus` 用于客服是否可接待；单条会话接待状态由线程 `status/currentResponderType/aiStatus` 决定并在聊天区状态条展示。
- App/PC 进入在线客服页面只读取 `GET /customer-service/reception/status` 和线程列表，不调用 `PUT /reception/status` 或 `POST /reception/enter`；客户端本地兜底默认离线，服务端返回状态后以服务端为准；只有客服主动点击“在线/忙碌/短暂离开/离线”才更新服务端接待状态。多端登录时以同一客服账号在同一企业空间的服务端状态为准。
- 在线客服页内分为“当前接待/历史会话”。当前接待展示可处理的 `queued/active/assisting/transfer_pending` 临时会话，数据源为 `/customer-service/workbench/threads`；历史会话展示 `closed_by_visitor/closed_by_staff/closed_timeout/closed_system/archived` 等终态会话，数据源为 `/customer-service/staff/service-history?threadType=temp_session&limit=50`，进入聊天页后只读，输入区显示“会话已结束”。客户维度历史仍使用 `/customer-service/customers/service-history`；按客服本人接待历史使用新增 staff service-history，`participation=transferred` 表示本人曾参与转接但当前归属可能不是本人。
- queued
- active / assigned / assisting
- closed
- AI / 人工接待状态：列表使用 `currentResponderType` / `aiStatus` 判断主按钮；聊天页详情刷新后在消息区顶部展示接待状态条，不占用右上角客户资料、翻译、更多操作位。AI 接待中显示“人工接管”，人工接待中只展示接待人状态；人工转 AI 接待本期暂不做。
- 客户画像 `profile-card` 客户端兼容解析临时订单字段：`temporaryOrders`、`tempOrders`、`openOrders`、`pendingOrders`、`currentOrders`、`orders`，可位于顶层、`trading` 或 `account` 下；聊天页客户画像弹层在历史交易概况后展示“临时订单数据”。
- 聚合聊天来源统一归一为客户来源渠道。App 和 PC 在线客服线程、线程详情、客户画像兼容读取 `source/from/channel/sourceChannel/entryChannel/platform/provider`；展示层映射为自有 App、抖音、WhatsApp、Telegram、网页等短标签。服务端后续应稳定返回统一枚举，客户端不把缺失来源伪造成企业来源。

群发：

- 客户端普通“群发”是消息侧能力，入口在消息首页右上角 `+`，路由为 `/bulk-send`；只面向当前账号自己的好友和自己所在且可发言的群，不进入客服工作台，也不需要服务端新增群发任务接口。
- 群发目标来源：好友读取 `friendsProvider`；群聊复用当前空间 `conversationsProvider(spaceId)` 中的 `ConversationType.group`，不做进入页面前的群权限预检。
- 群发发送策略：好友目标先调用 `POST /api/client/v1/direct-chats` 创建/复用直聊，再调用 `POST /api/client/v1/direct-chats/{chatId}/messages`；群聊目标调用 `POST /api/client/v1/groups/{groupId}/messages`。客户端顺序逐个发送，逐项记录发送中、成功、失败；某个目标失败不阻塞后续目标。
- 群禁言、非成员、无权限、敏感词等以普通发送接口返回为准，由客户端转换为单个目标的失败文案。
- 企业群发消息是管理能力，入口放在管理工作台或企业管理入口，不放在客服工作台；当前 App 已先实现界面，包含官方账号发送身份、全体人员/企业员工/企业客户/官方群目标范围、内容编辑、预览和接口待接入提示；客服角色默认没有企业群发权限。
- 企业群发建议使用服务端异步任务接口 `/api/admin/v1/enterprise-broadcasts`，支持 `preview`、提交、任务列表、任务详情、失败重试和取消。
- 企业群发目标支持 `all_members`、`staff`、`customers`、`official_groups`；其中 `all_members = staff + customers`，不包含未注册访客临时会话。
- 企业群发必须以企业默认官方账号或指定官方账号作为发送人，消息、会话摘要、Push 和任务详情都展示官方账号身份，同时记录实际操作人审计。
- 企业群发权限不做复杂拆分，建议统一用 `enterprise_broadcast.send` 控制是否可发；所有者/管理员默认具备，客服默认不具备。

待验证：

- 并发认领。
- AI 与人工接管状态。
- 关闭后的客户侧显示。

兼容降级：

- 客服工作台优先使用统一接口 `/api/client/v1/customer-service/workbench/*`。
- 当统一 `threads`、`dashboard` 或临时会话详情/发送/关闭接口返回 `INVALID_OPERATION` 时，移动端仅对 `temp_session` 回退到兼容接口 `/api/client/v1/customer-service/temp-sessions/*`。
- 回退接口仍是真实服务端数据：`/mine` 获取当前客服名下会话，`/queue` 获取待接入队列，`/{sessionId}` 获取详情，`/{sessionId}/messages` 发送消息。
- 回退不覆盖 `direct_customer`；客户直连会话不进入在线客服访客队列，按普通 IM/管理端客户会话口径处理。
- 在线客服列表只展示 `temp_session`；从在线客服进入复用聊天页时必须传入 `customerServiceThreadType=temp_session`、`customerServiceThreadId` 和访客标识，确保发送、画像、关闭等动作走客服接口而不是普通 IM。
- 临时会话终态作为写操作硬边界：客户端统一识别 `closed_by_visitor`、`closed_by_staff`、`closed_timeout`、`closed_system`、`archived`、`closed` 等状态；收到 `temp_session.closed` 推送、线程轮询返回终态，或发送/关闭等写操作返回 401/终态错误码时，立即切到“会话已结束”，停止后续发消息、关闭等写操作。Widget 临时窗口轮询 `GET /api/widget/v1/sessions/{id}` 时同时兼容 `status/session_status/sessionStatus`。
- “人工接管”继续调用 `takeover`；人工转 AI 接待本期暂不做，客户端不展示入口，也不调用 `resume-ai`。

### 10.2 App/PC 工作台拆分

当前 Flutter 仓库实现 App 端客服工作台。PC 端路线调整为：**Electron + React + TypeScript 的 Windows 本地客服客户端**。PC 客户端加载本地打包前端产物，直接请求现有服务端 API；不部署线上 Web 站点，不做 macOS 客户端，不走 Flutter Desktop。App 和 PC 客服客户端共享服务端线程、客户画像、订单、工单、权限和审计数据，但 UI 承载密度不同。

| 能力 | App 技术口径 | PC 技术口径 |
| --- | --- | --- |
| 会话入口 | `CustomerServicePage` 在线客服 Tab 仅展示 `temp_session` 的全部、排队、进行中、VIP，进入聊天页携带客服线程上下文，并标识客户来源渠道。 | PC 默认进入普通消息；在线客服作为独立一级入口，仅承载 `temp_session` 队列、聊天和右侧上下文，左侧列表和聊天标题展示来源渠道。 |
| 聊天上下文 | 聊天页顶部轻量接待状态条，右上角客户资料打开底部弹层。 | 右侧常驻上下文面板展示客户画像、交易、工单、触达、风险和 AI 分析。 |
| 客户画像 | 调用 `profile-card`，摘要展示基础资料、标签、账户概况、历史交易概况、临时订单数据和工单摘要。 | 调用同一数据域和更完整客户详情接口，展示总览、交易记录、资金流水、会话、工单、触达、KYC/合规、IB/设备。 |
| 快捷处理 | 输入区和 `+` 面板承载快捷话术、知识库、图片/附件、翻译、表情、定时发送，AI 建议以轻提示或快捷插入呈现。 | 工具栏承载知识库、快捷话术、AI 起草、转人工、分配客户经理、生成报告、升级和质检入口。 |
| 临时订单 | App 展示摘要列表和空态。 | PC 展示完整订单表、筛选、详情和风险提示。 |
| 管理治理 | App 只做核心查看和轻量操作。 | PC 保留角色化工作台入口，客服/管理员/所有者按权限展示不同工具；完整 SLA 监控、团队分配、质检、审计、批量筛选和报表后续逐步扩展。 |

服务端数据口径必须统一：App 和 PC 不应各自猜测客户等级、风险、订单状态、SLA 或脱敏规则。若 PC 需要更高密度数据，应在同一客户详情/工作台数据域上扩展字段，而不是另建不可兼容枚举。PC 端采用独立 TypeScript 技术栈，必须复用接口合同、错误码、权限规则、traceId/breadcrumbs 和合同测试。

PC 端信息架构采用 `消息 / 在线客服 / 通讯录 / 工作台`：`消息` 只展示普通 IM，`在线客服` 只展示 `temp_session` 和 AI 转人工/待人工接入/人工接待会话，`工作台` 按角色承载客服工具、客户管理、客服中心、群组监管和企业群发等入口。会话超时不作为单独模块，作为在线客服会话的正常状态展示；具体超时判定、关闭原因和 SLA 风险以后端字段与 Gateway 事件为准。

PC 在线客服八项能力的信息架构补充：

- `在线客服` 承载坐席高频动作：转接、输入预览、已读时间、静默撤回、快捷话术双击插入输入框、终态只读和拖拽分栏。
- `工作台 > 客服中心` 承载管理员/所有者实时监控：全量进行中会话筛选、2 到 4 个会话并排查看、客服状态墙、改派、冻结、解冻和强制关闭。
- `数据中心 > 在线客服` 承载统计和导出：对话总量、平均首次回复时长、平均处理时长、满意度、地区、来源、SLA、`cs_sessions` 和 `cs_staff_daily_stats`。
- 历史对话搜寻条件必须覆盖时间范围、客户 ID、注册客户用户 ID、访客用户 ID、关键字、渠道、状态、指派客服、曾参与客服、语言、会话 ID、发送人、来源、地区、满意度、SLA 风险和处理时长。

PC 消息页是桌面客户端布局，不复用 App 消息首页实现，也不套用 App/微信移动端截图的居中大标题、大搜索框、底部导航、移动端行高或气泡覆盖。App 截图默认只指导 App；除非需求明确写“PC 也参考”，否则 PC 只借鉴清晰度和信息层级，不直接照搬移动端结构。

PC 端“我/设置”采用桌面 IM 信息架构：

- 左上/左下头像入口展示个人头像、客服昵称和当前企业码，点击打开“我的账号/个人中心”弹层。
- 头像弹层承载个人资料、二维码、IM 在线状态、企业空间、切换空间、企业信息、收藏、语言/时区、自动翻译、线路切换、账号安全、登录设备、隐私与黑名单和退出登录。
- 齿轮只进入系统设置，承载新消息通知、在线客服提醒、SLA 超时提醒、托盘/开机自启/自动重连、显示设置、聊天设置、诊断包、反馈和关于；不再承载“我的账号”语义。
- IM 在线状态、在线客服接待状态、单条会话接待状态保持分离，避免把客服自己的在线状态误放到客户会话标题上。

PC 消息输入第一阶段采用统一 `MessageComposer`，同时服务普通 IM 和在线客服：

- 文本发送沿用 `direct/group/workbench thread messages` 现有接口。
- 图片和文件先调用 `POST /api/client/v1/media/upload`，再按 App 消息体结构发送 `messageType=image/file`，body 使用 `{ image: MediaResource }` 或 `{ file: MediaResource }`。
- 支持选择文件、粘贴截图、拖拽图片/文件、发送中状态、上传/发送失败提示和基础重试/取消。
- 消息流统一展示 text/image/file；voice/video 首版只按服务端消息体展示只读卡片，不做 PC 录制发送。
- 下载打开、断点续传、大文件弱网、离线媒体队列和多端媒体同步继续作为后续专项。

- 移动端远程推送后续必须做成移动专用 adapter 或原生 MethodChannel，不能让移动推送依赖进入桌面依赖图。
- 移动前台、后台、杀进程、锁屏、通知点击分别验收。
- 点击通知需要恢复正确空间再跳转。
- 来电通知需要可接听、拒绝、挂断。

### 13.5 性能和流畅性

关键代码：

- `features/startup/presentation/pages/startup_gate_page.dart`
- `features/chat/presentation/providers/conversations_provider.dart`
- `features/contacts/presentation/providers/contacts_provider.dart`
- `features/profile/presentation/pages/my_profile_page.dart`
- `core/widgets/user_avatar.dart`
- `features/chat/presentation/providers/gateway_provider.dart`

技术要求：

- 启动和重新打开优先进入可交互页面，首屏数据采用本地缓存优先，后台静默刷新服务端最新数据。
- 会话列表、联系人、个人资料不能等待头像预热后再展示；头像磁盘读取和预热放到后台执行。
- 头像内存缓存必须有上限，避免长期聊天、频繁切换群聊后内存持续增长。
- 头像预热默认只利用已有磁盘缓存，不主动批量下载缺失头像；可见头像由 `CachedNetworkImage` 自己按需加载。
- 头像图片按实际显示尺寸解码，避免小头像加载原图造成图形内存和解码开销过高。
- 头像、收藏缩略图、最近联系人、黑名单和组织成员等小图必须传入明确显示尺寸，由 `AuthNetworkImage` 设置解码尺寸，避免 48px 小图解码原始大图。
- 搜索框输入必须做短延迟防抖，适用于联系人、群列表、我的客户、在线客服、快捷回复、客户端普通群发、企业群发、聊天搜索、快捷话术、企业成员、客户管理、建群选人、选择群管理员和转让群主，避免连续输入期间反复刷新 provider 或重算长列表。
- 大客户池、在线客服线程等潜在长列表必须优先使用 builder/sliver 懒加载，避免一次性构建所有行。
- Gateway 保持实时消息主链路；网络恢复后主动重连、执行 `/sync` 并 flush 待发消息队列。

本轮优化结果：

- `prefetchAvatarUrls` 改为异步磁盘读取，默认不主动批量下载缺失头像，并限制内存缓存 180 项/16MB。
- `AuthNetworkImage` 根据控件显示尺寸设置解码尺寸，头像预热不再做全尺寸预解码，并去重同一 URL 的并发内存缓存读取。
- `Debouncer` 作为轻量工具放在 `lib/core/utils/debouncer.dart`，搜索类输入统一延迟 220ms 后再更新 query，页面销毁时必须取消定时器。
- 最近联系人、黑名单、组织成员和收藏缩略图补齐 48px 解码尺寸，减少小图列表的解码和图形内存浪费。
- `工作台 > 在线客服` 线程列表改为懒加载 `SliverList.builder`，避免大量排队/进行中会话一次性构建。
- `客户管理` 客户列表改为 `ListView.builder`，搜索框、概览卡、筛选条作为固定头部索引，客户 tile 按需构建。
- 会话、联系人、个人资料缓存命中时先更新 UI，头像预热后台执行。
- 首次启动预热预算下调：启动总预热最多 3 秒，会话预热最多 2 秒，头像预热最多 900ms。
- PENM00 release 模式真机测得冷启动历史记录 `TotalTime=426ms`、热启动 `TotalTime=64ms`；2026-05-23 二次性能体检冷启动 `TotalTime=436ms`，切换消息/通讯录/工作台/我的后 PSS 约 `250MB`；继续优化搜索防抖、小图解码和客户列表懒加载后最新 release 包冷启动 `TotalTime=429ms`，主 Tab 切换后 PSS 约 `290MB`。

## 15. 主要技术风险

| 风险 | 等级 | 说明 |
| --- | --- | --- |
| admin token 混用 | S0 | 需要逐接口确认 admin API 使用 admin token。 |
| 管理端只读会话误发消息 | S0 | 必须 UI 和 service 双重限制。 |
| 客户看到内部管理数据 | S0 | 需要角色权限和入口控制。 |
| 客户画像字段无脱敏 | S1 | 需服务端和客户端共同控制。 |
| 定时消息联调不完整 | S1 | 当前已创建服务端定时任务；列表、取消、编辑、多端同步和失败事件仍需联调。 |
| 媒体消息弱网离线排队不完整 | S1 | 文本消息具备待发队列；图片/文件/语音/视频上传失败仍需用户重试，断点续传需服务端或上传协议支持。 |
| 通话和推送未真机验证 | S1 | 自动化无法覆盖真实系统行为。 |
| 大数据量性能基线不足 | S1 | 已有启动基线和缓存优化；长列表帧率、超大群、弱网消息实时性、低端机内存仍需专项。 |
