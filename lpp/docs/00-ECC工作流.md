# LPP ECC 工作流

## 1. 目标

ECC 是 LPP 项目的工程协作流程，用来保证需求、实现、测试和交付保持一致。它不是额外的形式文档，而是每次工程改动必须遵守的执行顺序。

需求规格说明书、功能矩阵和技术方案是 ECC 流程中的核心产物。它们用于承接 ECC 各阶段的结论，而不是替代 ECC 流程。

核心目标：

- 防止没有需求就直接改代码。
- 防止功能做完但矩阵状态没有更新。
- 防止方案、代码和测试互相脱节。
- 防止权限、客户隐私、admin token、客服会话等高风险逻辑被误改。
- 防止入口混乱、App/PC 边界混用、低质量 UI 和不合理 UX 被直接实现。

## 2. 标准流程

所有非简单工程改动必须按以下顺序执行：

| 阶段 | 必做内容 | 主要文档 |
| --- | --- | --- |
| 1. 需求和规划 | 明确目标、角色、入口、流程、权限、验收标准。 | `01-需求规格说明书.md` |
| 2. 产品架构审查 | 审查入口是否该存在、属于哪个角色、是否高频、是否重复、是否污染主链路、App/PC 是否应该同构。 | `01-需求规格说明书.md` / `07-UI与交互设计说明.md` |
| 3. UX 流程审查 | 明确用户路径、信息层级、状态模型、空态、错误态、权限态、平台差异和可恢复路径。 | `07-UI与交互设计说明.md` |
| 4. UI 质量审查 | 明确页面结构、密度、字体、间距、头像、按钮、视觉层级、参考基准和目标产品气质。 | `07-UI与交互设计说明.md` / `03-技术方案.md` |
| 5. PC 高保真骨架门禁 | PC 端先建立高保真视觉骨架、组件规范、页面结构和验收方式；未通过前不得补业务功能。 | `07-UI与交互设计说明.md` / `03-技术方案.md` |
| 6. 架构与设计 | 明确模块边界、数据流、接口、缓存、状态管理和风险。 | `03-技术方案.md` |
| 7. 编码和开发 | 按既有架构小范围实现，不做无关重构。 | 代码 |
| 8. 测试和验收 | 默认执行轻量级验证；高风险改动、发布候选或用户明确要求时再执行重量级全量测试、接口联调、Android 真机或人工验收。 | `04-测试验收方案.md` |
| 9. 审查和安全 | 检查权限、token、客户隐私、重复点击、错误态和脱敏。 | `06-发布风险与检查清单.md` |
| 10. 部署和上线 | 明确 debug/release、真机、安装、发布前门槛。 | `06-发布风险与检查清单.md` |
| 11. 文档和复盘 | 更新需求、矩阵、方案、测试结论。 | `01/02/03/04` |
| 12. ECC 运维 | 维护文档结构和执行规则，避免规则过期。 | `00-ECC工作流.md` |

执行任务时以阶段为主线推进，文档是阶段输出。不得为了补齐某个交付文档而跳过需求确认、方案确认、测试验收或风险评估。

## 3. 文档优先门禁

以下改动必须先更新文档，再进入代码：

- 产品逻辑变化。
- 一级/二级入口变化。
- 页面布局、信息架构或 App/PC 端侧边界变化。
- 权限边界变化。
- 角色差异变化。
- API 字段、路径、状态或错误码变化。
- 聊天、好友、群、客服、客户管理、工作台、通话、推送等主链路变化。
- 数据持久化、缓存、同步、token、admin API 变化。

执行顺序：

1. 更新 `01-需求规格说明书.md`。
2. 更新 `02-功能矩阵和实现情况.md`。
3. 更新 `07-UI与交互设计说明.md`，补充产品/UX/UI 审查结论。
4. 更新 `03-技术方案.md`。
5. 修改代码。
6. 更新 `04-测试验收方案.md` 或 `06-发布风险与检查清单.md`。

## 4. 文档保鲜门禁

文档保鲜是 ECC 的强制要求。每次非简单工程改动结束前，必须回答以下问题：

| 问题 | 如果答案为“是” |
| --- | --- |
| 需求、角色、入口、流程或验收标准是否变化？ | 更新 `01-需求规格说明书.md`。 |
| 功能状态、完成度、风险或验收方式是否变化？ | 更新 `02-功能矩阵和实现情况.md`。 |
| 产品入口、信息架构、App/PC 边界、UX 流程或 UI 质量标准是否变化？ | 更新 `07-UI与交互设计说明.md`。 |
| 架构、模块边界、数据流、接口或缓存策略是否变化？ | 更新 `03-技术方案.md`。 |
| 测试方式、验收结果或真机结论是否变化？ | 更新 `04-测试验收方案.md`。 |
| endpoint、字段、token、错误码或服务端依赖是否变化？ | 更新 `05-接口与服务端依赖.md`。 |
| 发布风险、安全风险或阻塞项是否变化？ | 更新 `06-发布风险与检查清单.md`。 |
| 页面结构、交互状态或角色可见性是否变化？ | 更新 `07-UI与交互设计说明.md`。 |
| 代码证据、接口证据或测试证据是否变化？ | 更新 `08-代码现状与证据索引.md`。 |
| 权限、隐私、脱敏、审计或治理规则是否变化？ | 更新 `09-安全权限与数据治理.md`。 |
| 性能、稳定性、弱网、长列表或真机表现是否变化？ | 更新 `10-性能稳定性基线.md`。 |
| 新发现缺口、阻塞或延期项？ | 更新 `11-项目现状与缺口清单.md`。 |
| 当前任务影响版本交付判断？ | 更新 `12-版本交付报告.md`。 |

如果没有更新相关文档，最终回复必须说明“不需要更新”的原因。

## 5. 可以压缩流程的情况

以下情况可以不单独更新三份主文档，但最终回复需要说明：

- 纯文案微调。
- 明确的样式微调。
- 注释修正。
- 不改变产品语义的机械替换。
- 不影响运行逻辑的本地脚本说明调整。

## 6. 方案确认门禁

涉及以下内容时，必须先给方案并等待确认：

- 客户、客服、管理员、所有者的权限边界。
- 好友申请、群申请、客服会话、客户分配等状态闭环。
- admin API 和普通 client API 的 token 域。
- 客户画像、工单、交易信息、风险信息等敏感数据展示。
- 会话冻结、解冻、审计、风控等治理动作。
- Android 后台、推送、来电、音视频、小窗等系统行为。

用户只是讨论、纠正理解或评估可行性时，不得直接改代码。

## 7. 产品/UX/UI 质量门禁

涉及入口、页面结构、导航、核心流程、App/PC 端侧设计、角色工作台、客服会话、消息页、通讯录、设置或客户画像的改动，必须先通过产品/UX/UI 质量门禁，再进入实现。

### 7.1 产品经理审查

实现前必须明确：

- 这个入口是否应该存在，是否有真实用户任务支撑。
- 入口属于哪个角色，是否按客户、普通成员、客服、管理员、所有者区分。
- 入口是高频还是低频；高频入口才能进入主导航，低频入口必须归入头像、设置、工作台或具体业务页。
- 是否与已有入口重复，是否会形成“更多”这类垃圾桶入口。
- 是否污染主链路，例如在线客服 `temp_session` 不进入普通消息，管理工具不进入客服工作台。
- App 和 PC 是否应该同构；如果不同，必须说明端侧差异的理由。

### 7.2 UX 审查

实现前必须明确：

- 用户从哪里进入、完成任务需要几步、失败后如何恢复。
- 页面第一屏是否能让用户判断当前在做什么、下一步能做什么。
- 状态是否分离，不能混用 IM 在线状态、在线客服接待状态、单条会话状态、客户在线状态。
- Loading、Empty、Error、Disabled、Permission、Offline 是否都有明确表现。
- 是否有误导性文案、重复入口、难以理解的抽象名词或不可见的关键状态。

### 7.3 UI 审查

实现前必须明确：

- 页面目标气质：App 偏移动轻量，PC 偏高效率客服工作台，不能互相照搬。
- 字体、行高、头像、边距、按钮、标签、列表密度、留白是否与成熟产品基准一致。
- 图标是否统一使用 lucide 图标库；除品牌 Logo、用户头像、真实图片、平台官方 Logo 外，不得新增手写 SVG、emoji 图标、字符图标或混用其他图标库。
- 是否选定参考基准；App 消息页可参考微信移动端，PC 消息页只能参考桌面 IM 的清晰度和信息层级，不能照搬移动截图。
- 视觉层级是否服务主要任务，不能为了装饰牺牲信息密度和可读性。
- 页面是否看起来像可交付产品，而不是工程调试页或占位卡片堆叠。

没有通过以上审查的入口和页面，不允许进入编码实现；如果用户直接要求实现但产品/UX/UI 判断明显不合理，必须先指出风险并给出更合理替代方案。

### 7.4 PC 高保真视觉骨架门禁

PC 端进入业务功能实现前，必须先完成高保真视觉骨架和设计系统收敛。该门禁适用于 PC 的消息、通讯录、在线客服、工作台、头像弹层、设置页、客户上下文和后续管理扩展页面。

实现前必须输出：

- 设计原则：说明 PC 端目标气质、参考对象、参考范围和不参考的内容。
- 组件规范：明确颜色 token、字号、行高、头像尺寸、列表高度、边距、圆角、阴影、红点、hover、active、focus、disabled、loading、empty、error 状态。
- 图标规范：所有通用 UI 图标必须使用 lucide 图标库，并统一尺寸、线宽、颜色和 hover/active/focus 状态；不得用 emoji、字符、临时 SVG 代替 lucide 图标。品牌 Logo、用户头像、真实图片和明确授权的平台官方 Logo 不属于通用 UI 图标。
- 页面结构：明确左侧导航、列表区、聊天区、上下文区、弹层、设置页骨架和各入口归属。
- 高保真骨架：先实现可运行的页面外壳和核心组件，不接入或少接入业务功能，不用功能缺失破坏视觉系统。
- 验收方式：至少包含浏览器预览、关键页面截图或可视检查、自动化冒烟测试、窗口尺寸和长文本检查。

强制规则：

- 先不要实现业务功能，先按顶级产品标准做高保真 PC 视觉骨架。
- 如果新增需求会破坏信息架构、主导航边界、角色边界或视觉系统，必须先反驳并给出替代方案，不能直接照做。
- imagegen、截图和竞品图只能作为视觉效果参考，不能无脑照搬内容和信息架构。
- PC 不得照搬 App 或移动端微信结构；App 可参考移动端清爽列表，PC 必须按桌面高效率工作台设计。
- 业务功能只能装进已确认的组件体系里，不能每个功能单独长一套样式。

## 8. 交付定义

一个功能可以进入交付候选，至少需要满足：

- 需求规格中有明确描述。
- 功能矩阵中状态、风险、验收方式明确。
- 产品/UX/UI 审查结论明确，入口、角色、平台边界和视觉质量标准没有未决冲突。
- 技术方案中模块边界、接口和数据流明确。
- 代码实现符合既有架构。
- 测试或验收结论有记录。
- S0 问题清零，S1 问题有处理计划。

## 8.1 测试分级门禁

ECC 默认采用“轻量验证优先、重量级验证按需触发”的测试策略。不要为了每个小功能都跑完整回归，避免把开发节奏拖慢。

### 8.1.1 默认轻量级验证

适用场景：

- 小范围 UI 调整、文案调整、布局修复。
- 单个组件或单个页面的低风险功能补齐。
- 不改变接口合同、权限边界、token 域、消息底层、缓存同步和发布产物的改动。
- 用户没有明确要求跑全量测试。

默认执行方式：

- PC 端：优先 `npm run typecheck`，必要时补一个受影响页面的浏览器预览或目标用例。
- App 端：优先 `flutter analyze --no-fatal-infos` 或受影响文件/目录的定向 `flutter test`。
- 文档或纯方案：只做文档一致性检查，不跑代码测试。
- 只改 CSS/视觉骨架时，可以用浏览器截图/人工目检作为主要验证，必要时再跑 typecheck。

最终回复需要说明“已做轻量验证”以及没有跑重量级测试的原因。

### 8.1.2 重量级验证触发条件

以下情况必须执行重量级验证，或先明确告知用户需要重量级验证：

- 用户明确说“跑全量测试、重量级测试、完整回归、打包、发布前检查”。
- 发布候选、正式包、release APK、Windows 安装包、交付验收前。
- 登录、token、权限、admin/client/widget 域隔离、空间切换等安全边界变化。
- 消息底层、Gateway、离线缓存、未读、推送、在线客服接入、会话状态机、多端同步变化。
- 数据持久化、迁移、缓存清理、错误诊断上报、隐私脱敏变化。
- 大范围重构、跨多个主模块、公共组件或 API client 变更。
- 轻量验证失败，或无法判断风险范围。

重量级验证可包括：

- PC：`npm run typecheck`、`npm run build`、`npm run test:browser`。
- App：`flutter analyze --no-fatal-infos`、相关定向测试、必要时 `flutter test ../scripts/mobile/test/flutter`。
- 发布前：完整回归脚本、真机安装、角色/多端/弱网专项。

### 8.1.3 用户指令优先

如果用户要求“先快点修”“只做轻量验证”“重量级测试听我指令”，则默认不跑全量测试，除非改动触发 S0/S1 风险。若确实触发高风险，必须先说明原因，再决定是否升级验证。

## 9. 文档体检

每个版本或每周至少执行一次文档体检：

```bash
find lpp/docs -type f | sort
rg "待填写|待确认|未执行|待细化|需确认|TBD|TODO" lpp/docs
```

体检结果需要同步到：

- `11-项目现状与缺口清单.md`
- `12-版本交付报告.md`

## 10. Codex 执行规则

- 搜索优先使用 `rg`。
- 改代码前先读相关需求、矩阵和技术方案。
- 改入口、页面和布局前必须先做产品/UX/UI 审查，不得无脑照搬用户截图或上一轮临时方案。
- 不回滚用户已有改动。
- 不处理无关文件。
- 不把临时猜测写成当前事实。
- 最终回复必须说明改了什么、验证了什么、还剩什么风险。
