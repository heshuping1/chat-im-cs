# LPP UI 与交互设计说明

## 1. 文档目的

本文档用于承接 ECC 的产品架构、UX 流程和 UI 质量审查阶段，描述 App/PC 的页面结构、交互规则、状态反馈、角色差异、平台边界和视觉体验约束。

## 2. 适用范围

- Flutter 移动端 App，Android 真机优先。
- Electron + React + TypeScript 的 Windows PC 客服客户端。
- IM、通讯录、客户管理、在线客服、管理工作台、群管理、通话、设置、客户画像和客服工作台相关页面。

## 3. 全局设计原则

- 移动端以清晰、克制、可快速操作为主。
- 客户侧避免暴露内部管理语义。
- 管理员和所有者侧强调信息密度、状态可识别和操作可追溯。
- 客服侧强调当前会话、客户身份、快捷处理和风险提醒。
- App 客服工作台不照搬 PC 三栏大屏，优先保证移动端快速查看、快速回复、快速接管。
- PC 客服工作台应承载高密度信息，采用会话列表、聊天区和客户上下文常驻面板的多栏形态；落地形态为 Electron + React + TypeScript 的 Windows 本地客服客户端，不部署线上 Web 站点，暂停 macOS 桌面端。
- 所有可操作入口必须有 loading、禁用态、成功态和失败态。

## 4. 产品 / UX / UI 质量门禁

所有新增或调整入口、页面布局、导航结构、核心流程、App/PC 端侧差异、角色工作台、消息页、在线客服、通讯录、设置和客户画像的改动，必须先完成本节审查。没有审查结论，不进入实现。

### 4.1 产品经理审查

| 检查项 | 必须回答 |
| --- | --- |
| 入口存在性 | 这个入口是否有真实用户任务支撑，还是只是因为功能存在就放上去。 |
| 角色归属 | 客户、普通成员、客服、管理员、所有者分别是否可见；不可见时是否需要隐藏而不是禁用。 |
| 高频/低频 | 高频入口才能进入主导航；低频能力应归入头像、设置、工作台、会话详情或具体业务页。 |
| 入口重复 | 是否和已有入口重复；是否产生“更多”这类没有明确业务归属的垃圾桶入口。 |
| 主链路污染 | 在线客服 `temp_session` 不进入普通消息；管理工具不进入客服工作台；系统设置不混入个人中心。 |
| 端侧差异 | App 和 PC 是否应该同构；若不同，必须说明屏幕空间、操作频率、信息密度和设备能力差异。 |

### 4.2 UX 审查

| 检查项 | 必须回答 |
| --- | --- |
| 任务路径 | 用户从哪里进入，完成核心任务需要几步，是否存在更短路径。 |
| 第一屏理解 | 用户能否立刻判断当前位置、当前状态和下一步动作。 |
| 状态模型 | IM 在线状态、在线客服接待状态、单条会话状态、客户在线状态必须分离，不得混用。 |
| 异常恢复 | Loading、Empty、Error、Disabled、Permission、Offline 必须有明确文案和可恢复路径。 |
| 文案心智 | 文案必须贴近用户认知，不使用内部实现名；例如“转 AI 接待”比“AI 接管”更清楚。 |
| 跨端一致 | 同一业务对象在 App/PC 名称一致，但布局可按平台差异调整。 |

### 4.3 UI 审查

| 检查项 | 必须回答 |
| --- | --- |
| 产品气质 | App 偏移动轻量处理，PC 偏高效率客服工作台；不能把 App 页面拉宽，也不能把 PC 三栏照搬到手机。 |
| 视觉基准 | 必须明确参考对象和参考范围；App 消息页可参考微信移动端，PC 消息页只参考桌面 IM 的清晰度和信息层级。 |
| 基础尺度 | 字体、行高、头像、左右边距、列表间距、按钮大小必须统一，不能凭感觉临时拼。 |
| 信息密度 | PC 应提高信息密度但保持可读，App 应减少信息但保持关键状态可见。 |
| 视觉层级 | 主任务、次任务、状态、提示、危险操作必须有清晰层级。 |
| 成熟度 | 页面必须像可交付产品，避免工程调试页、占位卡片堆叠和大面积无意义留白。 |

### 4.4 实施前结论模板

进入编码前必须形成以下结论：

| 项目 | 结论 |
| --- | --- |
| 入口是否保留 | 保留 / 删除 / 降级到其它入口。 |
| 放置位置 | 主导航 / 头像弹层 / 设置 / 工作台 / 会话详情 / 输入工具区 / 其它。 |
| 可见角色 | 客户 / 普通成员 / 客服 / 管理员 / 所有者。 |
| App 形态 | 移动轻量入口、弹层、底部导航或页面。 |
| PC 形态 | 左侧导航、三栏工作台、右侧面板、弹层或设置页。 |
| 不做范围 | 明确本期不做什么，防止功能膨胀。 |
| 验收标准 | 至少包含入口位置、角色可见、核心状态和视觉质量检查。 |

## 6. 通用交互状态

| 状态 | 必须表现 |
| --- | --- |
| Loading | 防重复点击，显示当前动作正在执行。 |
| Empty | 说明当前没有数据，不误导用户为错误。 |
| Error | 给出可理解错误和重试入口。 |
| Disabled | 权限不足、状态不允许或请求中时明确禁用。 |
| Permission | 系统权限和业务权限要区分说明。 |
| Offline | 网络不可用时保留本地可读能力，写操作提示失败。 |

## 6.1 在线客服接入规则

- 默认采用手动接入模式：客户发起临时会话后进入在线客服队列，在线状态只表示客服可接待，不代表自动归属给某个客服。
- 排队会话显示 `接入`，未接入前聊天输入区禁用；`接入` 成功后会话进入 `进行中 / 我正在接待`，客服才可以人工回复。
- AI 接待中的会话显示 `人工接管`，接管前输入区禁用；接管成功后进入人工接待。
- 已被其他客服接入的会话不再出现在排队入口；普通客服如在进行中列表看到，应默认只读或显示正在接待人，只有具备权限的接管/转接操作才能改变归属。
- 历史/已关闭会话只读查看，不允许接入、接管、关闭或发送。

| 角色 | UI 重点 |
| --- | --- |
| 客户 | 聊天、客服服务、可见群、个人资料。 |
| 普通成员 | 基础聊天、组织通讯录、权限内客户。 |
| 客服 | 会话接待、客户画像、快捷回复、当前状态。 |
| 管理员 | 客户、会话、群、申请和审计入口。 |

## 8. 当前代码已体现的页面补充

| 页面/能力 | 代码现状对应的交互要求 |
| --- | --- |
| 快速登录页 | 全屏承载测试账号；按钮直接发起登录；登录中显示 loading；账号按身份分组，账号信息完整换行展示。 |
| 个人资料页 | 编辑字段保存时保持当前页面，不跳回上一页；头像上传失败不覆盖旧头像；绿泡泡号保存失败要展示服务端原因。 |
| 设置页 | 系统权限、服务端设置、本地显示偏好要在交互上区分；网络诊断不展示 token；退出登录需要 loading。 |
| 隐私设置页 | 保存成功后保留开关状态；保存失败回滚或提示重试；该设置对搜索和好友申请的真实影响需要双账号验收。 |
| 登录设备页 | 删除设备需要二次确认；当前设备不应被误删后导致用户困惑。 |
| 黑名单页 | 空态展示“暂无黑名单”；移除后刷新列表，并提示结果。 |
| 企业管理页 | 管理员/所有者入口可见；普通成员和客户不显示管理语义；企业功能开关保存后刷新当前空间能力。 |
| 企业成员页 | 角色修改和移除成员属于高风险操作，需要确认弹窗、loading、失败回滚。 |
| 加入企业页 | 企业搜索、邀请码和申请中状态必须清晰区分；申请未审批不显示为已加入。 |
| 公告列表/详情 | 列表空态不影响首页；详情加载失败允许返回；公告管理入口和公告查看入口要按角色区分。 |
| 在线状态 | 状态标识应弱化，不把离线/未知做成强错误；Gateway 断开时可显示最近状态但不标成实时。 |
| 消息首页 | App 消息首页参考微信消息列表：顶部居中显示 `消息(n)`，左侧小头像作为空间/账号入口，右侧圆形 `+`；搜索框单独占一行。底部导航已经有“消息/工作台”，顶部不再重复放“消息/工作台”切换。会话列表采用白底平铺行，头像、标题、时间、未读红点/徽标和最近消息层级清晰；企业 Logo 只在空间切换或企业信息中展示。 |
| PC 消息页 | PC 默认进入普通 IM 消息页，采用桌面客户端布局：左侧全局导航、可拖拽宽度的会话列表、中间聊天区；消息列表顶部是紧凑工具栏，包含标题、搜索、筛选、发起聊天、客户端群发和查找记录，不照搬 App/微信移动端的居中大标题、大搜索框、底部导航或移动端行高。资料、查找聊天记录和会话设置从聊天区右上角操作进入。未读数量在标题、会话项和导航红点体现；`temp_session` 在线客服临时会话不进入普通消息；聊天区展示发送方头像、消息内容、时间和图片/文件消息。 |
| PC 一级入口 | PC 首版是客服增强型 IM 客户端，不是通用协作套件。主导航保留头像入口、消息、通讯录、在线客服、工单中心、数据中心、工作台和设置；日历、会议、云文档、更多不进入首版主导航。音视频从会话和来电弹窗进入，知识库/话术从工作台或客服输入工具区进入，低频个人能力从头像弹层进入，系统能力从设置进入。 |
| PC 外壳视觉基准 | PC 端选用“E 轻奢白瓷”作为主视觉方向：浅色高级工作区、克制绿色主色、细边框、轻阴影、紧凑但不拥挤。在线客服页吸收“H 客服旗舰”的工作台布局能力：队列、聊天、接待状态、右侧客户上下文常驻，但仍套入 E 的轻量高级视觉外壳。左侧导航承载个人头像入口、核心导航和系统设置；导航采用 lucide 图标+文字、明确 active/hover/focus/红点状态，不放无归属入口。头像弹层采用真实个人入口心智，展示个人信息、企业空间、企业信息和 IM 状态；企业 Logo 只在企业空间/企业信息中出现。 |
| PC 设计系统收敛 | PC 端组件先统一再补功能：侧栏、列表、搜索、筛选、弹层、卡片、输入区、设置页都使用同一套色彩 token、焦点态、hover 态、阴影、边框和字体层级。消息页、在线客服页、头像弹层和设置页必须共享同一套桌面密度，避免一个页面像 IM、一个页面像后台表单、一个页面像移动端拉宽。 |
| 图标系统 | App 和 PC 的通用 UI 图标统一使用 lucide 图标库。按钮、导航、工具栏、空态、状态、渠道标识和设置项不得使用 emoji、字符图标、手写 SVG 或临时图标库。例外仅限品牌 Logo、用户头像、真实图片和明确授权的平台官方 Logo；如果 lucide 没有完全对应图标，选择语义最接近的 lucide 图标并通过文案补足含义。 |
| PC 高保真执行标准 | PC 功能开发前先做可运行高保真骨架。设计原则：客服增强型桌面 IM、轻奢白瓷浅色身份栏、浅色工作区、紧凑但不拥挤。头像弹层必须沿用主色调，不使用深色卡片或夜间模式质感；深色只作为未来独立暗色主题处理，不混入当前主视觉。组件规范：主色、提醒色、头像、列表、搜索、Tab、弹层、卡片、输入区、滚动条、hover、active、focus、disabled、empty、error 必须统一。页面结构：左侧个人入口和主导航，中间可拖拽列表，右侧聊天/详情/上下文，设置页采用左侧设置分组和右侧内容。验收标准：浏览器预览截图或可视检查、自动化冒烟、长文本、无头像、未读红点、窗口缩放和空态检查。 |
| PC 高保真骨架完成口径 | 进入业务功能补齐前，PC 已具备统一外壳：消息、在线客服、通讯录、头像弹层、系统设置、知识库、AI 助手、工单中心、数据中心、工作台均使用 E 轻奢白瓷视觉语言；在线客服采用 H 方案的信息布局；设置页覆盖通知、桌面、显示字号、语言时区、自动翻译、聊天、安全隐私、线路、诊断、反馈和可访问性入口。后续新增功能必须套入这套组件和页面密度，不得另起一套移动端或后台表格风格。 |
| PC 在线客服八项能力专项设计 | 2026-06-10 新增的历史对话、转接、管理员/所有者实时监控、统计、客户输入预览、已读时间、静默撤回和续聊，统一按 [PC 在线客服八项能力 DDD 与 UI/UX 设计](设计方案/PC在线客服八项能力DDD与UIUX设计.md) 落地。坐席高频动作放 `在线客服`，管理员/所有者监控放 `工作台 > 客服中心`，统计和导出放 `数据中心 > 在线客服`；普通 IM 私聊和群聊不纳入该专项。 |
| 通讯录页 | 个人空间展示好友、新朋友、群、收藏、最近联系人；员工空间展示组织、客户和群入口；客服额外展示客服快捷入口；管理员/所有者展示客户管理入口。 |
| 首页加号菜单 | 客户空间根据企业 `friendMode` 决定是否展示添加好友；客户始终可扫码、加入企业和进入客户端普通群发；员工可进入客户端普通群发，只有管理员/所有者可创建群和从首页加入空间。 |
| 聊天页 | 在线客服聊天页只承载 `temp_session` 访客临时会话，并且只能从在线客服入口进入并携带线程上下文；`direct_customer`/`im_direct` 是注册客户直连 IM，不混入在线客服队列。在线客服会话在消息区顶部展示轻量接待状态条，并同时显示客户来源渠道，例如自有 App、抖音、WhatsApp、Telegram、网页；AI 接待中显示“AI 正在接待 + 人工接管”，人工接待中显示“我正在接待/某客服正在接待”；人工转 AI 接待本期暂不做，App 不展示入口；右上角保持客户资料、翻译、更多，不继续堆按钮；管理端只读查看时输入区和发送动作必须禁用；群禁言原因要直接反馈到输入区；输入栏右侧固定只保留表情和 `+`，文本普通发送和定时发送均由键盘右下角发送键触发；`+` 工具面板提供定时消息入口，选择时间后在输入区上方展示发送时间。 |
| 客服工作台 | 客服页为在线客服、客户服务、效能三 Tab；在线客服是访客临时会话 `temp_session` 的唯一列表入口，排队、进行中、VIP 和搜索筛选都只按 `temp_session` 计算；在线客服列表增加“当前接待/历史会话”切换，当前接待来自 workbench 实时线程，历史会话来自 staff service-history，历史只展示终态会话并只读进入聊天；会话卡片必须标识客户来源渠道，便于客服区分自有 App、抖音、WhatsApp、Telegram 等聚合入口；状态切换使用在线/忙碌/短暂离开/离线分段控件，进入页面只读取状态，不自动改成在线；列表主按钮文案为排队“接入”、AI 接待中“人工接管”、人工进行中“继续沟通”、历史终态“查看”；认领、接管、关闭必须有 loading 和失败提示。 |
| 客户画像弹层 | 在线客服聊天页右上角客户资料入口打开底部弹层；注册客户展示客户画像、账户概况、历史交易概况、临时订单数据和工单信息；访客仅展示访客画像和注册提示；临时订单无数据时显示“暂无临时订单”，不得显示模拟订单；App 只做摘要，不承载完整交易表、资金流水表和大量工单处理流。 |
| App/PC 客服边界 | App 聊天页可展示客户姓名、VIP/风险/来源、AI 接待状态、SLA 倒计时、客户正在输入、发送失败、合规检查和 AI 建议回复等轻量上下文；PC 客服客户端采用右侧常驻上下文面板，展示客户详情总览、交易记录、资金流水、会话、工单、触达、KYC/合规、IB/设备、质检和审计入口。 |
| 管理工作台 | 管理员/所有者工作台按客户运营、服务运营、风控治理、群组运营、企业管理分类；客户管理需要展示全局统计和已分配/未分配维度；客户管理、客服中心、群组监管即使接口暂未接通也先展示页面结构和空态，不用整页“接口不可用”阻断用户理解；群组监管是 App 端轻量监管和应急处置，列表页不放全员禁言、移除成员、强制解散等全局动作，点击群组进入监管详情后只保留冻结/解冻应急动作，其它复杂治理显示接口待接入；服务运营统一入口为“客服中心”，页面包含客服人员、客户会话、在线客服；客户会话和在线客服会话都需要服务端分页与统计支持；客服人员展示账号状态和接待状态，账号状态缺字段时显示未知，不用接待状态冒充；管理员/所有者进入会话只读查看，不发言、不接管、不关闭。普通员工进入工作台只展示轻量企业入口。 |
| 客户端普通群发页 | 从消息首页 `+` 菜单进入；页面包含消息内容、搜索框、好友分组、群聊分组和底部发送按钮；发送对象是当前账号自己的好友和所在群聊；发送时逐项显示发送中、成功、失败，群禁言、无权限、非成员等由普通发送接口结果转为失败文案；部分失败不清空选择，方便用户查看失败原因。 |
| 客服快捷回复 | 快捷回复列表需支持加载、空态和错误态，插入聊天输入区时不覆盖用户已输入内容。 |
| 群设置与治理 | 群公告、二维码、备注、群管理员、转让群主、申请审批、单人禁言和已读回执均有代码入口；高风险操作需二次确认或明确失败回滚。 |
| 通话页 | 通话中区分语音、视频、来电和结束态；小窗不应遮挡核心按钮；权限拒绝要引导系统设置。 |
