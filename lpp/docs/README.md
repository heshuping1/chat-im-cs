## ECC 核心产物

以下文档是 ECC 流程中的核心产物，不是独立于 ECC 之外的另一套流程。

1. [需求规格说明书](01-需求规格说明书.md)：对应 ECC 的需求和规划阶段，说明项目要交付什么、面向哪些角色、每个模块的业务边界和验收口径。
2. [功能矩阵和实现情况](02-功能矩阵和实现情况.md)：贯穿 ECC 的规划、开发、测试和复盘阶段，说明每个功能当前状态、完成度、缺口、风险和验收方式。
3. [技术方案](03-技术方案.md)：对应 ECC 的 UI 设计、架构设计、编码实现和部署阶段，说明 Flutter 客户端、接口集成、本地数据、实时通信、通话、工作台和发布方案。

角色视图和端侧视图在 [需求规格说明书](01-需求规格说明书.md) 中维护。测试覆盖和人工测试分界在 [测试验收方案](04-测试验收方案.md) 中维护，执行后的结论回填到 [版本交付报告](12-版本交付报告.md)。

## 工程改动流程

所有非简单工程改动必须按 [ECC 工作流](00-ECC工作流.md) 执行。用户明确要求先讨论方案时，必须停在方案阶段，等确认后再改代码。

1. 明确需求：更新或确认 [需求规格说明书](01-需求规格说明书.md)。
2. 明确状态：更新或确认 [功能矩阵和实现情况](02-功能矩阵和实现情况.md)。
3. 产品/UX/UI 审查：更新或确认 [UI 与交互设计说明](07-UI与交互设计说明.md)，明确入口是否合理、App/PC 边界、用户路径和视觉质量标准。
4. PC 高保真骨架门禁：PC 端先输出设计原则、组件规范、页面结构和验收方式；高保真骨架通过前，不补业务功能。
5. 明确方案：更新或确认 [技术方案](03-技术方案.md)。
6. 执行实现：按既有架构修改代码，避免无关重构。
7. 验证结果：按 [测试验收方案](04-测试验收方案.md) 执行检查；默认轻量级验证，重量级全量测试只在高风险、发布候选或用户明确要求时执行。
8. 评估发布：按 [发布风险与检查清单](06-发布风险与检查清单.md) 判断是否可交付。

如果需求、功能矩阵和技术方案互相冲突，先统一文档口径，再进入代码实现。

PC 端功能开发前必须先做高保真视觉骨架和设计系统收敛。如果用户提出的实现要求会破坏信息架构、主导航边界、角色边界或视觉系统，必须先指出风险并给出替代方案，不能直接照做。

App 和 PC 的通用 UI 图标统一使用 lucide 图标库。除品牌 Logo、用户头像、真实图片和明确授权的平台官方 Logo 外，不得新增 emoji 图标、字符图标、手写 SVG 或混用其他图标库。

## 文档维护规则

- 三个核心产物必须保持可交付质量，不能只写任务备忘。
- 新增重要功能时，至少补充需求、功能矩阵、技术方案和测试验收口径。
- 接口字段、权限边界、角色差异、状态闭环和数据持久化变化必须同步文档。
- 文档使用中文，面向产品、测试、开发和交付阅读。

## 文档保鲜规则

文档保鲜依赖流程门禁，而不是事后集中补写。任何非简单工程改动都必须同时判断文档是否需要更新。

### 必须同步更新的情况

| 改动类型 | 必须检查的文档 |
| --- | --- |
| 新增或调整产品能力 | `01`、`02`、`03`、`04` |
| 修改页面、入口、交互状态 | `01`、`03`、`07` |
| 修改产品入口、导航结构、App/PC 边界或视觉体系 | `01`、`02`、`03`、`07` |
| 修改接口、字段、token、错误处理 | `03`、`05`、`08`、`09` |
| 修改权限、角色、客户数据可见性 | `01`、`02`、`09`、`06` |
| 修改测试、验收、发布口径 | `04`、`06`、`10`、`12` |
| 新增或修改自动化脚本 | `04`、`08`、`12`、`13` |
| 发现缺口、阻塞、未验证风险 | `02`、`06`、`11`、`12` |

### 状态证据要求

功能矩阵中的状态必须有证据支撑：

- 写“可用”时，至少要有代码证据和测试/真机/接口证据。
- 写“部分可用”时，必须写清楚已完成部分和未完成缺口。
- 写“需服务端”时，必须在 `05-接口与服务端依赖.md` 或 `11-项目现状与缺口清单.md` 中能找到对应依赖。
- 写“需真机验证”时，不能在版本交付报告中标记为已验收。

### 定期体检命令

每个版本或每周至少执行一次文档体检：

```bash
find lpp/docs -type f | sort
rg "待填写|待确认|未执行|待细化|需确认|TBD|TODO" lpp/docs
```

体检结果需要写入：

- `11-项目现状与缺口清单.md`
- `12-版本交付报告.md`
