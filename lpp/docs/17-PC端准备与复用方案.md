# PC 客服客户端准备与复用方案

更新时间：2026-05-24

本文用于启动 PC 客服客户端前的准备工作。最新产品和技术口径调整为：**PC 客户端采用 Electron + React + TypeScript，交付 Windows 本地客户端；不部署线上 Web 站点；不做 macOS 客户端；Flutter Desktop 不作为 PC 主线**。

## 1. 当前结论

| 问题 | 结论 |
| --- | --- |
| PC 客户端技术路线 | Electron + React + TypeScript + Vite。 |
| 交付形态 | Windows 本地安装包，Electron 加载本地打包产物。 |
| 是否需要部署 Web 端 | 不需要。开发时可用 localhost，交付时前端静态产物打包进客户端。 |
| 首版服务对象 | 客服 PC 客户端，默认进入消息；在线客服作为独立高频入口。 |
| 管理员/所有者功能 | 通过角色化工作台展示权限内入口；完整管理后台仍作为后续扩展。 |
| macOS 客户端 | 暂停。历史 macOS 构建仅作为体检记录。 |
| Flutter Desktop | 不作为 PC 主线。 |
| App 和 PC 如何复用 | 不复用 Flutter UI；复用接口合同、权限规则、线路切换规则、Gateway 事件、客服会话状态机、诊断字段和测试验收口径。 |

## 2. 工程方案

已新增独立工程目录 `lpp_pc_client`。

| 层级 | 责任 |
| --- | --- |
| Electron Main | 窗口、托盘、系统通知、文件打开/保存、诊断导出、自动更新预留。 |
| Electron Preload | 通过 `window.desktopApi` 暴露安全白名单接口，Renderer 禁止直接使用 Node。 |
| Renderer | React PC 客户端 UI：消息、在线客服、通讯录、角色化工作台、设置。 |
| Shared | 主进程和渲染进程共享类型，例如通知、托盘状态、诊断包。 |

本地能力接口：

- `desktopApi.notify(...)`
- `desktopApi.openFile(...)`
- `desktopApi.saveFile(...)`
- `desktopApi.openExternal(...)`
- `desktopApi.getAppVersion()`
- `desktopApi.exportDiagnostics(...)`
- `desktopApi.setTrayStatus(...)`

## 4. 一期明确不做

- 不做 macOS 客户端。
- 不做线上 Web 站点部署。
- 不做完整管理后台，但保留角色化工作台入口和无权限态。
- 不做完整数据中心、绩效中心、复杂质检后台；首版只保留数据中心入口骨架和空态。
- 不做服务端新增能力的假数据闭环；接口缺失时显示空态或接口待接入。

## 5. 复用和接口口径

- PC 客户端直接调用现有服务端 API：平台登录、租户选择、token 刷新、普通 IM、客服线程、客户画像、文件上传下载和 Gateway。
- PC 当前默认进入正式环境：未登录时展示登录页，登录成功后使用租户级 `accessToken` 调用 `/api/client/v1/*`；IM 会话、消息历史、普通 IM 文本发送、在线客服线程、客服消息发送、客户画像、好友、企业成员和组织架构不再使用本地示例数据兜底。
- 未接入正式接口的模块必须显示空态或“接口待接入”，不能用本地假数据伪装成功。
- 所有业务权限仍以后端返回角色和权限为准，前端只做显示控制，不自行扩大权限。
- PC 客户端必须复用 App 已确认的规则：`temp_session` 只在在线客服入口展示、普通消息不展示临时会话、管理员只读、客服接入/接管/关闭状态机、traceId/breadcrumbs 诊断字段。
- 会话超时以后端状态为准；PC 只展示 `closed_timeout`、`reasonCode/reasonText`、SLA 风险和倒计时，不在前端自行判定关闭。
- 缺服务端接口时记录到 `服务端支持.md`，客户端只展示空态或接口待接入。

| 人工验收 | 多账号真实客服会话、音视频听感、弱网、长时间在线、Windows 不同版本兼容性。 |

```bash
cd /Users/treesoft/Downloads/lpp-flutte/lpp
scripts/pc/test/run_pc_browser_tests.sh
```

脚本会进入 `lpp_pc_client` 并顺序执行 `npm run typecheck`、`npm run build`、`npm run test:browser`。如需只跑浏览器用例，可使用 `--no-typecheck --no-build`；其余参数会透传给 Playwright。

## 7. 启动顺序

1. 完成 `lpp_pc_client` 工程基础构建、类型检查和本地运行。
2. 接入登录、空间选择和 token 刷新。
3. 接入 Gateway 和普通 IM 会话列表。
4. 实现在线客服三栏工作台：队列、聊天、客户上下文。
5. 实现系统通知、托盘、诊断导出。
6. 接入普通 IM、图片/文件、快捷回复、翻译和定时发送。
7. 做 Windows 安装包、自动更新和桌面专项验收。
8. 再评估 P1/P2 的多窗口、音视频、完整客户详情、质检审计和管理扩展。
