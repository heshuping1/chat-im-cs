## 1. 验收目标

测试验收用于判断当前版本是否具备交付条件。LPP 不能只依赖单元测试，也不能只依赖人工点测，需要按功能性质选择证据类型。纯逻辑、参数构造、权限矩阵、模型转换、provider 状态、组件交互等功能，可以通过代码审查和自动化测试完成验证；真实账号、验证码送达、设备权限、媒体能力、多账号、多端同步、真实服务端数据和人工体验，才必须进入联调或人工验收。

- 自动化测试。
- 静态分析。
- 接口联调。
- Android 真机。
- 多角色、多账号、多端。
- 发布风险检查。

自动化脚本目录、执行边界和证据回填规则见 [自动化执行脚本说明](13-自动化执行脚本说明.md)。

## 3.1 轻量级验证和重量级验证

ECC 测试默认采用分级策略：小改动先做轻量级验证；重量级测试只在高风险、发布候选或用户明确要求时执行。

### 3.1.1 轻量级验证默认策略

| 改动类型 | 默认验证 | 说明 |
| --- | --- | --- |
| PC 单页面/单组件 UI 或低风险逻辑 | `npm run typecheck`，必要时浏览器预览受影响页面 | 不默认跑 `npm run build` 和完整 `test:browser`。 |
| PC 消息、在线客服、通讯录等局部功能 | `npm run typecheck` + 目标页面手动/浏览器检查；必要时只跑相关 Playwright 用例 | 影响 Gateway、未读、发送链路时升级。 |
| App 单组件/单 provider | `flutter analyze --no-fatal-infos` 或定向 `flutter test <受影响测试文件>` | 不默认跑全量 Flutter 测试。 |
| 文档、方案、矩阵更新 | 文档一致性检查 | 不跑代码测试。 |
| 纯样式调整 | 浏览器/真机截图或目检 + 必要的 typecheck | 只在样式改动影响公共布局时补自动化。 |

轻量级验证的最终回复必须写清楚：

- 执行了哪些轻量命令或检查。
- 哪些重量级测试没有跑。
- 为什么本轮不需要重量级测试。

### 3.1.2 重量级验证触发条件

以下情况需要执行重量级验证，或至少在执行前明确提示用户：

- 用户明确要求“完整回归、全量测试、重量级测试、打包、发布前检查”。
- 正式包、release APK、Windows 安装包、发布候选或交付验收。
- 登录、验证码、token、权限、空间切换、admin/client/widget 域隔离变化。
- 普通 IM、群聊、在线客服、Gateway、未读、已读、推送、离线缓存、多端同步变化。
- 文件/图片/音视频、系统权限、后台、通知、弱网、错误诊断上报变化。
- 公共 API client、公共组件、缓存层、状态管理层、大范围重构。
- 轻量验证失败，或无法判断风险边界。

## 5. 自动化覆盖与人工测试分界

自动化测试执行完成后，测试结论必须分成两部分记录：

1. 自动化已覆盖功能：说明哪些模块已经通过模型、用例、provider、datasource、权限矩阵或组件测试验证。
2. 仍需人工测试功能：说明哪些能力因为依赖真机、相机、通知、通话、多账号、多端、服务端真实数据或复杂交互，不能只靠自动化判断。

自动化测试和代码验证可以直接作为功能矩阵证据。只有涉及设备能力、系统权限、真实网络、多端状态同步、真实服务端数据或角色真实账号可见性的场景，即使自动化通过，也还要进入联调或人工验收。

### 5.3 人工测试清单生成规则

人工测试清单不单独凭感觉编写，必须由三类来源合并：

- 自动化测试缺口：本节“必须人工测试”列中的相关项。
- 功能矩阵缺口：[功能矩阵和实现情况](02-功能矩阵和实现情况.md) 中状态为“待联调/人工”“部分可用”“需服务端”“未实现”的条目；“代码已验证”的条目只把剩余联调/人工项加入清单。
- 发布风险缺口：[发布风险与检查清单](06-发布风险与检查清单.md) 中未关闭的 S0/S1 风险。

人工测试清单最终回填到 [版本交付报告](12-版本交付报告.md)，并关联失败日志、requestId、录屏或截图。
