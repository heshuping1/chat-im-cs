# PC 端国际化方案

## 1. 目标与范围

本方案仅覆盖 `lpp_pc_client` PC 客户端，不包含 `lpp_mobile` 移动端。

目标语言：

| 语言 | Locale |
| --- | --- |
| 简体中文 | `zh-CN` |
| 繁体中文 | `zh-TW` |
| 英语 | `en` |
| 日语 | `ja` |
| 越南语 | `vi` |
| 泰语 | `th` |

默认语言为 `zh-CN`。任何缺失翻译必须回退到 `zh-CN`，不得出现空白文案、渲染异常或业务流程中断。

## 2. DDD 设计原则

国际化属于展示层能力，不属于领域能力。语言切换只能影响用户界面展示，不得改变领域模型、业务状态、接口协议、缓存结构或领域规则。

核心原则：

1. Domain/Data/Runtime 层不得依赖 i18n Provider、React Hook、当前语言或翻译字典。
2. 领域对象、DTO、store、IPC payload 中只保存稳定 code、id、type、status，不保存翻译后的文案。
3. 用户可见文案只允许在 Presentation 边界转换，包括 React component、presenter、view-model、display adapter。
4. 业务状态、错误码、消息类型、角色类型通过 `xxxLabelKey()` 或 display descriptor 映射为 translation key，再由 UI 调用 `t()`。
5. 服务端返回内容和用户生成内容保持原样展示，不做自动翻译。
6. 语言切换不得触发业务请求参数变化，除非后端接口明确需要 UI locale，且该字段只作为展示偏好传递。

## 3. DDD 分层边界

推荐边界：

| 层 | 允许做什么 | 禁止做什么 |
| --- | --- | --- |
| Domain | 保存业务状态、领域 code、领域规则 | import i18n、返回翻译文案、读取 locale |
| Data/API | 归一化 DTO、保留服务端 code、处理协议兼容 | 把 API code 翻译成用户文案 |
| Runtime/Electron | 处理系统能力、IPC、通知桥接底层数据 | 依赖 React i18n Hook |
| Application/View Model | 组合页面数据，输出 display descriptor 或 translation key | 把翻译后的字符串写回领域状态 |
| Presentation | 调用 `t()`，渲染本地化文案 | 修改业务枚举、协议值、缓存 key |

示例：客服状态。

```ts
// 领域/数据层：稳定状态
type CustomerServiceStatus = 'queueing' | 'serving' | 'closed';
```

```ts
// 展示映射层：状态到翻译 key
export function customerServiceStatusLabelKey(status: CustomerServiceStatus) {
  switch (status) {
    case 'queueing':
      return 'customerService.status.queueing';
    case 'serving':
      return 'customerService.status.serving';
    case 'closed':
      return 'customerService.status.closed';
    default:
      return 'customerService.status.unknown';
  }
}
```

```tsx
// Presentation：最终翻译
const { t } = useI18n();
return <span>{t(customerServiceStatusLabelKey(status))}</span>;
```

## 4. 不翻译范围

以下内容不得纳入 UI 国际化替换：

- 用户生成内容：聊天正文、客户留言、工单内容、知识库原文。
- 业务实体名称：昵称、企业名、空间名、群名、客户名、标签名。
- 服务端协议值：API 字段名、状态码、错误 code、Gateway event type、缓存 key。
- 业务状态内部值：模块 key、枚举值、store 持久化 key、路由标识、Electron IPC channel。
- 文件名、URL、电话号码、邮箱、邀请码、企业码。

展示给用户的本地兜底文案、按钮、标题、提示、错误说明、空态、加载态、通知标题等应逐步国际化。

## 5. 总体策略

采用“轻量 i18n 底座 + DDD 展示映射 + 渐进迁移”的方式：

1. 先新增 i18n Provider、语言类型、字典、fallback 和语言切换能力。
2. 保持默认 `zh-CN`，确保不切换语言时现有体验不变化。
3. 为领域状态、错误码、消息类型、角色类型建立稳定的 label key mapper。
4. 按模块逐步替换硬编码文案，每个模块独立验收。
5. 业务逻辑、API、缓存结构、IPC 能力不因国际化改造而变化。
6. 每批迁移后运行类型检查、构建和必要的 UI smoke 验证。

不建议第一阶段直接引入重型 i18n 框架或全量重构文案。当前 PC 端硬编码中文较多，先建立可回滚底座，再分模块迁移风险更低。

## 6. 推荐目录

在 `lpp_pc_client/src/renderer` 下新增：

```text
i18n/
  locales.ts
  dictionary.ts
  I18nProvider.tsx
  useI18n.ts
  messages/
    zh-CN.ts
    zh-TW.ts
    en.ts
    ja.ts
    vi.ts
    th.ts
```

职责：

- `locales.ts`：定义支持语言、默认语言、语言标签、locale 归一化。
- `messages/*.ts`：每种语言的字典。
- `dictionary.ts`：聚合字典。
- `I18nProvider.tsx`：提供 `locale`、`setLocale`、`t`。
- `useI18n.ts`：组件读取国际化能力的 Hook。

业务展示映射应放在对应模块 owner 下，例如：

```text
data/customer-service/customer-service-display.ts
messages/models/messageDisplayKeys.ts
contacts/models/contactDisplayKeys.ts
data/api/api-error-display-keys.ts
```

这些文件只返回 translation key 或 display descriptor，不返回最终翻译字符串。

## 7. 核心接口约定

组件侧只依赖以下接口：

```ts
type AppLocale = 'zh-CN' | 'zh-TW' | 'en' | 'ja' | 'vi' | 'th';

type I18nContextValue = {
  locale: AppLocale;
  setLocale: (locale: AppLocale) => void;
  t: (key: string, params?: Record<string, string | number>) => string;
};
```

示例：

```tsx
const { t } = useI18n();

return <button type="button">{t('common.save')}</button>;
```

参数替换示例：

```tsx
t('message.unreadCount', { count: 3 });
```

字典：

```ts
message: {
  unreadCount: '{count} 条未读',
}
```

## 8. 错误处理设计

错误处理必须先归一化为稳定错误类型，再在展示层翻译。

推荐流程：

```text
API/raw error -> normalized error code -> display key -> t(display key)
```

示例：

```ts
type UserFacingErrorCode =
  | 'network'
  | 'unauthorized'
  | 'forbidden'
  | 'tenantAlreadyMember'
  | 'unknown';
```

```ts
export function errorMessageKey(code: UserFacingErrorCode) {
  switch (code) {
    case 'network':
      return 'error.network';
    case 'unauthorized':
      return 'error.unauthorized';
    case 'tenantAlreadyMember':
      return 'error.tenantAlreadyMember';
    default:
      return 'error.unknown';
  }
}
```

禁止：

```ts
throw new Error(t('error.network'));
```

允许：

```ts
throw new AppError('network');
```

再由 Presentation 层转换：

```tsx
t(errorMessageKey(error.code));
```

## 9. 存储策略

第一阶段使用独立 `localStorage` key 保存语言偏好：

```text
lpp.pc.locale
```

暂不直接改造 `PcSettings.language`。当前 `PcSettings.language` 是显示文案型字段，直接改成 locale code 会引入设置 schema 迁移风险。待底座和主要页面稳定后，可单独做设置迁移。

读取策略：

1. 读取 `localStorage['lpp.pc.locale']`。
2. 若值不在支持列表中，回退 `zh-CN`。
3. 写入时只允许写入 `AppLocale`。

## 10. 字典规则

`zh-CN` 是模板语言，必须完整。

其他语言要求：

- key 结构与 `zh-CN` 一致。
- 缺失 key 必须在检查脚本中暴露。
- 运行时仍需 fallback 到 `zh-CN`。
- 不允许在组件中拼接半句翻译，带变量的句子使用完整 key。
- key 命名使用业务上下文，不使用中文拼音或页面行号。

推荐命名：

```text
common.save
common.cancel
app.loading
nav.messages
settings.language
auth.login
message.imageFallback
customerService.status.queueing
contacts.searchPlaceholder
error.network
```

## 11. 迁移原则

每次只迁移一个明确模块，避免跨模块大范围替换。

优先级：

1. 全局入口、加载、错误边界、空态。
2. 设置页和语言选择闭环。
3. 登录注册。
4. 侧边栏和主框架导航。
5. 消息模块。
6. 在线客服模块。
7. 通讯录、企业空间、收藏、知识库、工单、数据中心。

迁移时注意：

- 页面组件中的静态展示文案使用 `t()`。
- model 或 data 层的用户可见兜底文案应先改为稳定 code 或 translation key mapper。
- 服务端错误 code 保持原样，只在展示层映射为本地化文案。
- 用户数据和服务端业务内容原样展示。
- 不允许将翻译后的字符串写入 store、cache、API request、IPC payload。

## 12. 日期、数字和排序

已迁移模块中，用户可见格式化应逐步接入当前 `locale`：

- 日期时间：`Intl.DateTimeFormat(locale, options)`。
- 数字：`Intl.NumberFormat(locale, options)`。
- 展示排序：可使用 `localeCompare(value, locale)`。

业务排序、服务端排序和稳定 ID 排序不得因语言切换而改变。

语言与时区解耦。切换语言不得自动改变用户时区。

## 13. DDD 门禁

新增或迁移代码必须满足：

- `src/renderer/i18n/**` 不 import 业务 feature。
- `src/renderer/data/**`、`src/renderer/*/models/**` 不 import `useI18n`、`I18nProvider`。
- Domain/Data 层不得调用 `t()`。
- store 持久化值不得包含翻译后的 UI 文案。
- label mapper 单测覆盖未知状态 fallback。
- 字典完整性检查覆盖 6 种语言。

建议增加扫描规则：

```text
禁止在 data/**、models/**、runtime/** 中出现：
- useI18n
- I18nProvider
- t(
- localeLabels
```

允许这些层返回：

```ts
type DisplayKey = string;
type DisplayDescriptor = {
  key: string;
  params?: Record<string, string | number>;
};
```

## 15. 风险控制

- 默认语言保持 `zh-CN`，降低首次接入风险。
- 底座 PR 与模块迁移 PR 分离。
- 翻译 key 完整性由脚本检查，不依赖人工逐页比对。
- DDD 边界由扫描脚本和 code review 双重检查。
- 每个模块可独立回滚。
- 不在国际化任务中重构业务状态、API、IPC 或缓存模型。
