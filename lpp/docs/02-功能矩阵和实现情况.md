## 1. 状态口径

| 状态 | 含义 |
| --- | --- |
| 可用 | 有入口、代码实现、接口或本地逻辑，且具备自动化、接口联调或真机证据。 |
| 代码已验证 | 功能逻辑可通过代码审查、模型测试、组件测试、provider/usecase 测试或权限矩阵测试判断，且当前证据已经覆盖该功能的核心规则；剩余项通常是联调、真机体验或异常边界。 |
| 待联调/人工 | 已完成我能做的代码、自动化、接口脚本或单机真机验证后，仍必须依赖真实账号、验证码送达、真机权限、多账号、多端、真实服务端数据、权限拒绝态或人工体验验收。不是指一定需要验证码；标此状态前应先说明我已经验证了什么。 |
| 部分可用 | 只实现了功能的一部分，或者只能覆盖局部场景，尚不能按完整功能交付。 |
| 未实现 | 没有可用入口或只有占位。 |
| 需服务端 | 客户端无法单独完成，需要服务端接口、字段、事件、权限或审计支持；当前说明必须明确属于“缺失接口”“接口不可用/待修复”“字段/事件缺失”还是“接口语义待确认”。 |
| 不做/延期 | 本轮明确不纳入交付或后续独立工程承接。 |

### 1.1 验证责任分层

后续更新矩阵时默认按以下顺序处理，不能直接把功能丢给人工验收：

| 层级 | 谁验证 | 适用内容 | 结果写法 |
| --- | --- | --- | --- |
| L1 代码验证 | Codex | 路由、页面入口、接口路径、请求体、响应模型、权限判断、本地缓存、状态机。 | `代码已验证`，证据写具体文件和测试。 |
| L2 自动化验证 | Codex | provider/usecase/model/widget/权限矩阵/弱网纯函数/合同测试。 | `代码已验证` 或 `可用`，证据写测试命令和用例数。 |
| L3 API/脚本验证 | Codex | 真实环境接口可调用、错误码、权限拒绝、临时会话模拟、群发任务。 | `可用` 或 `待联调/人工`，只把真实账号/业务副作用留作缺口。 |
| L4 单机真机验证 | Codex | 已连接 Android 上的启动、页面可见性、弱网提示、截图、安装、基础导航。 | `可用` 或 `代码已验证`，证据写设备和截图。 |
| L5 必须配合验证 | 用户/服务端 | 验证码真实送达、多账号双端、音视频听感、后台推送、真实权限数据、服务端缺接口。 | 才能写 `待联调/人工` 或 `需服务端`。 |

### 1.2 支持客户端口径

后续矩阵必须同时写清楚支持客户端，避免把 App 已有能力误认为 PC 已有能力，或把 PC 专属能力误塞回 App。

| 支持客户端 | 含义 |
| --- | --- |
| App | Flutter Android/iOS 客户端。当前主交付端是 Android；iOS 按同一套 Flutter 代码能力评估，设备权限和推送仍需单独验收。 |
| PC | Electron + React + TypeScript Windows 客服客户端。PC 功能范围应大于 App，承载更高密度的 IM、在线客服、客户上下文、桌面提醒、多窗口和效率工具。 |
| App + PC | 两端都应支持，但实现形态可以不同。App 偏移动轻量处理，PC 偏高效率工作台。 |
| 后续/管理端 | 不进入 App 或 PC 客服端首版，后续由 PC 管理扩展、Web 管理后台或服务端能力承接。 |

### 1.4 PC 查漏补缺清单

PC 功能目标必须大于 App，但不能把未实现能力写成已交付。当前优先补齐顺序如下：

| 优先级 | PC 缺口 | 判断依据 |
| --- | --- | --- |
| P0 | token 刷新和空间切换完整链路 | PC 已补正式环境登录、选择企业、个人空间选择、401 自动清登录态；下一步补 refreshToken 自动续期、多企业切换后的缓存隔离和 Gateway 重连验收。 |
| P0 | Gateway 实时事件接入 | PC 已接 SignalR `/ws/client`、`msg.new/read/recalled`、`space.notice`、客服线程、SLA、好友、presence、强制下线等事件；下一步用真实多端联调消息到达速度、通知点击跳转和离线补偿。 |
| P0 | 通讯录组织架构真实接口 | PC 已接 `/departments/` 和 `/departments/{departmentId}/members`，并从正式好友、成员、群会话聚合通讯录；下一步补真实大组织、权限、搜索和详情字段验收。 |
| P0 | PC 自动化脚本纳入常规回归 | 新增 `scripts/pc/test/run_pc_browser_tests.sh`，后续每次 PC 改动都先跑。 |
| P1 | 图片/文件/拖拽/截图粘贴/下载打开 | PC 已补图片/文件选择、粘贴和拖拽发送；下载打开、文件记录、弱网断点和大文件体验继续补。 |
| P1 | 桌面通知、托盘、弱网重连、诊断包 | PC 常驻使用，需要稳定提醒和问题定位。 |
| P1 | 客户上下文真实接口联调 | 右侧常驻上下文是 PC 相对 App 的核心优势。 |
| P2 | 多窗口、音视频、完整客户详情、质检审计和管理扩展 | 复杂能力后续扩展，不阻塞首版 IM + 在线客服。 |

### 1.5 PC 端功能完成度基线

本表后续作为 PC 完成度检查的主表。判定规则：如果 App 与 PC 没有产品差异，PC 默认按 App 的业务口径实现；如果 PC 因桌面形态需要差异化，必须在“PC 形态/差异”里写明理由，不能另起一套不兼容逻辑。

### 1.6 App/PC 全量一致性核对清单

本表是后续逐项核对 App 与 PC 功能一致性的父级主清单。新增功能必须先补到本表，再在下面的模块章节展开子级明细。核对口径：

- `必须一致`：接口、权限、状态、数据口径和异常处理必须一致，UI 可以按端形态微调。
- `形态可不同`：业务结果一致，但 App/PC 交互可以不同，例如 App 用底部弹层，PC 用右侧常驻面板。
- `PC 专属`：只属于桌面客户端的效率能力，App 不要求实现。
- `App 专属`：只属于移动端设备能力或移动入口，PC 不要求实现。
- `管理扩展`：管理员/所有者或后续后台能力，不作为 PC 客服首版主链路。

维护规则：

- 父级只记录“功能是否存在、App/PC 状态、差异和优先级”，不堆接口细节和长测试证据。
- 子级模块清单必须能追溯到父级编号，例如 IM 消息章节里的图片消息应归属 `MATRIX-IM-006`。
- 如果子级发现了新功能点，先补父级编号，再补子级说明，避免模块章节各自生长成另一套清单。
- 如果 App/PC 功能无差异，子级实现默认按 App 口径；如果 PC 要差异化，必须在父级“核对结论/差异”写明理由。

### 5.1 弱网健壮性结论

| 场景 | 当前客户端能力 | 剩余缺口 |

| 消息发送失败 | 文本消息乐观插入，网络错误后进入待发队列，重连后按序重发；服务端拒绝类错误不重试。 | 媒体上传失败目前直接失败，无法完整离线排队；大文件弱网断点续传需服务端/上传协议支持。 |

## 12. 发布前必须处理

S0 必须清零：

- 任一主要角色无法登录或进入空间。
- 普通消息无法发送。
- 定时消息改坏普通发送或 `+` 工具面板。
- admin API 使用普通 tenant token。
- 客户能看到内部工作台或客户池。
- 管理员/所有者以只读视角进入客户会话后仍能发言。
- 客服无法处理客户咨询。
- 群禁言权限与产品规则相反。
- P0 路径频繁崩溃或白屏。

S1 需要修复或明确延期：

- 好友申请双方状态不一致。
- 客户标签保存或筛选失败。
- 客服接管并发异常。
- 会话冻结/解冻语义不清。
- 音视频已有人工作证据，但双真机、后台、锁屏、小窗和音频路由未形成专项报告；推送和后台能力未形成真机报告。
- 大数据量性能、低端机和后台长稳基线仍缺专项报告。
