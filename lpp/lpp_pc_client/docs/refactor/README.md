## 1. 这个目录解决什么问题

本目录是 PC 客服 IM 客户端重构的知识库入口，用于支持 Codex 和工程师在不依赖聊天上下文的情况下继续工作。

核心目标：

1. 渐进式披露：先从短入口进入，再按任务需要阅读深层文档。
2. 方案可查：PC 端最终技术方案和重构原则可查。
3. 任务可查：阶段、任务、状态、风险、验收可查。
4. 决策可查：重要架构选择有 ADR 记录。
5. 验证可查：测试、手工验收、诊断日志证据可查。

## 2. 快速入口

新会话或新任务按这个顺序读：

1. `../../AGENTS.md`
2. `./PC端核心架构技术方案.md`
3. `./PC端重构任务矩阵.md`
4. 当前阶段详细方案或执行计划

如果只是执行一个明确任务，至少确认：

```text
任务编号：
所属阶段：
影响范围：
风险等级：
验收等级：
是否涉及技术选型：
是否涉及公共能力：
是否需要诊断日志：
是否需要负责人确认：
```

## 4. 核心理念

1. 不推倒重写，渐进式迁移。
2. 不为审计而审计，所有审计结论必须能落到重构任务。
3. 不重复造轮子，先查已有能力，再决定抽象或引入依赖。
4. 页面做装配，规则进 domain/application/service。
5. Gateway、API、Electron 都是系统边界，必须 typed、validated、observable。
6. 核心链路要能靠测试、日志、验证记录排查。
7. 文档是执行资产，不是事后总结。
8. 文件大小只是预警，职责边界和修改路径稳定性才是治理目标。

---

## 4.1 Agent-first 采纳原则

结合 OpenAI Harness Engineering 文章，本项目采纳以下原则：

1. `AGENTS.md` 只做短入口和地图，不做百科全书。
2. 代码仓库是记录系统；聊天中的共识、架构判断、验证证据要尽量沉淀到仓库。
3. 复杂计划是一等工件，需要版本化、可检索、有进度和决策日志。
4. 让应用对 Codex 可读：核心链路必须提供结构化日志、诊断记录和可复现验证步骤。
5. 架构边界最终要从文档升级为机械约束，例如 lint、结构测试、依赖方向检查。
6. 在 Gateway、API、Electron IPC 等边界处验证数据形状，避免页面或业务逻辑猜测 raw payload。
7. 自动 doc lint、doc-gardening、完整可观测性栈先登记为后续工程任务，不在 P0/P1 过早引入重型系统。

## 7. 执行日志要求

复杂任务必须在执行计划或验证记录中追加进度和决策日志。

建议格式：

```text
日期：
任务编号：
动作：
修改范围：
决策：
验证：
遗留风险：
下一步：
```

这不是为了增加流程，而是为了让后续 Codex 能直接从仓库恢复上下文。

## 8. 暂缓自动化项

以下事项有价值，但不在当前 P0/P1 立即实现：

1. 文档结构 linter。
2. 文档交叉链接 CI。
3. 定期 doc-gardening agent。
4. 自动检测文档与代码行为漂移。

原因：当前工程门禁和重构地基尚未稳定。先通过轻量文档入口和任务矩阵建立秩序，后续在 P8 工程门禁阶段自动化。
