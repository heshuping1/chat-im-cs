# 普通 IM P0 实时与未读一致性详细方案

状态：当前有效

日期：2026-06-06

适用范围：`lpp_pc_client/src/renderer/data/gateway`、`lpp_pc_client/src/renderer/data/im-read`、`lpp_pc_client/src/renderer/messages`

## 1. 设计目标

本文档只覆盖两个 P0 增量需求：

1. `PC-REQ-20260606-001`：Gateway CORS / Origin 与真实 push 联调闭环。
2. `PC-REQ-20260606-002`：自己消息未读一致性服务端修正。

这两个需求必须基于当前 PC 端已有实现继续补齐，不重新设计一套脱离代码的实时消息或未读系统。

## 2. 当前代码基础

### 2.1 Gateway 实时链路

当前代码已经具备以下基础：

| 能力 | 当前代码入口 |
| --- | --- |
| Gateway 组件入口 | `src/renderer/components/GatewayBridge.tsx` |
| SignalR 连接管理和重试 | `src/renderer/data/gateway/gateway-connection-manager.ts` |
| Gateway 事件注册 | `src/renderer/data/gateway/gateway-event-registry.ts` |
| Gateway 事件路由 | `src/renderer/data/gateway/gateway-event-router.ts` |
| IM Gateway 消息副作用 | `src/renderer/data/gateway/gateway-im-side-effects.ts` |
| IM Gateway 缓存写入 | `src/renderer/data/gateway/im-gateway-cache.ts` |
| Gateway 诊断 | `src/renderer/data/gateway/gateway-health-diagnostics.ts`、`gateway-diagnostics.ts` |
| 断线补偿 | `src/renderer/data/gateway/message-gap-sync-coordinator.ts` |

当前连接策略：

1. `GatewayBridge` 使用 `${apiBaseUrl}/ws/client` 建立 SignalR 连接。
2. URL 查询参数携带 `deviceId`、`clientInstanceId`、`clientPlatform=pc`。
3. 鉴权通过 `accessTokenFactory: () => session.tenantToken`。
4. 明确使用 `withCredentials: false`，不依赖跨域 cookie。
5. 连接失败或关闭后由 `GatewayConnectionManager` 进入 retry，不阻塞页面。
6. 连接成功或重连成功后触发 `triggerMessageGapSync` 做断线补偿。

### 2.2 未读一致性链路

当前代码已经具备以下基础：

| 能力 | 当前代码入口 |
| --- | --- |
| IM read 状态模型 | `src/renderer/data/im-read-model.ts` |
| IM read store 入口 | `src/renderer/data/im-read/im-read-store.ts` |
| 发送成功后本地 readSeq 补偿 | `src/renderer/data/im-read/im-send-succeeded-service.ts` |
| 当前会话 read receipt 同步 | `src/renderer/messages/hooks/useDirectReadReceiptSync.ts` |
| Gateway 收到消息后的未读归并 | `src/renderer/data/gateway/gateway-im-side-effects.ts` |
| 会话列表展示模型 | `src/renderer/messages/models/messageConversationListModel.ts` |

当前未读规则：

1. 自己发送成功后，`send.message_succeeded` 会推进 `myReadSeq`、清空当前会话未读，并调用 read 接口。
2. Gateway 收到自己消息时，`mergeImGatewayMessage` 会识别 `selfMessage`，强制 `unreadCount = 0`，并推进本地 readSeq。
3. 当前打开且可见的会话收到新消息时，会推进 readSeq、清空未读，并调用 read 接口。
4. 非当前会话收到他人新消息时，才增加未读和触发提醒。
5. API snapshot 如果落后于本地 pending read，会触发 `retry_pending_read` 补偿。

## 3. 需求一：Gateway CORS / Origin 与真实 push 联调闭环

### 3.1 当前问题

PC 当前普通 HTTP API 可访问，但真实 UI 审计中 `/ws/client/negotiate` 被浏览器 CORS 拦截，导致以下能力不能宣称完整闭环：

1. `msg.new` 实时到达。
2. `msg.read` 实时到达。
3. 撤回、空间通知、好友事件等 Gateway 事件实时刷新。
4. 当前会话实时已读和非当前会话实时红点。

当前前端已有 refetch 和 gap sync 补偿，但补偿不能替代真实 push。

### 3.2 交付范围

本需求交付：

1. 服务端允许 PC 来源访问 `/ws/client/negotiate` 和 `/ws/client`。
2. PC 使用现有 `GatewayBridge` 连接，不新增第二套实时通道。
3. 真实环境验证 `msg.new`、`msg.read` 能进入 `GatewayEventRouter` 并写入现有缓存和 read store。
4. Gateway 失败时仍保持现有不阻塞页面和重试策略。
5. 联调结论回填到发布检查和服务端支持文档。

本需求不交付：

1. 不改用 cookie 鉴权。
2. 不设置 `withCredentials: true`。
3. 不新增轮询替代 Gateway。
4. 不绕过 `GatewayEventRouter` 直接在组件里写缓存。

### 3.3 服务端要求

服务端必须支持：

1. `OPTIONS /ws/client/negotiate`
2. `POST /ws/client/negotiate`
3. WebSocket/SSE/LongPolling 连接到 `/ws/client`

CORS 要求：

1. 允许开发态来源：`http://127.0.0.1:5173`，必要时包括 `http://localhost:5173`。
2. 允许 Electron 打包态来源：按实际打包环境确认，不能要求浏览器 cookie。
3. 允许请求头：`Authorization`、`Content-Type`。
4. PC 端当前 `withCredentials: false`，服务端不能依赖跨域 cookie。
5. 如果服务端未来要求 credentials，必须先更新架构边界和测试，不能直接改前端。

### 3.4 前端设计

前端沿用当前实现：

1. `GatewayBridge` 继续负责连接创建、事件订阅、连接状态和 heartbeat。
2. `GatewayConnectionManager` 继续负责 start failure、close、retry、stale generation 防护。
3. `gateway-event-router` 继续负责 IM / 在线客服 / space / friend 等事件分流。
4. `gateway-im-side-effects` 继续负责 IM 消息写缓存、未读归并和提醒策略。
5. `message-gap-sync-coordinator` 继续负责连接成功和重连后的补偿。

允许的前端增量只包括：

1. 根据真实联调补充诊断字段。
2. 根据服务端标准事件补充 Gateway payload normalizer。
3. 根据真实事件补充单测、浏览器测试或联调记录。

禁止的前端改动：

1. 禁止在页面组件中直接处理 raw Gateway payload。
2. 禁止新增绕过 `GatewayConnectionManager` 的连接逻辑。
3. 禁止用 mock push 结果替代真实 UI 联调结论。

### 3.5 验收标准

1. PC 启动后 Gateway 状态能从 `connecting` 进入 `connected`。
2. `/ws/client/negotiate` 不再被 CORS/Origin 拦截。
3. 另一个客户端发送普通 IM 消息后，PC 能通过 `msg.new` 实时展示消息。
4. 非当前会话收到他人消息时，PC 实时出现未读红点和提醒。
5. 当前打开会话收到他人消息时，消息实时展示但不产生未读红点。
6. 另一端已读后，PC 能通过 `msg.read` 更新 read receipt。
7. Gateway 断开后进入 retry；重连后触发 gap sync，不丢消息。

## 4. 需求二：自己消息未读一致性服务端修正

### 4.1 当前问题

前端已经对自己消息未读做了多层兜底，但服务端如果继续返回不一致数据，仍会导致多端和 refetch 场景存在风险。

已知风险包括：

1. `lastMessage` 是当前账号自己发送，但会话列表仍返回 `unreadCount > 0`。
2. push/pull/history 中缺少可稳定判断自消息的 sender 字段。
3. 服务端 readSeq 与 PC 本地 pending readSeq 不一致，导致 refetch 后未读复活。

### 4.2 交付范围

本需求交付：

1. 服务端修正会话列表、push、pull/history 中的 self/sender 字段。
2. 当最后一条消息是当前账号自己发送时，服务端返回 `unreadCount: 0`。
3. PC 继续使用现有 `im-read-model` 和本地补偿，不新增第二套未读模型。
4. 真实环境验证本机发送、另一设备发送、refetch、history、push 场景的一致性。

本需求不交付：

1. 不承诺全平台强一致已读，除非服务端和多端协议另行完成。
2. 不在前端伪造服务端持久化 readSeq。
3. 不用显示层硬改掩盖 normalizer 或 API 合同问题。

### 4.3 服务端字段要求

如果消息是当前账号发送，服务端至少稳定返回以下字段之一用于判定：

1. `isSelf: true`
2. `isMine: true`
3. `direction: "out"` 或 `direction: "outgoing"`
4. `senderUserId` 等于当前用户 ID
5. `senderId` 等于当前用户 ID
6. `fromUserId` 等于当前用户 ID
7. `senderPlatformUserId` / `platformUserId` 等于当前用户平台 ID
8. `senderLppId` / `lppId` 等于当前用户 LPP ID

会话列表要求：

1. `lastMessage` 是当前账号发送时，`unreadCount` 必须为 `0`。
2. 稳定返回 `lastMessageSeq`。
3. 稳定返回当前用户 `lastReadSeq` 或等价字段。

push/history 要求：

1. `msg.new` payload 必须包含 `conversationId`、`conversationType`、`conversationSeq`、`messageId`、`messageType`。
2. 自己消息必须携带 self/sender/direction 证据。
3. `msg.read` payload 必须包含 `conversationId`、`readSeq` 和 reader identity。

### 4.4 前端设计

前端沿用当前实现：

1. `im-read-model.ts` 继续作为未读状态机。
2. `im-send-succeeded-service.ts` 继续在发送成功后推进本地 readSeq 并调用 read 接口。
3. `gateway-im-side-effects.ts` 继续在 Gateway 收到自己消息时强制本地 unread 为 0，并推进 readSeq。
4. `useDirectReadReceiptSync.ts` 继续合并 direct peer read status。
5. `messageConversationListModel` 继续负责会话列表展示态，不能绕过 read model。

允许的前端增量只包括：

1. 根据服务端最终字段补充 normalizer 和合同测试。
2. 补充真实 payload 下的 `isMine` / `isSelf` / sender identity 测试。
3. 补充诊断，定位服务端返回 self lastMessage 但 unreadCount 非 0 的场景。

### 4.5 验收标准

1. 本机发送文本、图片、文件后，当前会话不出现未读。
2. 本机发送成功并拿到 `conversationSeq` 后，PC 调用 read 接口并持久化 readSeq。
3. 另一个设备用同一账号发送消息后，PC 收到 push/refetch/history 时不增加未读。
4. `lastMessage` 为自己消息时，会话列表不显示未读红点。
5. 非当前账号发送的新消息仍能正常增加未读。
6. 当前打开会话收到他人消息时不产生未读，并调用 read。
7. refetch 后收藏、撤回、删除、转发和未读状态不回退。
