# 普通 IM P1 服务端能力补齐详细方案

状态：当前有效

日期：2026-06-06

适用范围：`lpp_pc_client/src/renderer/messages`、`lpp_pc_client/src/renderer/data/api`

## 1. 范围

本方案覆盖普通 IM 后续继续开发优先级中的 P1 服务端能力补齐。它基于当前 PC 代码已经实现的前端入口和兜底行为，不重新设计消息中心。

关联需求：

1. `PC-REQ-20260606-003`：会话置顶、免打扰、隐藏、删除、恢复持久化。
2. `PC-REQ-20260606-004`：本机已落库消息搜索。
3. `PC-REQ-20260606-005`：批量转发和批量删除服务端批量接口。
4. `PC-REQ-20260606-006`：群资料与消息元数据字段闭环。

以上需求的需求状态和设计状态均为完成，开发准入为准入。其中 `PC-REQ-20260606-003`、`PC-REQ-20260606-005`、`PC-REQ-20260606-006` 已完成前端接入，等待真实服务端联调；`PC-REQ-20260606-004` 转入 PC 本地数据方案，不新增服务端全文搜索接口。

## 3. 设计原则

1. 服务端状态必须走 API 持久化，PC 不把本地 session 状态伪装成跨设备能力。
2. 新接口全部进入 `src/renderer/data/api/endpoints.ts` 和 `MessagesApiClient` 或既有 API client，不在组件里拼接路径。
3. 接口返回先经过 normalizer 或模型函数对齐，再进入 React Query 缓存和 UI。
4. 对服务端部分失败保持可见：批量能力必须展示成功、失败、部分失败，而不是只给单一成功提示。
5. 字段缺失时保持空状态或降级展示，不制造假数据。

## 4. 会话持久化方案

### 4.1 目标行为

用户在 PC 对普通 IM 会话执行置顶、取消置顶、免打扰、取消免打扰、隐藏、删除或恢复后：

1. 当前列表立即反馈操作结果。
2. refetch 后状态仍保持。
3. 重新登录、刷新、换设备后状态一致。
4. 服务端失败时回滚本地乐观状态并展示错误。

### 4.2 接口

| 操作 | Endpoint | Method | Body | 返回 |
| --- | --- | --- | --- | --- |
| 置顶/取消置顶 | `/api/client/v1/conversations/{conversationId}/pin` | `PUT` | `{ "isPinned": true }` | `{ "conversationId": "...", "isPinned": true, "updatedAt": "..." }` |
| 免打扰/取消免打扰 | `/api/client/v1/conversations/{conversationId}/mute` | `PUT` | `{ "isMuted": true }` | `{ "conversationId": "...", "isMuted": true, "updatedAt": "..." }` |
| 隐藏/恢复 | `/api/client/v1/conversations/{conversationId}/visibility` | `PUT` | `{ "hidden": true }` | `{ "conversationId": "...", "hidden": true, "updatedAt": "..." }` |

删除会话如果语义不同于隐藏，需要服务端明确是否复用 visibility，还是新增 delete endpoint。PC 不能把隐藏当成永久删除承诺。

### 4.3 前端改造点

1. 在 `endpointPlan` 增加 conversation pin/mute/visibility endpoint。
2. 在 `MessagesApiClient` 增加 `setConversationPinned`、`setConversationMuted`、`setConversationVisibility`。
3. 将 `useMessageInteractionHandlers.handleConversationMenuAction` 从本地 Set 改为调用 mutation；本地 Set 只保留成功后的即时 UI 同步。
4. 操作成功后 invalidate conversations query，并保留当前会话选择策略。
5. 会话排序继续沿用现有 `sortMessageConversations`，不改排序规则。

### 4.4 验收

1. 置顶后 refetch 仍在顶部。
2. 免打扰后列表展示静音图标，刷新仍存在。
3. 隐藏/删除后会话从列表移除，恢复接口返回后可重新出现。
4. 接口失败时 UI 不保留错误状态。

## 5. 本机已落库搜索方案

### 5.1 目标行为

用户在会话内搜索关键字或按类型筛选时，结果应覆盖本机 SQLite 本地库已经落库的消息，而不只覆盖当前组件已加载消息。

本方案不做跨设备全量历史搜索，不新增服务端全文搜索接口。

### 5.2 本地查询

LocalDataService query：

| 字段 | 说明 |
| --- | --- |
| `keyword` | 搜索关键字，可为空 |
| `messageType` | `all/text/image/file/voice/video/link/favorite` |
| `cursor` | 本地分页游标 |
| `limit` | 默认 50 |
| `fromTime` / `toTime` | 可选时间范围 |

### 5.3 前端改造点

1. 保留 `filterVisibleMessages` 作为当前已加载消息的即时过滤兜底。
2. 接入本地库搜索 query，搜索面板打开并输入 keyword 或选择历史类型时触发。
3. 搜索结果点击后优先在当前列表 `scrollToMessage`；本地库有消息但当前窗口未加载时，从本地库拉取上下文页。
4. 搜索 loading、empty、error 状态由搜索面板展示，不影响主消息列表可用性。
5. UI 文案不得暗示本机未落库的服务端历史也会命中。

### 5.4 验收

1. 搜索本机已落库但当前组件未加载的关键字可以命中。
2. 类型筛选可以命中本机已落库的图片、文件、视频等消息。
3. 点击结果能定位到消息；缺上下文页时给明确提示。
4. 搜索失败不清空当前聊天列表。

## 6. 批量操作方案

### 6.1 目标行为

批量转发和批量删除由服务端统一处理，并返回成功、失败、部分失败结果。PC 根据结果更新提示和缓存。

### 6.2 接口

| 操作 | Endpoint | Method | Body | 返回 |
| --- | --- | --- | --- | --- |
| 批量转发 | `/api/client/v1/messages/batch-forward` | `POST` | `{ "messageIds": ["m1"], "targetConversationId": "c2" }` | `{ "successIds": ["m1"], "failedItems": [] }` |
| 批量删除 | `/api/client/v1/messages/batch-delete` | `POST` | `{ "messageIds": ["m1", "m2"] }` | `{ "successIds": ["m1"], "failedItems": [{ "messageId": "m2", "message": "..." }] }` |

### 6.3 前端改造点

1. 在 `MessagesApiClient` 增加 `batchForwardMessages` 和 `batchDeleteMessages`。
2. 批量删除优先调用 batch endpoint，失败时不自动回退到逐条删除，避免双写和语义不一致。
3. 批量转发在多选场景使用 batch endpoint；单条右键转发可继续复用单条接口。
4. 成功后 invalidate messages 和 conversations，部分失败时保留失败项提示。

### 6.4 验收

1. 全部成功：退出多选，清空选择，提示成功数量。
2. 部分失败：退出多选，提示成功/失败数量，并允许用户重新选择失败项。
3. 全部失败：不清空用户可见上下文，展示服务端错误摘要。
4. 服务端返回未知 messageId 时不破坏本地列表。

## 7. 群资料与消息元数据字段方案

### 7.1 目标行为

群资料、群成员、@、回复引用在发送、push、pull、history、refetch 后保持可读字段一致。

### 7.2 字段要求

| 场景 | 字段要求 |
| --- | --- |
| 群公告 | `announcementId`、`content`、`updatedAt`、`updatedByDisplayName` |
| 群成员 | `userId`、`displayName`、`groupNickname`、`role`、`avatarUrl` |
| @ mentions | `mentions: [{ "userId": "...", "displayName": "..." }]` 在 send、push、pull、history 中结构一致 |
| 回复引用 | `replyToMessageId` 和 `reply: { "messageId": "...", "senderDisplayName": "...", "preview": "...", "messageType": "..." }` |

### 7.3 前端改造点

1. 群成员展示继续通过 `groupManagementModel` 归一化角色；新增字段只扩展 normalizer，不在 UI 里直接猜字段。
2. @ 输入继续复用 `extractMentions`，服务端返回 mentions 后补齐消息展示和提醒判断。
3. 回复引用继续复用 `withReplyBody` 本地摘要；历史消息优先读取服务端 `reply` 摘要。
4. 群公告沿用 `useMessageGroupManagement` 查询结果，不用本地假公告。

### 7.4 验收

1. 群公告真实显示，空公告显示空状态。
2. 群主、管理员、成员角色在群资料和成员列表一致。
3. @ 消息刷新后仍能识别被提及成员。
4. 回复消息刷新后仍展示引用摘要。
