# 普通 IM 详细方案

状态：当前有效

日期：2026-06-07

适用范围：`lpp_pc_client/src/renderer/messages`

## 1. 范围

普通 IM 覆盖 PC 端私聊和群聊的会话列表、消息收发、消息展示、媒体、未读/已读、消息操作、当前已加载范围内的搜索和群聊基础能力。

P27 纠偏任务已完成普通 IM 核心链路优先修复，覆盖本地消息库、local-first 聊天首屏、query key/scope 统一、Gateway gap sync fallback 边界和客服 unread ledger 边界。执行入口见 [PC端重构任务矩阵.md](../refactor/PC端重构任务矩阵.md) 和 [2026-06-07-pc-im-local-message-store-recovery.md](../superpowers/plans/2026-06-07-pc-im-local-message-store-recovery.md)。

P0 增量需求 `PC-REQ-20260606-001` 和 `PC-REQ-20260606-002` 的详细设计见 [05-普通IM-P0实时与未读一致性详细方案.md](05-普通IM-P0实时与未读一致性详细方案.md)。该设计基于当前 Gateway、IM read model 和消息缓存实现，不另起一套实时或未读方案。

P1 需求 `PC-REQ-20260606-003`、`PC-REQ-20260606-005`、`PC-REQ-20260606-006` 已完成前端接入，等待真实服务端联调；`PC-REQ-20260606-004` 转为本机已落库搜索，不新增服务端全文搜索接口。详细设计见 [06-普通IM-P1服务端能力补齐详细方案.md](06-普通IM-P1服务端能力补齐详细方案.md)。

不覆盖：

1. 跨设备全量历史搜索。
2. 恢复入口和删除独立语义；当前删除复用 visibility 隐藏语义。
3. 批量转发和批量删除的服务端原子事务承诺。
4. 缺字段情况下的假群公告、假角色或假 sender。
5. PC 成功消息历史本地库由 P27 的 IndexedDB message store 承载；不得再把 React Query、Zustand 内存状态或 hot cache 表述为消息本地库。

## 3. 数据流

1. 会话列表通过 API 拉取，并由 React Query 缓存。
2. 打开会话必须先从本地消息库读取已有成功消息并展示首屏，再后台拉取 direct/group messages 做服务端快照同步。
3. 发送文本或媒体时先进入发送控制器和 send outbox；发送成功后必须物化进本地消息库，不能只停留在 React Query cache。
4. 本地 echo、服务端返回、Gateway push 和 refetch 通过 normalizer/message core 对齐展示模型，并由统一写入路径同步 React Query cache 和本地消息库。
5. 未读/已读由服务端 read 接口、本地 readSeq 和当前打开会话可见性共同决定；客服 unread 不得长期依赖 IM list compat 字段作为主源。
6. 群聊自己发送消息的消息流入口只信任服务端消息快照中的 `readCount`；缺少 `readCount` 或 `conversationSeq` 时不展示入口，不用本地猜人数。
7. 点击群聊已读入口时才实时调用 `GET /api/client/v1/groups/{groupId}/read-receipts?messageId={messageId}` 拉取详情浮层；详情数据只在展示层使用，不写入本地消息库、不参与 `msg.read` 归约、不做 `readCount++`。
8. Gateway 事件用于实时更新消息、已读、撤回和会话摘要；当前 gap sync 只有 fallback refetch，真实 afterSeq/cursor 精确补洞依赖服务端合同。

## 4. 核心约束

1. 自己发送的消息不能产生未读。
2. 当前打开会话收到实时消息不产生未读红点。
3. 非当前会话收到消息应增加未读并触发提醒策略。
4. 缺少 reply 摘要时不凭 `replyToMessageId` 伪造引用内容。
5. 群 sender 缺少可读字段时不显示 UUID 作为姓名。
6. 批量操作优先使用 batch 接口，但不宣称服务端未定义的原子事务。
7. React Query 只承载服务端快照和 UI cache，不得被表述为成功消息本地库。
8. send outbox 只承载失败、发送中断和重试恢复，不等同成功消息历史库。
9. 进入已有本地消息的聊天窗口时不得用整块 loading 遮挡本地已有消息。
10. 群聊已读详情请求是用户点击后的轻量查询，不得演变为消息主链路批量拉取、轮询或本地 readCount 累加。

## 5. 服务端依赖

服务端缺口统一维护在 `../05-服务端支持.md`。普通 IM 当前重点依赖：

1. 会话置顶接口。
2. 会话免打扰接口。
3. 会话隐藏/删除/恢复接口。
4. 群公告接口。
5. 群成员昵称和角色稳定字段。
6. @ mentions 在 send、push、pull、history 中的字段一致性。
7. 回复引用摘要字段。
8. 批量转发/批量删除接口。
9. Gateway CORS/Origin。
10. self message unread 一致性字段。
11. afterSeq/cursor 精确 gap sync 合同。
12. 精确 gap sync 所需的 afterSeq/cursor 合同。

其中 Gateway CORS/Origin 和 self message unread 一致性已登记为 P0 已完成需求：

1. `PC-REQ-20260606-001`：Gateway CORS / Origin 与真实 push 联调闭环。
2. `PC-REQ-20260606-002`：自己消息未读一致性服务端修正。

以下服务端能力已登记为 P1 准入待开发需求，详细实现边界见 `06-普通IM-P1服务端能力补齐详细方案.md`：

1. `PC-REQ-20260606-003`：会话置顶、免打扰、隐藏、删除、恢复持久化。
2. `PC-REQ-20260606-004`：本机已落库消息搜索，不新增服务端全文搜索。
3. `PC-REQ-20260606-005`：批量转发和批量删除服务端批量接口。
4. `PC-REQ-20260606-006`：群资料与消息元数据字段闭环。
