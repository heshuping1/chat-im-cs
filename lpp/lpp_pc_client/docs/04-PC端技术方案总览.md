## 4. 接口和状态边界

| 类型 | 当前入口 | 规则 |
| --- | --- | --- |
| API endpoint | `src/renderer/data/api/endpoints.ts` | 不在组件里拼接口 |
| API client | `src/renderer/data/api/`、`src/renderer/data/api-client.ts` | 页面通过 client 或 domain service 调用 |
| API 合同 | `src/renderer/data/api-contract/` | 字段变化先改 normalizer 和合同测试 |
| Gateway | `src/renderer/data/gateway/` | 失败不阻塞页面，真实能力必须联调验证 |
| 服务端缺口 | `05-服务端支持.md` | 不用前端假持久化掩盖缺口 |
| 服务端数据状态 | React Query | 按 session、tenant、conversation 隔离；只承载服务端快照和 UI cache，不得冒充消息本地库 |
| 本地成功消息状态 | P27 本地消息库 | P27 执行后由 IndexedDB message store 承载成功消息 read model；scope 必须包含 apiBaseUrl、tenant/user、conversation type/id |
| 会话状态 | auth store | 支持桌面端恢复 |
| UI 状态 | workspace-ui store | 管理模块、会话、布局、面板 |
| 本地补偿状态 | im-read、send outbox、upload state | 只做补偿；send outbox 只管失败/中断恢复，不等同成功消息历史库 |

## 5. 设计准入检查

新增功能进入开发前，设计状态必须为 `完成`。最低检查项：

1. 已明确代码入口和模块边界。
2. 已明确是否需要新增或变更 API、Gateway 事件、字段、错误码。
3. 已明确状态归属：服务端数据、会话状态、UI 状态、本地补偿状态或运行时状态。
4. 已明确验证策略：必跑命令、专项单测、人工验收或替代验证。
5. 已明确是否需要更新 `05-服务端支持.md`、`technical/`、ADR 或发布文档。

P27 纠偏完成后，任何进入聊天页、消息加载速度、消息缓存、历史分页、Gateway 补洞、客服未读相关任务，必须先检查 `P27-IM-*` / `P27-CS-*` 的 owner 和验证记录。后续不得把 React Query、Zustand 内存状态或 hot cache 表述为 PC 端消息本地库。
