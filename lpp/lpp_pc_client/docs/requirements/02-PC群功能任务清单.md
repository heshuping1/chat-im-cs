# PC 群功能任务清单

状态：当前有效

日期：2026-06-06

适用范围：`lpp_pc_client/src/renderer/messages`

## 1. 定位

本文档用于跟踪 PC 普通 IM 群功能。判断顺序：

1. 服务端 API 是否已有。
2. 前端是否已经接入。
3. 是否已经有验证。
4. 是否可以宣称交付。

详细设计见 `../technical/06-普通IM-P1服务端能力补齐详细方案.md`，服务端缺口见 `../05-服务端支持.md`。

## 2. 状态定义

| 状态 | 含义 |
| --- | --- |
| 已完成 | 代码已接入，并有最小验证或明确人工验收路径 |
| 可开发 | 服务端 API 已有或字段已约定，前端可以继续做 |
| 待联调 | 前端有入口，但必须用真实服务端数据确认 |
| 待服务端 | 缺服务端 API 或字段，不应前端假实现 |
| 暂缓 | 当前不进入本阶段 |

## 4. 本轮已完成

1. `GroupMemberDto` 增加 `groupNickname` 和 `nickname` 兼容字段。
2. 群 sender 显示优先使用群昵称。
3. 群成员列表、群头像、成员操作确认文案优先使用群昵称。
4. @ 候选和 @ 提取支持群昵称。
5. 补充了群昵称相关单测。
6. 群成员邀请支持从联系人选择器勾选，同时保留 userId 手输路径；提交前会去重并排除已在群成员。
7. 群消息 `mentions` 和 `reply` 摘要支持从服务端顶层字段或 body 别名字段归一化到 `body.mentions`、`body.reply`，覆盖 push/pull/history/refetch。

## 5. 下一批建议

优先做真实联调，不建议继续前端假实现：

1. 群公告真实数据联调。
2. 群设置、全员禁言、成员禁言联调。
3. 群成员管理联调：邀请、移除、管理员、转让群主。
4. @ mentions 在 push/pull/history 中的真实字段确认和提醒口径复测。
5. 回复引用摘要在 history/refetch 中的真实字段确认。
6. 群二维码、群已读详情、群文件管理需先明确服务端接口和产品范围。
