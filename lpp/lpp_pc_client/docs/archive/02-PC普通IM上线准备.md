# PC 普通 IM 上线准备

本文档用于 PC 普通 IM 发布候选通过最终轻量回归后的上线准备。范围只包含 PC 端普通 IM，不包含 App、在线客服临时会话、主题系统或管理后台能力。

## 发布范围

- 私聊：文本、emoji、图片、文件、名片展示与发送后的本地回显。
- 群聊：群 sender 可读展示、群成员事件系统提示、群成员列表、@ 人输入基础。
- 未读/已读：微信式规则；非当前会话收到消息保留未读，点击会话后 markRead，当前打开会话实时消息直接已读。
- 新消息提示：点击含未读会话后显示“↑↑ N 条新消息”，可跳到本地已加载范围内第一条未读。
- 图片/文件：按钮选择、拖拽、粘贴发送；上传失败提示；图片加载过程图、预览、下载；文件名/大小展示和下载。
- 消息操作：回复、复制、转发、多选、撤回、删除、收藏、翻译、媒体下载。
- 历史搜索：当前已加载消息范围内的关键词搜索、类型筛选和结果跳转。
- 批量操作：批量转发和批量删除按单条接口顺序执行，部分失败时展示成功/失败数量。
- 服务端一致性：收藏、撤回、删除、转发成功后触发 refetch，避免状态闪回。

## 不发布范围

- App 端能力。
- 在线客服临时会话、会话池、接待状态、客服自动分配。
- 主题系统、换肤、整体视觉重构。
- 管理后台、广播、工单、知识库、AI 助手等非普通 IM 能力。
- 全量服务端聊天历史搜索。
- 会话置顶设置/取消、免打扰持久化、隐藏/删除/恢复会话的服务端闭环。
- 群公告读取或编辑。
- 批量转发/批量删除服务端原子事务。
- 服务端缺失字段的前端假数据补齐。

## 发布前检查

### 自动回归

- `npm run typecheck` 必须通过。
- PC 普通 IM 专项必须通过。
- 如上线前有任何业务代码变更，必须重新执行必跑命令。

### 人工检查

- 未打开会话收到消息后，会话列表红点增加，不自动已读。
- 点击会话后红点清除，聊天区显示“↑↑ N 条新消息”。
- 当前打开会话收到实时消息后，消息直接展示且不产生红点。
- 自己发送的消息不产生未读。
- 私聊文本、emoji、图片、文件、名片展示正常。
- 群聊 sender 不显示 UUID，群事件展示为系统提示。
- 图片/文件发送失败时有明确提示，待发送附件不丢。
- 撤回、删除、收藏、转发后，切换会话或等待 refetch 不回退。
- 历史搜索和类型筛选可用，空状态清晰。
- 批量转发、批量删除部分失败时有明确数量提示。
- 试用说明、反馈闭环、服务端缺口、发布候选检查文档均为最新。

## 必跑命令

只跑 PC 普通 IM 专项，不跑全量：

```bash
npm run typecheck
npx playwright test tests/browser/workspace-smoke.spec.ts -g "PC ordinary IM closure|PC ordinary IM stage two|PC ordinary IM stage three|PC ordinary IM stage four|updates IM unread|marks incoming messages" --workers=1
```

本次上线准备的最终轻量验证结果：

- `npm run typecheck`：通过。
- PC 普通 IM 专项：`7 passed`。

## 试用/灰度策略

- 第一批：内部客服或灰度客服账号，覆盖至少一个私聊和一个群聊。
- 第二批：扩展到小范围真实客服，优先选择消息量稳定、可快速反馈的团队。
- 每批至少观察一个完整工作日，再决定是否扩大范围。
- 灰度期间保留 App 或旧 PC 稳定路径作为兜底。
- 灰度期间只接受 P0/P1 修复、数据兼容修复和文档勘误，不新增功能。

## 观察指标

- 普通 IM 会话列表加载失败率。
- 消息列表加载失败率。
- WebSocket/push 到达后消息展示延迟。
- markRead 请求失败率和 requestId。
- 会话未读数异常反馈数量。
- 图片/文件上传失败率。
- 图片预览和文件下载失败反馈数量。
- 撤回、删除、收藏、转发操作失败率。
- refetch 后状态回退反馈数量。
- 群 sender 显示 UUID 的反馈数量。
- 批量转发/批量删除部分失败或静默失败反馈数量。

## 高风险告警

出现以下任一情况需要立即升级：

- 未打开会话被自动已读。
- 非当前会话收到消息但未读红点不增加。
- 当前打开会话收不到实时消息。
- push/pull/history 任一路径出现消息空白。
- 图片/文件发送失败无提示或附件丢失。
- 撤回、删除、收藏、转发 refetch 后状态回退。
- 群 sender 大面积展示 UUID。
- 批量操作静默失败。
- API 401/403/500 无法取得 requestId 或无法定位环境。

## 暂停上线条件

- 出现任何未关闭 P0。
- 出现阻断性 P1 且没有稳定兜底路径。
- 未读/已读规则出现高风险异常。
- 消息收发出现丢消息、重复消息或空白消息。
- 图片/文件发送链路对试用团队不可用。
- 服务端返回结构变化导致 normalize 大面积失效。
- 必跑回归命令失败。

## 回滚条件

- 灰度后出现误已读、红点清错或消息丢失。
- refetch 后大量操作状态回退，影响客服判断。
- 图片/文件发送或下载在主流程中不可用。
- 群聊 sender 无法识别，影响真实客服沟通。
- 批量操作静默失败且无法快速修复。
- 新修复引入第一到第九阶段已有能力回退。

回滚后处理：

- 保留问题现场截图、日志、会话 ID、消息 ID、requestId。
- 按 [release/03-PC普通IM试用反馈闭环.md](../release/03-PC普通IM试用反馈闭环.md) 分级和归因。
- 前端问题补最小专项测试后再修复。
- 服务端缺口更新 [05-服务端支持.md](../05-服务端支持.md)。

## 反馈入口和责任人分工模板

### 反馈入口

- 试用客服：按反馈模板提交问题，附截图/录屏、会话 ID、消息 ID、requestId。
- 一线支持：确认是否可复现，补齐环境、账号角色、操作步骤。
- 前端研发：判断 PC 展示、normalize、cache、refetch、交互状态是否异常。
- 服务端研发：判断消息写入、push/pull/history、markRead、媒体、操作持久化是否异常。
- 产品/测试：确认问题分级、发布影响和是否暂停扩大灰度。
