# PC 普通 IM 真实闭环审计

审计时间：2026-05-27

本次审计停止使用“阶段完成/冻结”作为结论来源，只看 PC 普通 IM 在真实 UI、真实 API、真实状态闭环下的可用性。范围仅包含 `lpp_pc_client` 普通 IM，不包含 App、在线客服、主题系统。

## 真实验收环境

- PC 页面：`http://127.0.0.1:5173/`
- 服务端：`https://chat.hearteasechat.com`
- 测试账号：使用测试员工账号，不在文档记录真实密码或 token。
- 测试数据：通过真实好友申请 API 建立测试客服与测试管理员的好友关系，再创建真实私聊会话。

## 结论分级

- 真可用：已通过真实 UI + 真实 API 或真实 API + refetch 验证。
- 只在 mock 可用：现有专项测试覆盖，但本次没有真实服务端数据或真实环境支撑。
- 不可用：真实 UI/API 失败，且无法由服务端缺口解释。
- 服务端/环境缺口：前端路径存在，但被服务端接口、字段、权限、CORS 或测试数据阻断。
- 需要人工真实环境确认：需要双端、Electron 打包态、真实群聊或真实生产/灰度权限才能确认。

## 服务端/环境缺口

1. Gateway CORS / Origin：真实浏览器 UI 中 `/ws/client/negotiate` 被 CORS 拦截，实时 push 链路不能认定真可用。已补充到 [05-服务端支持.md](../05-服务端支持.md)。
2. 真实测试数据：当前员工测试账号初始没有好友、没有群聊；必须先通过真实好友申请建立测试数据，群聊相关能力仍缺真实群数据。
3. 服务端历史消息字段：真实历史返回 `senderDisplayName` 和 `preview` 为空，PC 当前能靠会话/身份和 body 展示文本、图片、文件，但 sender 可读性、会话预览质量依赖服务端字段补齐或前端更强兜底。
4. Electron 打包态：本次验证覆盖本地浏览器页面；Electron 下文件选择器、下载路径、系统通知、Gateway Origin 仍需人工或 Electron E2E 确认。

## 高风险降级结论

- 不能再把 PC 普通 IM 判定为“发布冻结完成”。
- 图片/文件核心发送在真实 API 和真实浏览器 UI 下已验证可用，但 Electron 打包态仍需确认。
- 实时 push、实时未读/已读、群聊、批量操作、撤回/删除/收藏/转发真实 refetch 仍不能用 mock 结论替代。
- 下一步应先补真实联调脚本或 Electron E2E，而不是继续推进发布阶段文档。
