## 3. 需求准入规则

新增或变更需求必须先完成需求和设计，再进入开发。

| 字段 | 可选值 | 规则 |
| --- | --- | --- |
| 需求状态 | 待补齐 / 评审中 / 完成 / 变更中 / 暂缓 | 只有 `完成` 才能开发 |
| 设计状态 | 待补齐 / 评审中 / 完成 / 变更中 / 暂缓 | 只有 `完成` 才能开发 |
| 开发准入 | 未准入 / 准入 | 仅当需求状态和设计状态均为 `完成` 时可标 `准入` |
| 实现状态 | 待排期 / 已排期 / 开发中 / 待验收 / 已完成 / 暂缓 | 反映代码实现进度 |

需求或设计进入 `变更中` 后，相关功能必须暂停新增实现，只允许明确的 P0/P1 修复或回滚。

## 7. 不交付范围索引

以下内容不能在当前版本中被误认为已完整交付。详细功能状态以 `03-PC端功能矩阵.md` 为准。

1. App 端新增能力。
2. 管理后台完整能力。
3. 跨设备全量历史搜索；`PC-REQ-20260606-004` 只覆盖本机已落库消息搜索。
4. 服务端未提供的会话置顶、免打扰、隐藏、删除、恢复持久化。
5. 批量转发和批量删除的服务端原子事务，除非 `PC-REQ-20260606-005` 实现并验收完成。
6. 缺字段情况下的假持久化、假群公告、假角色或假客户数据。
7. 未经服务端确认的多端强一致状态。
