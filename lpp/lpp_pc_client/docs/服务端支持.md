# PC 普通 IM 服务端支持缺口

本文件只记录 PC 普通 IM 在真实服务端持久化闭环中缺少的接口或字段。PC 端不会为这些能力硬编假持久化；已有真实接口的能力会走真实 API 并依赖 refetch 校验一致性。

## 已有真实接口

- 消息列表：`GET /api/client/v1/direct-chats/{conversationId}/messages`、`GET /api/client/v1/groups/{conversationId}/messages`
- 发送消息：`POST /api/client/v1/direct-chats/{conversationId}/messages`、`POST /api/client/v1/groups/{conversationId}/messages`
- 上传媒体：`POST /api/client/v1/media/upload`
- 已读：`POST /api/client/v1/direct-chats/{conversationId}/read`、`POST /api/client/v1/groups/{conversationId}/read`
- 撤回：`POST /api/client/v1/messages/{messageId}/recall`
- 删除消息：`POST /api/client/v1/messages/{messageId}/delete`
- 转发：`POST /api/client/v1/messages/forward`
- 收藏：`POST /api/client/v1/favorites`、`GET /api/client/v1/favorites/list`
- 翻译：`POST /api/client/v1/translate/message`、`POST /api/client/v1/translate/text`

## 会话置顶

- 前端场景：会话列表排序需要支持置顶 > 未读 > 最新时间，用户应能设置/取消置顶。
- 期望 endpoint：`/api/client/v1/conversations/{conversationId}/pin`
- method：`PUT`
- request body：`{ "isPinned": true }`
- response data：`{ "conversationId": "...", "isPinned": true, "updatedAt": "..." }`
- 当前前端临时处理方式：只读取会话列表返回的 `isPinned` 并参与排序；不提供假持久化置顶操作。

## 会话免打扰

- 前端场景：用户在 PC 对某个普通 IM 会话开启/关闭免打扰，刷新、重新登录、换设备后保持一致。
- 期望 endpoint：`/api/client/v1/conversations/{conversationId}/mute`
- method：`PUT`
- request body：`{ "isMuted": true }`
- response data：`{ "conversationId": "...", "isMuted": true, "updatedAt": "..." }`
- 当前前端临时处理方式：会话列表可读取 `isMuted`；右键免打扰只做本地 session 级展示，并提示需要服务端接口支持。

## 删除/隐藏/恢复会话

- 前端场景：用户删除、隐藏或恢复普通 IM 会话，refetch 后列表状态保持一致。
- 期望 endpoint：`/api/client/v1/conversations/{conversationId}/visibility`
- method：`PUT`
- request body：`{ "hidden": true }`
- response data：`{ "conversationId": "...", "hidden": true, "updatedAt": "..." }`
- 当前前端临时处理方式：右键隐藏/删除会话只在本地 session 内隐藏；删除会话会先弹确认，不伪造服务端删除结果。

## 群公告

- 前端场景：群资料面板展示真实群公告，公告为空时显示明确空状态。
- 期望 endpoint：`/api/client/v1/groups/{conversationId}/announcement`
- method：`GET`
- request body：无
- response data：`{ "announcementId": "...", "content": "...", "updatedAt": "...", "updatedByDisplayName": "..." }`
- 当前前端临时处理方式：没有接口时显示“暂无群公告”，不造假公告。

## 群成员昵称和角色

- 前端场景：群消息 sender、群资料成员列表、头像资料弹层展示群昵称、群主/管理员/成员角色。
- 期望 endpoint：沿用 `GET /api/client/v1/groups/{conversationId}/members`
- method：`GET`
- request body：无
- response data：成员项建议稳定返回 `{ "userId": "...", "displayName": "...", "groupNickname": "...", "role": "owner|admin|member", "avatarUrl": "..." }`
- 当前前端临时处理方式：优先读取 `displayName`、`role`、`memberRole`；缺失时只展示已有字段，不显示 UUID 作为姓名。

## @ mentions

- 前端场景：群聊输入 `@成员` 后发送，push/pull/history 回显仍能展示被提及人；后续可扩展“有人 @ 我”提醒。
- 期望 endpoint：沿用群消息发送 `POST /api/client/v1/groups/{conversationId}/messages`
- method：`POST`
- request body：`{ "messageType": "text", "body": { "text": "@Mina 请看下" }, "mentions": [{ "userId": "user-mina", "displayName": "Mina" }] }`
- response data：建议返回 `{ "messageId": "...", "conversationId": "...", "conversationSeq": 123, "serverTime": "...", "mentions": [...] }`
- 当前前端临时处理方式：PC 已按现有发送接口提交 `mentions`；展示层以正文真实文本为准。服务端需确认 push/pull/history 的 `mentions` 字段结构和移动端兼容规则。

## 回复引用详情

- 前端场景：回复消息在本地回显、push、pull、history、refetch 后都展示引用摘要。
- 期望 endpoint：沿用消息收发接口。
- method：`GET` / `POST`
- request body：发送时 `{ "replyToMessageId": "m1" }`
- response data：历史/推送消息建议返回 `{ "replyToMessageId": "m1", "reply": { "messageId": "m1", "senderDisplayName": "Jason", "preview": "原消息摘要", "messageType": "text" } }`
- 当前前端临时处理方式：PC 会读取 `body.reply`、`body.replyTo`、`body.quotedMessage`、`body.quote`、`message.reply` 中的摘要；如果历史接口只返回 `replyToMessageId` 而没有摘要，PC 无法凭空展示引用内容。

## 批量转发/批量删除

- 前端场景：多选后批量转发或删除，需要明确成功、失败、部分失败状态。
- 期望 endpoint：`/api/client/v1/messages/batch-forward`、`/api/client/v1/messages/batch-delete`
- method：`POST`
- request body：`{ "messageIds": ["m1", "m2"], "targetConversationId": "group-1" }`
- response data：`{ "successIds": ["m1"], "failedItems": [{ "messageId": "m2", "message": "..." }] }`
- 当前前端临时处理方式：PC 按现有单条接口顺序执行，批量转发和批量删除都会汇总部分失败提示；如需原子性或性能保障，需要服务端批量接口。

## PC Gateway CORS / Origin

- 前端场景：PC 普通 IM 依赖 `/ws/client` 接收实时 `msg.new`、`msg.read`、撤回等事件；开发态来源为 `http://127.0.0.1:5173`，Electron 打包态可能为 `file://` 或应用自定义来源。普通 HTTP API 当前可从 PC 页面访问，但真实 UI 审计时 `/ws/client/negotiate` 被浏览器 CORS 拦截，导致 push 实时链路不能在该环境认定为真可用。
- 期望 endpoint：`/ws/client/negotiate`、`/ws/client`
- method：`OPTIONS` / `POST` / WebSocket
- request body：SignalR negotiate 请求体由客户端库生成；鉴权通过 access token。
- response data：SignalR negotiate 标准响应；预检响应需允许 PC 来源、`Authorization`、`Content-Type`，且当带凭据时 `Access-Control-Allow-Credentials` 与 `Access-Control-Allow-Origin` 组合必须合法。
- 当前前端临时处理方式：Gateway 连接失败时不阻断页面，PC 仍通过会话列表和消息列表轮询/refetch 补偿；但实时 push、当前会话实时已读、非当前会话实时红点必须在服务端 CORS/Origin 放开后重新做真实联调，不能用 mock 测试替代。

## 自己消息未读一致性

- 前端场景：当前账号自己发送的普通 IM 消息，包括本机 local echo、另一个设备发送后的 push、pull/refetch/history 回显，都不能产生未读红点、未读筛选数量、新消息提示或提醒。
- 期望 endpoint：`GET /api/client/v1/conversations`、普通 IM push、普通 IM history/pull 接口。
- method：`GET` / WebSocket。
- request body：无。
- response data：如果 `lastMessage` 是当前账号发送，服务端应返回 `unreadCount: 0`，并稳定返回可判定自消息的字段之一：`senderUserId`、`senderId`、`fromUserId`、`senderPlatformUserId`、`platformUserId`、`senderLppId`、`lppId`、`isSelf`、`isMine`、`direction: "out"`。
- 当前前端临时处理方式：PC 展示层已用 `effectiveConversationUnreadCount` 兜底，识别到最后一条是自己消息时强制显示未读为 0；自己发送 text/image/file 成功并拿到 `conversationSeq` 后，PC 会立即调用对应 read 接口并本地持久化 readSeq，防止刷新/refetch 后未读复活；Gateway 收到自己消息也只更新消息和最近会话，不增加未读、不提醒。服务端仍应修正 self lastMessage 携带 `unreadCount > 0` 或缺少 self/sender 字段的返回，避免多端未读状态不一致。
