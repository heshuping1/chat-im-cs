# PC 普通 IM 试用反馈闭环

本文档用于 PC 普通 IM 小范围真实客服试用期间的问题收集、复现、分级、定位和回归。范围只包含 PC 普通 IM，不包含 App、在线客服临时会话、主题系统或管理后台能力。

## 当前基础

- 试用说明已建立：[02-PC普通IM试用说明.md](02-PC普通IM试用说明.md)。
- 服务端缺口已记录：[05-服务端支持.md](../05-服务端支持.md)。
- API 层 `ApiError` 已携带 `requestId`，普通 IM 失败提示通过 `formatError` 展示服务端错误信息。
- 已存在 `POST /api/client/v1/client-errors` 的 API client，但设置页“用户反馈”入口仍为待接入状态；试用期先按本文档模板人工收集。
- PC 普通 IM 专项测试覆盖未读/已读、normalize、图片文件、名片、群事件、操作 refetch、批量部分失败等核心闭环。

## 问题分级

- P0：阻断试用或造成严重错误。包括误已读、未读红点被错误清除、消息丢失或收不到、发送成功但双方状态严重不一致、客户敏感信息错误展示。
- P1：核心链路不可用但有替代路径。包括图片/文件无法发送且无明确提示、消息类型空白、撤回/删除/转发/收藏 refetch 后回退、群 sender 显示 UUID、批量操作静默失败。
- P2：影响效率或信任感但不阻断。包括历史搜索结果不完整、跳转不准确、草稿偶发丢失、失败提示不够明确、菜单项显示不够精确。
- P3：体验优化或文案问题。包括间距、样式、提示文案、空状态、试用说明补充项。

## 常见问题定位路径

### 收不到消息

1. 确认发送端是否成功发送，记录发送端消息 ID 和时间。
2. 检查 PC 是否在正确租户、账号、会话列表内。
3. 检查 WebSocket/push 是否收到；如果未收到，刷新会话列表或等待 pull/refetch 是否出现。
4. 如果 push 没有、pull 有：优先定位网关或订阅问题。
5. 如果 push/pull 都没有：优先定位服务端消息写入、权限、会话 ID 或账号身份。
6. 反馈需附：会话 ID、发送端消息 ID、PC 当前账号 userId/lppId、发生时间、Network/WebSocket 截图。

### 未读/已读异常

1. 明确问题是“未打开会话被已读”还是“当前打开会话未自动已读”。
2. 记录用户是否点击过会话，是否当前正在该会话聊天页。
3. 检查会话返回的 `unreadCount`、`lastReadSeq`、`lastMessageSeq`。
4. 检查是否调用了 `/direct-chats/{conversationId}/read` 或 `/groups/{conversationId}/read`，记录 requestId。
5. 如果未点击会话却调用 markRead：按 P0 处理。
6. 如果点击会话后红点未清：按 P1 处理，检查 markRead 请求和会话 refetch 数据。

### 图片/文件失败

1. 区分失败阶段：选择附件、上传、发送消息、展示、预览、下载。
2. 上传失败记录 `/media/upload` 的 status、code、message、requestId。
3. 发送失败记录消息发送接口的 requestId。
4. 展示失败记录媒体字段：`url`、`thumbnailUrl`、`fileName`、`mimeType`、`sizeBytes`。
5. 确认失败后附件是否保留、是否有明确错误提示。
6. 如果失败无提示或附件丢失：按 P1 处理。

### 消息类型显示异常

1. 记录原始 `messageType` 和 `body` 结构。
2. 确认是否为空 body、未知类型、字段命名差异或嵌套结构。
3. 如果是未知类型，PC 应展示“暂不支持的消息类型：xxx”，不能空白。
4. 如果 App 能显示但 PC 空白，优先补 normalize 映射和专项用例。
5. 涉及名片时，记录 `contactCard` / `contact_card` / `contact` 字段。

### 群 sender 异常

1. 记录消息里的 sender 字段：`senderUserId`、`senderDisplayName`、`senderPlatformUserId`、`senderLppId`。
2. 记录群成员接口返回字段：`userId`、`platformUserId`、`lppId`、`displayName`、`role`、`memberRole`。
3. 如果展示 UUID，确认 UUID 来自消息字段还是群成员字段缺失。
4. 如果服务端没有可读姓名字段，记录到 [05-服务端支持.md](../05-服务端支持.md)。

### 操作 refetch 后回退

1. 明确操作类型：收藏 / 撤回 / 删除 / 转发。
2. 记录操作请求的 endpoint、request body、response data、requestId。
3. 操作成功后等待 refetch 或切换会话回来，确认服务端消息列表是否返回新状态。
4. 如果前端本地状态正确但 refetch 后回退，多半是服务端状态未持久化或历史接口未返回状态字段。
5. 如果服务端返回正确但 UI 回退，补前端 normalize/cache 专项测试。

### 批量操作部分失败

1. 记录批量对象数量、成功数量、失败数量。
2. 当前 PC 使用单条接口顺序执行，不具备服务端批量原子性。
3. 如果部分失败有明确提示，按 P2 或 P3 处理。
4. 如果静默失败、全部提示成功但实际失败，按 P1 处理。
5. 如需原子性或完整失败明细，需要服务端批量接口支持。

## 复现脚本

### 基础未读脚本

1. PC 登录客服账号，停留在消息列表但不打开会话 A。
2. 另一端向会话 A 发送文本消息。
3. 观察会话 A 未读红点是否增加。
4. 点击会话 A，确认红点清除并显示“↑↑ N 条新消息”。
5. 当前停留会话 A，再从另一端发送消息，确认消息直接展示且不出现未读红点。

### 消息类型脚本

1. 分别发送文本、emoji、图片、文件、名片、群事件。
2. 检查 PC 是否展示正确内容或明确兜底。
3. 对图片执行预览和下载。
4. 对文件执行下载。
5. 对名片执行点击查看资料。

### 操作一致性脚本

1. 发送一条文本消息。
2. 依次执行回复、收藏、转发、撤回、删除。
3. 每次操作后切换会话再切回，确认 refetch 后状态不回退。
4. 对群聊执行 @ 人发送，确认本地回显和历史回显可读。

### 批量失败脚本

1. 选择多条消息进入多选。
2. 批量转发到目标会话。
3. 模拟其中一条失败，确认提示包含成功数量和失败数量。
4. 批量删除同理验证部分失败提示。

## 是否需要服务端配合

满足任一条件时，需要服务端配合：

- 前端没有收到 push，pull/history 也没有目标消息。
- markRead 请求正确发送，但 refetch 后 `unreadCount`、`lastReadSeq` 仍错误。
- 收藏、撤回、删除、转发请求成功，但消息列表 refetch 不返回对应状态。
- 群 sender 缺少可读姓名、群昵称或角色字段。
- 历史消息只返回 `replyToMessageId`，没有引用摘要。
- @ mentions 发送后 push/pull/history 字段不一致。
- 会话置顶、免打扰、隐藏/删除/恢复、群公告等能力缺接口。
- 批量转发/批量删除需要原子性、失败明细或服务端统一事务。

更像前端问题的情况：

- 服务端返回的数据正确，但 UI 不显示、显示空白或显示 UUID。
- push/pull/history/local echo 同一消息类型展示不一致。
- 操作后本地状态正确但切换会话触发前端 cache 后异常。
- 上传或发送失败已有 requestId，但前端没有保留附件或没有明确提示。

## 关闭问题标准

一个试用问题关闭前必须满足：

- 已确认根因归属：前端 / 服务端 / 数据 / 网络 / 权限 / 产品缺口。
- 已补充修复说明或服务端缺口说明。
- 已跑对应专项测试命令。
- 如涉及新边界，已补最小专项测试。
- 反馈人确认真实环境复测通过，或记录无法复现原因。
