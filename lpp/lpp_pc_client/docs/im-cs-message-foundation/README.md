## 重要结论

本目录保留当前有效方案和执行清单。后续无论是人工研发还是 AI 协作，都必须优先阅读并遵守本目录下的当前有效文档。

## 必读顺序

1. [01-企业级IM与在线客服架构蓝图.md](./01-企业级IM与在线客服架构蓝图.md)

   用于统一顶级 IM + 在线客服系统应该具备的完整架构、协议、实时链路、同步模型、IM 域、客服域、发送、通知、缓存、多端、安全和可观测性。

2. [02-消息系统闭环交付方案.md](./02-消息系统闭环交付方案.md)

   用于指导实际落地。重点看核心 6 个模型、不变量、最小完整架构、数据源合并规则、Reconcile 闭环、测试闭环和最终完成定义。

3. [03-场景验收矩阵.md](./03-场景验收矩阵.md)

   用于判断系统是否真的完成。所有 `需补实现`、`需补测试`、`依赖服务端` 项都必须被明确处理，不能只靠主观判断说“完成了”。

4. [04-模块化重构任务清单.md](./04-模块化重构任务清单.md)

   用于新会话按模块逐项执行。每个模块开始前必须先填写模块设计卡；每个任务都要明确改动范围、禁止事项、测试、验收和剩余缺口。

## AI 协作规则

AI 在处理 IM + 在线客服消息系统相关任务时，必须遵守以下规则：

- 只能把本目录下当前有效文档作为方案和执行依据。
- 不得从 `archive-历史方案/` 中选择旧方案覆盖当前方案。
- 不得只根据某个 bug 现象做局部补丁，必须先对照 `03-场景验收矩阵.md` 判断影响面。
- 实现前必须确认是否违反 `02-消息系统闭环交付方案.md` 的不变量。
- 执行 `04-模块化重构任务清单.md` 中任一模块前，必须先补齐该模块设计卡。
- 模块完成后必须按 `04` 中的代码实现合理性审查清单自审。
- 修改实时消息、未读、提醒、已读、发送、客服线程时，必须说明对应的模型入口：
  - `GatewayConnectionManager`
  - `MessageDeliveryService`
  - `SnapshotReconcileService`
  - `ConversationOwnershipResolver`
  - `ImReadView`
  - `CustomerServiceUnreadLedger`
- 如果发现当前有效文档之间有冲突，以优先级处理：
  1. `03-场景验收矩阵.md` 的具体场景期望
  2. `02-消息系统闭环交付方案.md` 的不变量和完成定义
  3. `01-企业级IM与在线客服架构蓝图.md` 的总体架构原则
  4. `04-模块化重构任务清单.md` 的执行顺序和任务拆分

## 对当前有效文档的边界说明

入口目录可以保留方案文档和执行清单。文档数量不是门禁，是否会误导才是门禁。

但旧文档不删除，统一归档到 `archive-历史方案/`。这是为了保留决策过程和问题追溯能力，不代表旧文档仍然有效。

后续如果要新增文档，必须满足：

- 明确属于当前有效文档还是历史归档。
- README 必须同步更新入口和用途。
- 不能新增一个和现有方案冲突、却没有优先级说明的文档。
- 任务清单、执行记录、服务端缺口清单可以独立成文档，避免把方案文档写得过长。
