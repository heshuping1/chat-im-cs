# PC IM + 在线客服模块化重构任务清单

日期：2026-06-02

本清单用于新会话按模块逐项执行重构。执行依据仍是同目录下 `01-企业级IM与在线客服架构蓝图.md`、`02-消息系统闭环交付方案.md`、`03-场景验收矩阵.md`。


## 1. 使用规则

这个清单用于新会话按模块逐项执行，不能跳模块重构。模块编号固定，后续执行和反馈都按 `Mxx / TASK-xx-yy` 引用。

执行顺序：

1. 从底层到上层执行。
2. 每次只做一个模块。
3. 每个模块先补设计和边界测试，再迁移调用点。
4. 一个模块没有测试和验收通过，不进入下一个模块。
5. 默认沿用当前代码方案；换库、换架构、重写模块必须先说明理由并确认。
6. 当前 API 不支持的能力只标记为服务端缺口，不在 PC 里伪造。

状态标记：

- `[ ]` 未开始。
- `[~]` 进行中。
- `[x]` 已完成。
- `[!]` 阻塞，需要 API、产品或技术确认。

每个任务完成时必须写清：

- 改了哪些文件。
- 对应哪个模型。
- 跑了哪些单测。
- 哪些场景手动验收了。
- 还有哪些缺口。

## 4. 代码实现合理性审查清单

每个模块完成后，必须按下面清单自审。未通过自审，不进入下一个模块。

### 4.1 DDD 边界

- [ ] 外部 API payload 没有进入领域层。
- [ ] HTTP response item 没有被 UI、Reminder、Read model、Ledger 直接读取。
- [ ] 模块没有 import 不该依赖的内部模型。
- [ ] 模块之间通过 Command、Domain Event、Value Object、Effective View 交互。
- [ ] IM Domain 没有读取客服 ledger/workbench/tempSession 原始结构。
- [ ] CustomerService Domain 没有读取 IM read view/raw conversation unread。

### 4.2 数据一致性

- [ ] push、refetch、detail/history 同一消息不会重复写入。
- [ ] 旧 seq/snapshot 不覆盖新 push。
- [ ] 无 seq/version 的数据只补弱字段。
- [ ] 当前 API 不支持的能力没有在 PC 里伪造。
- [ ] 多账号/多工作区通过 scope 隔离。

### 4.3 未读和提醒

- [ ] IM 未读只由 `ImReadView` 解释。
- [ ] 客服访客未读只由 `CustomerServiceUnreadLedger` 解释。
- [ ] UI 不读取 raw unread。
- [ ] Reminder 不解析 raw message/thread。
- [ ] Badge 不叠加 realtime reminder 数量。
- [ ] 自己消息不产生未读和桌面提醒。

### 4.4 实时性和异常

- [ ] Gateway 失败持续 retry。
- [ ] Gateway 异常状态可见，不静默伪装实时正常。
- [ ] 重连后触发当前 API refetch 补偿。
- [ ] 补偿失败不回滚已收到 push。
- [ ] 日志能解释 push 到 effective view 的耗时。

### 4.5 技术选型

- [ ] 默认沿用当前代码方案。
- [ ] 没有未经确认替换 SignalR、React Query、发送 runtime、缓存体系或本地存储方案。
- [ ] 新增抽象确实减少重复或固定边界，不是为了形式化。
- [ ] 没有为了 API 字段不统一在 PC 内写字段兼容。

### 4.6 测试

- [ ] 本模块单测通过。
- [ ] 相关边界测试通过。
- [ ] 相关场景矩阵项已更新状态或记录缺口。
- [ ] `typecheck` 影响已确认。
- [ ] 必要的手动验收已记录。

### M01：基础设施层 - GatewayConnectionManager

#### [ ] TASK-01-01 完善 Gateway 模块设计

目标：

- 按 `02` 的完整模块设计模板补齐 Gateway 模块小设计。

必须写清：

- 领域职责：只管理连接生命周期，不写业务 cache。
- 输入：session scope、gatewayUrl、auth token、start/stop command。
- 输出：connection health event、connected/retrying/stopped 状态、诊断日志。
- 内部状态：connection、retry timer、generation、heartbeat state。
- 不变量：一个 scope 一个有效连接；旧 generation 不写新状态。
- 当前 API 支撑：`/ws/client` + SignalR。
- 服务端缺口：heartbeat ack 是否有明确协议。

验收：

- 文档或代码注释中能明确模块边界。

#### [ ] TASK-01-02 补 Gateway 边界测试

目标：

- 防止长连接治理再次被业务逻辑污染。

测试覆盖：

- 初始 start 失败持续 retry。
- SignalR reconnect 最终 close 后 manager 接管 retry。
- session 切换后旧连接回调不写新状态。
- gateway health 日志包含 start、failed、retry、connected、closed。
- Gateway 不 import IM cache / CS ledger / UI store 写业务状态。

建议命令：

```powershell
npx.cmd vitest run tests/unit/gateway-connection-manager.spec.ts tests/unit/architecture-boundaries.spec.ts
```

验收：

- `GW-01`、`GW-04`、`GW-05`、`GW-15` 至少有测试。

#### [ ] TASK-01-03 收敛 GatewayBridge 调用点

目标：

- `GatewayBridge` 只负责把连接事件交给 manager / delivery，不直接做业务写入。

禁止：

### M05：主动查询合并层 - SnapshotReconcileService

#### [ ] TASK-05-01 设计并实现 SnapshotReconcileService

目标：

- 当前最大缺口：防止主动查询覆盖 push。

输入：

- IM conversation snapshot。
- CS workbench snapshot。
- tempSession 过渡数据。
- detail/history 弱字段。

规则：

- `incomingSeq > localSeq` 才能更新强状态。
- `incomingSeq === localSeq` 只补弱字段。
- `incomingSeq < localSeq` 不覆盖强状态。
- 无 seq/version 只能补弱字段。

禁止：

- 不直接清未读。
- 不直接写 badge。
- 不弹通知。

测试覆盖：

```powershell

```

验收：

- `SYNC-01`、`SYNC-05`、`SYNC-07`、`SYNC-11` 从 `需补实现` 变成已覆盖。

范围：

- IM conversation list。

### M06：当前 API 补偿同步 - MessageGapSyncCoordinator

#### [ ] TASK-06-01 完善当前 API refetch 补偿

目标：

- 长连接恢复、seq gap、snapshot 落后时触发当前 API 补偿。

当前 API 策略：
