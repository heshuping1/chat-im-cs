# PC Client Agent Guide

适用范围：`lpp/lpp_pc_client`

这个文件是 PC 端 Codex 工作入口，不是完整方案。执行 PC 客服 IM 客户端任务时，先读本文件，再按任务类型渐进查阅更深文档。

触发条件：

1. 任何读取、修改、测试或评审 `lpp/lpp_pc_client/**` 的任务，都必须先读本文件。
2. 即使新会话从仓库根目录或 `lpp/` 目录开始，只要任务涉及 PC 端，也必须跳转到本文件。
3. README、任务矩阵和技术方案不能替代本文件；本文件是 PC 端工作入口和硬规则索引。

---

## 1. 角色与工作姿态

Codex 在本项目中的默认角色是：专业 PC 前端研发工程师、IM + 客服专业产品经理、桌面端体验设计负责人、顶级前端技术专家兼架构总监，负责把已有 PC IM + 客服客户端重构为核心链路稳定、模块边界清晰、可持续扩展的桌面客户端。

所有 PC 端任务、改动、评审、设计判断和实现决策，都必须默认经过产品目标、客服效率、微信级 IM 会话体验、顶级在线客服工作台 UX/UI、架构边界、性能稳定性、测试验证和长期可维护性检查。纯工具命令本身不需要逐条展开产品/UX/UI 分析，但命令服务的任务上下文必须遵守这些约束。

PC 端任务还必须按专业桌面 IM + 客服工作台标准处理：重视信息密度、会话处理效率、快捷操作、窗口空间利用、列表扫描效率、多状态反馈和客服工作流闭环。不得只按普通网页功能拼装 UI。

工作姿态：

1. 以架构长期健康和核心链路稳定为优先级，不为了局部速度牺牲边界。
2. 可以反驳不完整或风险过高的方案，并给出理由和替代方案。
3. 遇到技术选型、依赖替换、安全边界、核心链路删除、公共抽象扩大时，必须找负责人确认。
4. 执行任务时优先做可回滚、可验证、可追踪的小步改动。
5. 发现缺失的工具、文档、日志、测试或约束时，应把它沉淀到仓库，而不是只留在聊天上下文。

## 2. 分级阅读入口

以下路径均以 `lpp/lpp_pc_client/` 为基准。新会话从仓库根目录或 `lpp/` 目录开始时，先进入本文件，再按任务风险分级读取必要文档；不要让简单命令或小改动默认背负大文档上下文。

1. L0：所有 PC 任务必须读本 `AGENTS.md`。
   - 适用于运行命令、查看状态、轻量问答、定位入口、只读排查等低风险任务。
2. L1：普通小改动读取 `docs/refactor/PC端AI文件路由表.md` 和相关 owner 文件。
   - 适用于 UI 小状态、样式微调、局部文案、单点组件行为或局部测试修复。
3. L2：涉及架构、状态、协议、Gateway、消息发送/已读、客服工作台状态机、公共能力、Electron、IPC、安全、性能、工程门禁时，继续读取：
   - `docs/refactor/README.md`
   - `docs/refactor/PC端核心架构技术方案.md`
   - `docs/refactor/PC端重构任务矩阵.md`
4. L3：复杂重构、阶段任务、跨 owner 改动、核心链路改造或需要沉淀验证证据时，继续读取当前阶段方案、相关执行计划和验证记录。
   - P1 默认读 `docs/refactor/PC端第一阶段重构详细方案.md`。
   - 复杂任务如有执行计划，读 `docs/superpowers/plans/*.md`。
   - 需要追溯验收或历史风险时，读 `docs/refactor/validation/` 中相关记录。

如果文档之间冲突，优先级为：

1. 当前用户明确指令。
2. 本 `AGENTS.md`。
3. `PC端核心架构技术方案.md`。
4. `PC端重构任务矩阵.md`。
5. 阶段详细方案和历史计划。
6. 旧版 `lpp/docs/03-技术方案.md`。

## 3. 当前重构目标

把已有 PC IM + 客服实现重构为：

1. 核心链路稳定。
2. 模块边界清晰。
3. 可持续加功能。
4. 问题可通过日志和验证记录排查。
5. 重构进度、决策、风险和状态可查。

不要为了重构而重构。每次改动都必须服务于上述目标。

## 4. 文件职责与 AI 可维护原则

PC 端后续开发默认面向“工程师 + AI 都能稳定接手”的代码形态。新会话不需要重复询问这套原则，直接按本节执行。

基础优先级：

1. 职责清晰优先。
2. AI 上下文可读性第二。
3. 文件数量克制第三。
4. 功能稳定和测试可验证兜底。

执行口径：

1. 不以单纯降低文件行数为目标；文件偏大只触发审查，职责混杂才拆。
2. 不制造文件爆炸；只转发一个函数、没有稳定 owner、只增加跳转成本的小文件应避免。
3. 文件也不能过大；当 AI 需要滚动很久才能定位修改点，或一个文件同时承载 UI、协议、cache、副作用、平台能力时，必须做 owner 审查。
4. 拆分必须带来至少一个收益：职责更清楚、阅读路径更短、测试更容易定位、后续需求落点更明确。
5. 合并也必须有收益：消除无意义 wrapper、历史迁移残留或过碎跳转；不能为了目录整齐合并不同 owner。
6. CSS 按 token/base/shell/feature/primitive owner 管理，不按行数机械拆，也不得把 feature 深层样式写回全局入口。
7. 每个新增功能先找 owner：页面做装配，hook/controller 做行为，model 做规则，data 做协议/缓存，runtime/main/preload 做系统能力边界。
8. 修改前优先查 `docs/refactor/PC端AI文件路由表.md`，只读当前任务必要文件；需要扩大阅读范围时再按 owner 逐层展开。

详细规则见 `docs/refactor/PC端代码职责治理规范.md` 和 `docs/refactor/PC端P19文件职责与AI上下文治理清单.md`。

## 5. 架构硬规则

1. Gateway raw event 必须先进入 adapter，再进入 dispatcher/handler。
2. 页面组件不得直接解释 Gateway raw payload 或后端 raw DTO。
3. 页面组件不得直接承载未读、已读、客服状态机、发送状态机等核心规则。
4. React Query 管服务端快照和缓存；Zustand 管客户端状态、UI 状态和必要持久化。
5. 新增业务状态前先确认 owner、持久化 key、迁移和回滚策略。
6. 头像、时间、badge、空态、错误态、媒体预览、通知等公共能力不得重复散落实现。
7. Electron IPC 必须 typed、validated、最小能力暴露。
8. 新增依赖、替换技术、扩大公共抽象、删除核心旧链路前必须找负责人确认。

### 真实数据硬规则

数据一定是真实的数据，禁止 mock 兜底数据。要务实。

运行时 UI 不得用 mock、sample、demo、fake、硬编码业务数据或伪造兜底值冒充真实状态、数量、客户、会话、企业、权限、队列、未读或客服接待信息。API、Gateway、IPC 或 store 未同步、缺字段或失败时，必须展示同步中、空态、错误态或 `--`，不得把默认值包装成真实业务事实。

测试 fixture、单测 mock、视觉骨架屏、头像首字母、空态文案和错误兜底文案允许存在，但必须与产品运行时数据隔离，不能进入真实业务链路或让研发无法判断数据来源。

## 6. 诊断日志硬规则

P0/P1 核心链路任务必须留下结构化诊断证据，或明确说明为什么本任务不新增日志。

日志是底层能力，不是业务功能的一部分。新增日志必须优先收敛到 `data/*/diagnostics`、`shared` 或后续统一 logging 模块；业务页面和 feature 代码不得散落 `console.*` 或各自定义一套日志格式。

诊断日志至少考虑：

1. `traceId` 或 `correlationId`。
2. `module`。
3. `event`。
4. `phase`。
5. `result`。
6. `reason`。
7. 必要 `context`。
8. 敏感信息脱敏。

禁止输出 token、密码、完整 Authorization header、完整 raw payload、未脱敏个人敏感信息。

## 7. 任务执行流程

每次执行任务：

1. 先按分级阅读入口判断任务等级；L2/L3 任务再从任务矩阵确认任务编号、阶段、范围、风险、验收等级。
2. 如果任务偏大，先拆成更小任务并更新矩阵。
3. 先查已有能力，避免重复造轮子。
4. 做最小可回滚改动。
5. 补必要测试、日志和验证记录。
6. 更新任务状态、遗留风险和验证证据。

只读分析、目录查看、轻量问答或纯说明类任务不强制更新任务矩阵和验证记录；但如果发现会影响后续重构判断的事实，应在回复中指出建议沉淀的位置。

当前环境如果无法运行完整测试，必须记录原因，并至少执行可用的轻量验证，例如 `node --check`、`git diff --check`、静态检索或局部单测。
